Objective 3.4 — Resilience and Recovery in Security Architecture

Resilience is the ability of a system or organization to continue delivering services despite failures, disasters, human errors, or attacks, or to return to an operational state within acceptable timeframes after an outage. Within the CIA triad, resilience primarily safeguards the Availability component: a service that is unavailable when needed causes real damage, even if no data has been compromised or stolen.

Designing for resilience means eliminating Single Points of Failure (SPOF) — those components whose failure causes the entire service to stop — and ensuring that every layer of the infrastructure (compute, network, storage, power, physical site) has a continuity mechanism or rapid recovery path. Each measure adds cost and complexity, so decisions must be justified by RPO and RTO values consistent with the criticality of the service.

EXAM TIP 3.4 — General context: the exam often links resilience and recovery in the context of a BIA (Business Impact Analysis). When a scenario describes a decision about hot/warm/cold sites, backup frequency, or test type, the answer criterion is always: which option meets the required RTO and RPO at the lowest possible cost?

High Availability (High Availability)

High availability (HA) is the property of a system designed to minimize unplanned downtime. The goal is not the total elimination of failures — which is impossible — but reducing their impact to acceptable levels, often expressed as a percentage of annual uptime (e.g., "five nines" = 99.999% ≈ 5 minutes of downtime per year).

HA is achieved through redundancy: every critical component has a counterpart ready to replace it. The two main techniques for implementing HA at the server and application level are load balancing and clustering, which address different needs and are not interchangeable.

Load Balancing

Load Balancer
A load balancer is a device or software that distributes incoming traffic among multiple backend servers. Its primary objective is to optimize resource utilization and guarantee horizontal availability: if a server goes down, the balancer stops sending it traffic and redistributes it to the other active nodes. This way users experience no interruptions, as long as at least one node remains functional.
Load balancing algorithms
The main methods are: Round Robin (each request goes to the next server in cyclic order), Least Connections (prefers the server with the fewest active connections), IP Hash (always assigns the same client to the same server, useful for stateful sessions), and Weighted (more powerful servers receive more traffic). The choice of algorithm affects performance, session consistency, and fault tolerance.
Health Check
The load balancer periodically queries each backend node (ping, TCP check, HTTP check) to verify its health. If a node does not respond, it is automatically removed from rotation. This makes failover transparent to users: no manual intervention, no interruption perceived by the client.
Practical example — Load Balancing: an e-commerce site receives thousands of requests per second. Three web servers are behind a load balancer. Server 2 suffers a hardware failure at 14:05. The load balancer detects the health check failure within 10 seconds and stops sending it traffic: users continue browsing without noticing anything, distributed between servers 1 and 3. The IT team replaces server 2 and returns it to rotation without downtime.

Clustering

High Availability Cluster (HA Cluster)
An HA cluster is a group of servers (nodes) that share resources and monitor each other. Unlike load balancing, cluster nodes often share a disk or common storage resource and have an automatic failover mechanism: if the primary node goes down, a secondary node acquires the primary's IP address and resources and continues the service — sometimes without active connections being interrupted. This is essential for stateful applications such as databases, ERP systems, and file servers.
Active/Active vs Active/Passive
In an Active/Active cluster all nodes process traffic simultaneously, maximizing resource utilization. In an Active/Passive cluster, secondary nodes are on standby and become active only when the primary fails. Active/Passive has a higher cost (idle hardware) but a more predictable failover and consistent state.
Heartbeat
Cluster nodes continuously exchange liveness signals (heartbeats) over a dedicated network. If a node does not receive the primary's heartbeat within the threshold, it initiates the failover procedure. A misconfigured heartbeat can cause the split-brain scenario: both nodes believe themselves to be primary and attempt to acquire the same resources, risking data corruption.

PITFALL — Load Balancing ≠ Application Failover: load balancing distributes requests, but if the application itself is stateful (e.g., a session with temporary data on the server), failover to another node may cause loss of the user session. To avoid this, sticky sessions or an external shared session storage (e.g., Redis) are needed. Confusing load balancing with complete HA leads to wrong answers on the exam.
EXAM TIP 3.4-A: if a scenario says "web servers must handle traffic spikes while guaranteeing availability" → Load Balancing. If it says "the database must keep working even if the primary node fails without losing transactions" → HA Clustering. If it says both → load balancer in front + DB cluster behind.

RAID and Redundant Storage

RAID (Redundant Array of Independent Disks) is a technology that combines multiple physical disks to improve performance, fault tolerance, or both. It is not a substitute for backup: it protects against disk failure, not against accidental deletions, ransomware, or disasters.

RAID level Mechanism Fault tolerance Typical use
RAID 0 Striping (data split across multiple disks) None — one broken disk = data lost Pure performance, no resilience requirement
RAID 1 Mirroring (identical copy on 2 disks) Survives failure of 1 disk OS, critical logs, simple recovery
RAID 5 Striping + distributed parity (min. 3 disks) Survives failure of 1 disk Balance of capacity/performance/tolerance
RAID 6 Striping + double parity (min. 4 disks) Survives failure of 2 disks High-reliability enterprise storage
RAID 10 Mirror + striping (RAID 1+0) Survives multiple failures if not in the same mirror High-performance, high-availability databases
PITFALL — RAID ≠ Backup: RAID protects against hardware failure of a disk. It does not protect against ransomware (which encrypts all disks), accidental deletions, logical filesystem corruption, or physical loss of the site. A system with RAID 5 hit by ransomware still loses all data. Backups are mandatory even with RAID.

Site Considerations — Disaster Recovery Sites

When the entire primary site becomes inaccessible or operationally compromised — due to fire, flood, prolonged power outage, physical attack, or natural disaster — an organization must have an alternate site from which to resume critical operations. The three standard models are defined based on the time required to return to operations (RTO) and the associated maintenance cost.

Hot Site

Hot Site
A hot site is an alternate facility that is already fully operational: servers running, network configured, data synchronized in real time or near-real-time through replication. In the event of a disaster, staff can transfer immediately and resume operations within minutes or at most a few hours. It is the solution with the lowest RTO, but also the most expensive: it requires duplicate hardware, software licenses, redundant connectivity, and continuous management of the secondary site. Data is kept synchronized via synchronous or asynchronous replication, so RPO is near zero.

Warm Site

Warm Site
A warm site is a partially equipped site: the hardware and network infrastructure exists, but systems are not fully updated with the most recent data. When a disaster occurs, a period of hours or days is required to install updates, restore the most recent backups, and reconfigure services before returning to operation. The cost is intermediate: hardware and facilities are paid for, but not the continuous operational management typical of a hot site. RPO depends on how frequently backups are synchronized to the warm site.

Cold Site

Cold Site
A cold site provides only the physical infrastructure: space, electrical power, and network connectivity. There are no pre-installed servers or synchronized data. In the event of a disaster, the organization must purchase or transport hardware, install operating systems and applications, and restore backups. Recovery time is measured in days or weeks. It is the least expensive solution in terms of maintenance, suitable for organizations with high tolerance for downtime or for non-critical systems. It is essential, however, that documentation, recovery procedures, and backups are always up to date and available off-site.
Site type State at activation Typical RTO Typical RPO Cost
Hot Site Fully operational, data in sync Minutes – few hours Minimal (near zero) Very high
Warm Site Hardware ready, data partially updated Hours – days Hours (depends on backup freq.) Medium
Cold Site Physical structure only, no systems Days – weeks Depends on last backup Low
Practical example — Choosing the DR site: a bank with 24/7 transactions requires a hot site: every minute of downtime means lost transactions and reputational damage. A manufacturing company whose ERP systems are used only during the day shift might choose a warm site: it has a few hours to restore everything. A small professional firm that archives tax documents might settle for a cold site: it does not need to be operational within hours, and the cost of a hot site is unsustainable.
EXAM TIP 3.4-B: the exam often presents a scenario with a specific RTO requirement (e.g., "critical systems must be back online within 2 hours") and asks which type of DR site is most appropriate. The criterion is simple: low RTO → hot site; moderate RTO → warm site; high RTO or limited budget → cold site. Never choose a cold site if the scenario specifies an RTO of hours.

Geographic Dispersion

Geographic Dispersion
Geographic dispersion involves placing redundant systems, DR sites, and backups in physically distant locations from one another. The goal is to ensure that a single geographically localized event — an earthquake, a flood, a regional blackout, a physical attack on a datacenter — cannot simultaneously compromise both the primary site and the backup. The recommended minimum distance depends on the type of risk: for seismic risks, tens or hundreds of kilometers are discussed; for local risks (fire, flooding) a shorter distance may suffice.
Practical considerations on dispersion
Dispersion increases latency in communications between sites (important for synchronous database replication) and can complicate operational management. It must be balanced against the need for real-time replication: the greater the distance, the higher the network latency, and therefore the harder it is to maintain low RPOs with synchronous replication.
PITFALL — Geographic dispersion does not guarantee availability: having two datacenters in different countries does not equate to resilience if both depend on the same internet connectivity provider, the same cloud provider, or the same centralized authentication system. Physical dispersion must be accompanied by provider diversity (ISP, cloud, hardware) to be truly effective.

Platform Diversity and Multi-cloud

Platform diversity is a resilience strategy that consists of avoiding total dependence on a single vendor, operating system, cloud provider, or technology. The principle is that a zero-day vulnerability, a provider mass outage, or an unexpected commercial change cannot paralyze the entire organization.

Platform Diversity
Intentional use of heterogeneous systems: for example, balancing Windows and Linux servers, using hardware from different vendors for routers and switches, or distributing application workloads across different cloud environments. Diversity reduces the common attack surface: an exploit that works on one operating system does not automatically work on the other. On the other hand, it increases management complexity: staff must be able to work with different technologies and patching processes must cover heterogeneous environments.
Vendor Diversity
Use of different vendors for critical components. For example, having two different ISPs for internet connectivity (avoiding a single point of failure in the network), or using hardware from different vendors for border firewalls. This also applies to cloud providers: depending on a single cloud provider makes the organization vulnerable to that provider's outages (e.g., the AWS 2021 outage that made thousands of sites unreachable).

Multi-cloud Systems

Multi-cloud
A multi-cloud architecture uses services from at least two different public cloud providers (e.g., AWS + Azure, or GCP + Azure) to run application workloads. The main reasons are: avoiding vendor lock-in (contractual or technical dependence on a single provider), reducing the risk of provider outage, leveraging the best services of each provider for specific workloads, and meeting data sovereignty requirements that mandate using providers with datacenters in specific jurisdictions.
Multi-cloud complexity
A multi-cloud architecture requires unified management and monitoring tools, consistent identity and access policies across different providers, and attention to the security of integration APIs. The attack surface increases because each provider introduces its own configurations, its own IAM models, and its own potential vulnerabilities. The resilience advantage is only achieved if systems on different clouds are truly independent (separate data, not a single shared database).
Practical example — Multi-cloud for resilience: a video streaming company uses AWS for content delivery in North America and Azure for Europe. Main data is replicated between the two platforms. When in 2021 AWS us-east-1 went offline for hours, the company redirected American traffic to Azure: users experienced degraded performance but not a total service interruption.
EXAM TIP 3.4-C: when the exam asks which strategy reduces the risk of dependence on a single cloud vendor → Multi-cloud. When it asks how to reduce the risk that an exploit hits all systems simultaneously → Platform diversity. These are related concepts but with different applications.

Continuity of Operations (Continuity of Operations)

Continuity of Operations (COOP) is the set of processes, procedures, and structures that allow an organization to continue performing its critical functions during and after a disruptive event. It is not just a technical matter: it also includes planning for the people, processes, and infrastructure needed to sustain operations under adverse conditions.

The BCP (Business Continuity Plan) is the document that defines how the organization intends to maintain continuity. It includes the prioritization of critical functions, minimum operational requirements, alternate sites, communication procedures, and roles of responsibility in an emergency. Unlike the DRP (Disaster Recovery Plan) — which focuses on restoring IT systems — the BCP covers the entire business operation.

Critical functions
Not all business functions have the same priority. During planning, the organization must identify which processes, systems, and data are indispensable to survive an outage and in what order they must be restored. An accounting system might tolerate 48 hours of downtime; an online payment system cannot.
Recovery order
The recovery order establishes the sequence in which systems are restored during a disaster recovery event. The sequence must follow technical dependencies: first network and directory infrastructure (Active Directory, DNS), then databases, then applications that depend on the databases, then front-end services. Ignoring dependencies leads to failed restores or systems that start but do not function.
Non-persistent systems and gold images
Non-persistent systems (e.g., virtual desktops in VDI, ephemeral containers, auto-provisioned cloud environments) are designed to be destroyed and recreated from a reference image (gold master image) in case of compromise or failure. This greatly simplifies recovery: instead of restoring a complex backup, the certified image is relaunched. The security advantage is that any unauthorized changes (malware, misconfigurations) are eliminated on every reboot.

Capacity Planning

Capacity planning is the process of forecasting and managing the resources needed to sustain operations, both under normal conditions and during failover events or peak loads. If the DR site is activated but does not have sufficient capacity to sustain the primary site's load, operational continuity fails regardless.

People
People capacity planning concerns the availability of qualified personnel both under normal conditions and during an emergency. It includes: documenting procedures (because if the person who knows the procedure is unavailable, recovery stalls), cross-training so that multiple people can perform the same critical operations, and planning the chain of command during an emergency. A DR plan dependent on a single person creates a human SPOF.
Technology
Technological capacity planning ensures that hardware, software, licenses, and bandwidth are sufficient to sustain the operational load even during failover. A warm site with hardware undersized relative to the primary site cannot handle the full load: a bottleneck is created that effectively makes the backup site unusable under stress. The plan must account not only for average capacity, but also for foreseeable peaks (e.g., fiscal months, promotional campaigns).
Infrastructure
Infrastructure capacity planning covers physical space, electrical power, cooling, network connectivity, and storage. A backup datacenter must have PDUs (Power Distribution Units) and UPS sized for the expected actual load, sufficient cooling for the hardware, and adequate network bandwidth to handle production traffic. These elements have long procurement lead times: they cannot be added in an emergency.
EXAM TIP 3.4-D: capacity planning is tested in the exam with scenarios that describe a technically successful but operationally failed failover (e.g., "the DR site is active but applications are slow because servers are undersized"). The identified problem is a capacity planning deficiency in the Technology or Infrastructure dimension.

Testing Continuity and Recovery Plans

A business continuity or disaster recovery plan that is never tested is, in fact, a statement of intent with no practical value. Tests serve to verify that technical systems work as expected, that staff know the procedures, and that actual times (RTO, RPO) meet requirements. There are four main methodologies, ordered by cost, impact, and level of verification.

Tabletop exercise
A tabletop exercise is a structured discussion in which key stakeholders analyze a hypothetical scenario (e.g., "the primary datacenter has burned down") and verbally describe the actions they would take. No system is actually activated or modified. The value of this type of test lies in identifying gaps in procedures, ambiguities in roles, and communication deficiencies, at virtually zero cost and with no risk to production. It is the method with the lowest impact, but the lowest level of technical verification.
Failover Test
A failover test actually executes the switchover procedure to the backup site or to redundant systems, verifying that automatic or manual failover occurs within the expected timeframes. It can be performed in a planned manner (with scheduled maintenance) or as a surprise to verify that automatic systems respond correctly. The failover test validates the actual RTO: if the manual says "restore in 2 hours" but the test requires 6 hours, the plan must be revised. It is more disruptive than a tabletop: it can cause brief interruptions if poorly managed.
Simulation
A simulation replicates a disaster event more realistically than a tabletop, involving operational teams in executing procedures in a controlled environment (not in production). For example, a ransomware attack can be simulated on test systems, executing the isolation, notification, and recovery process as if it were real. Simulation reveals procedural and technical problems that a tabletop cannot surface, but requires more resources and planning.
Parallel Processing Test
In a parallel processing test, the backup site is brought online and critical systems are run simultaneously on both the primary and secondary sites. Results are compared to verify that the backup site produces identical output. It is the most expensive and complex test (requires double resources for the duration of the test), but also the most realistic: it verifies not only that the DR site starts up, but that applications produce correct results. It allows identification of data synchronization or configuration problems before the backup site is actually needed.
Test type Real systems activated Service interruption Verification level Cost
Tabletop No None Procedural / communication Minimal
Failover Yes (DR site) Possible brief interruption Technical + failover time Medium
Simulation Yes (test environment) Test environment only Procedural + technical Medium-high

EXAM TIP 3.4-E: the exam clearly distinguishes the four test types. If the scenario says "no system can be interrupted or put at risk" → Tabletop. If it says "we want to verify that the DR site is actually functional without risking production" → Parallel Processing. If it says "we want to measure the actual failover time" → Failover Test.

Data Backup and Recovery

Backup is the foundation of data protection and recovery. A backup is a copy of data made at a specific point in time, stored in a way that allows restoration in case of loss, corruption, accidental deletion, or attack (e.g., ransomware). Designing an effective backup strategy requires decisions about: frequency, type, location, encryption, and restore test procedures.

Backup Types

Full Backup
A full backup copies the entire dataset on every run, regardless of previous changes. It is the simplest to restore — only a single backup file is needed — but requires more storage space and more time for each run. It is typically performed weekly or less frequently, combined with incremental or differential backups on intermediate days.
Incremental Backup
An incremental backup copies only the data modified since the last backup performed (full or incremental). It is fast and requires little space for each individual run. The problem arises at restore time: to reconstruct the full state, the full backup plus all subsequent incremental backups must be applied in sequence. If any of the incrementals is corrupted or missing, the restore stops.
Differential Backup
A differential backup copies all data modified since the last full backup, regardless of how many differential backups have been performed in the meantime. Each differential is therefore larger than the corresponding incremental (because it accumulates all changes since the full), but restore requires only two elements: the full backup plus the last differential. It offers a good compromise between storage space and recovery simplicity.
Type What is copied Space required Backup speed Restore complexity
Full Entire dataset Maximum Slow Minimal (one file)
Incremental Changes since last backup (full or incr.) Minimum per run Very fast High (full + all incrementals)
Differential All changes since last full Medium (grows over time) Medium Medium (full + last diff.)
PITFALL — Incremental vs Differential: the most common confusion is forgetting the reference point. Incremental is compared to the last backup performed (of any type). Differential is always compared only to the last full backup. This difference is almost always tested as a multiple-choice question on the exam.

Backup Location: Onsite and Offsite

Onsite Backup
On-site backups offer fast recovery: data is nearby, bandwidth is not an issue, and restore times are minimal. The problem is that a catastrophic event affecting the primary site (fire, flood, theft) destroys both the original systems and their backups. Onsite backups protect against hardware failures or accidental deletions, not against physical disasters.
Offsite Backup
Off-site backups are stored in a geographically separate location — a second datacenter, a cloud storage provider, a physical tape storage facility. They protect against disasters that affect the entire site. The limitation is recovery speed: transferring tens of terabytes over the internet or waiting for physical tape delivery can take hours or days, extending RTO. For this reason many organizations adopt the 3-2-1 rule: 3 copies of data, on 2 different media, with 1 off-site.
Backup frequency
The frequency with which backups are performed directly determines RPO: the more frequent the backup, the less data can be lost. A daily backup implies an RPO of 24 hours (at most one day of work is lost). An hourly backup lowers RPO to 60 minutes. The optimal frequency depends on the criticality of the data and the volume of daily changes: performing full backups every hour on petabyte datasets is technically impractical without specialized backup systems.

Snapshots and Images

Snapshot
A snapshot is a point-in-time photograph of the system, typically implemented at the hypervisor or storage level. Unlike a traditional backup (which physically copies all files), a snapshot records only the data blocks that change after the snapshot itself (copy-on-write technique). This makes them extremely fast to create and inexpensive in terms of initial space. They are widely used in virtualized environments before updates or configuration changes: if the update goes wrong, revert to the snapshot in a few minutes. The limitation is that snapshots depend on the original storage system: if the datastore is compromised, the snapshots are also lost.
System Image
A system image is a complete and independent copy of the disk or volume, including operating system, applications, configurations, and data. Unlike a snapshot, an image can be restored to different hardware or a different cloud environment. Images are essential for disaster recovery of critical systems (e.g., DNS servers, domain controllers) and for maintaining gold master images from which non-persistent systems (VDI, cloud instances) are generated.
PITFALL — Snapshot ≠ Backup: a snapshot is not an independent backup. If the datastore or original disk is corrupted, deleted, or destroyed, the snapshots are also lost. Snapshots are operational tools for fast recovery from recent errors, not a long-term protection strategy. They must be accompanied by real backups copied to another location.

Replication

Synchronous replication
In synchronous replication, every write operation on the primary system is considered complete only when it has been confirmed on the secondary system as well. This guarantees that at no point are the secondary's data behind the primary: RPO ≈ zero. The cost is the latency added by each write: the primary system must wait for the secondary's confirmation. It works well for short distances (low network latency); over large geographic distances, latency can become unacceptable for applications with high write volumes.
Asynchronous replication
In asynchronous replication, write operations on the primary are completed immediately without waiting for the secondary's confirmation. Data is replicated with a delay (lag). This allows greater geographic distances and better primary performance, but introduces an RPO greater than zero: in case of sudden disaster, the most recent changes not yet replicated are lost.
Replication vs Backup
Replication maintains an up-to-date real-time copy, ideal for reducing RPO. But if data is accidentally deleted or encrypted by ransomware, replication immediately carries the corruption to the secondary system as well. Replication protects against hardware failures, not against human errors or logical attacks. Backups with retention policies (historical versions of data) are the only tool that allows returning to a state prior to the corruption.

Journaling

Filesystem journaling
Journaling is a feature of modern filesystems (ext4, NTFS, XFS) that records planned operations in a journal (log) before actually performing them on disk. If the system goes offline unexpectedly (power outage, crash) while a write was in progress, on reboot the filesystem checks the journal and completes or cancels incomplete operations in a few seconds. This prevents filesystem corruption that on non-journaled systems required hours of scanning (fsck). Journaling is not a backup: it does not allow recovering previous versions of files, but ensures that the filesystem remains in a consistent state.
Database journaling / Write-Ahead Logging (WAL)
Relational databases use a similar mechanism called Write-Ahead Logging: each transaction is first written to the log (journal) and only then applied to the tables. In case of a crash, the database recovers by applying or rolling back the transactions recorded in the log. WAL also enables replication: the primary's log is sent to the secondary which applies it, maintaining a synchronized copy without full copy overhead. PostgreSQL, Oracle, and SQL Server all use variants of this approach.
Practical example — Journaling in action: a PostgreSQL database is processing 500 transactions per second. At 03:22 a UPS runs out and the server shuts down abruptly while 12 transactions were partially written. On reboot, PostgreSQL reads the WAL: the 12 incomplete transactions are identified and rolled back (automatic rollback). The database is consistent and operational in less than 30 seconds, without requiring manual intervention.

Backup Security

Backup encryption
Backups must be encrypted — both in transit and at rest — because they often contain the organization's entire information assets. An unencrypted backup stored on an off-site tape or on a cloud service is a data exfiltration vector if intercepted or stolen. Encryption must be applied before transfer and must use modern standards (AES-256). Encryption key management is critical: a lost key makes the backup unrecoverable.
Backup encryption key management
Keys used to encrypt backups must be stored separately from the backups themselves — ideally in a Key Management System (KMS) or in an HSM. If the key is stored together with the encrypted backup (e.g., same folder, same cloud credential), encryption adds no real protection: anyone who accesses the backup also accesses the key. In a disaster recovery scenario, the recovery process must include access to the key as a documented step.
Backup restore testing
An untested backup is not a reliable backup: it may be corrupted, incomplete, incompatible with current hardware, or protected by a lost key. Restore tests must be performed periodically in an isolated test environment, verifying that data is complete, intact, and restorable within the RTO timeframes. Many organizations discover that their backups are unusable only when they actually need them.
EXAM TIP 3.4-F: the exam may present scenarios where a backup exists but cannot be used. The most common causes are: unencrypted backups with a lost key, untested backups that turned out to be corrupted, onsite-only backups destroyed along with the primary site, backups that do not include all data (e.g., open databases excluded). Identifying the specific problem in the scenario is the key to choosing the correct answer.

Redundant Power (Power)

Electrical power is one of the most critical and often underestimated dependencies of infrastructure resilience. A power interruption — even for a few seconds — can cause data loss, filesystem corruption, hardware failures (disks mid-write, unflushed memory), and service outages. Power resilience is achieved through two complementary technologies: UPS and generators.

UPS — Uninterruptible Power Supply

UPS (Uninterruptible Power Supply)
A UPS is a device that provides instant power from batteries when the main power fails. The key to the term is "uninterruptible": unlike a generator (which takes seconds to start), the UPS kicks in within milliseconds, with no perceived interruption by electronic systems. UPS batteries provide autonomy typically from a few minutes to tens of minutes, depending on capacity. This time serves to: keep systems operational while the generator starts, perform a controlled shutdown of systems if the generator is not available, and protect against brief interruptions without needing to start any generator.
UPS types
UPS units fall into three main categories: Offline/Standby (intervene only when they detect an interruption, with a brief switchover time of 10-20ms), Line-Interactive (also handle voltage fluctuations through an autotransformer), and Online/Double Conversion (power always passes through the batteries, offering complete protection against all types of electrical anomalies: interruptions, surges, line noise). Datacenters almost always use Online UPS for critical systems.
Autonomy and sizing
UPS sizing depends on the load to be powered (in Watts or VA) and the required autonomy. An overloaded UPS provides less autonomy than expected. In modern datacenters, UPS units are often modular (batteries are added as needed) and redundant (N+1: one extra module beyond what is necessary, so that failure of one module does not interrupt power delivery).

Generators

Generator
A generator powers critical systems during prolonged power outages — events that exceed UPS autonomy. Typical datacenter generators are diesel or natural gas powered and can sustain high electrical loads for hours or days. Startup time from cold is typically 10-30 seconds, during which UPS units keep systems operational. Automatic transfer (ATS, Automatic Transfer Switch) detects the interruption and starts the generator without manual intervention.
Fuel and autonomy
Diesel generators require periodic refueling: during prolonged events (hurricanes, regional blackouts), access to fuel supplies can become critical. Continuity plans must include priority fuel supply contracts and calculate how many hours of autonomy the tank provides when full. Some datacenters maintain 48-72 hours of fuel at full load; for longer duration events, refueling agreements are needed.
Periodic generator testing
Diesel generators must be started and loaded periodically (typically monthly) to verify their operation and prevent mechanical problems from prolonged inactivity. A generator that does not start when needed completely nullifies the investment. Tests must include verification of the automatic transfer (ATS) and simulation of real loads, not just no-load starts.

Practical example — UPS + Generator in a datacenter: at 02:14 the external power grid fails due to a transformer substation fault. Within 8 milliseconds the datacenter UPS units take the load: servers notice nothing. Power monitoring detects the outage and signals the ATS. After 18 seconds the diesel generator starts and reaches the correct voltage. The ATS switches the load from the UPS to the generator. Systems remain operational for the next 36 hours until the utility restores the line. Without the UPS, those 18 seconds of generator startup would have caused a crash of all servers.
EXAM TIP 3.4-G: UPS and generators are tested together in "power outage" scenarios. Remember: the UPS covers the gap between the outage and generator startup. If a scenario describes a 20-minute outage without a generator but with a UPS → the UPS alone is not enough for prolonged outages. If a scenario describes a generator that starts but servers crashed in the 20 seconds before → the UPS was missing.

Summary flashcards — CH9

  • High Availability (HA) — Property of a system designed to minimize unplanned downtime through redundancy and automatic failover.
  • Load Balancing — Distribution of traffic across multiple servers; the balancer automatically excludes non-functioning nodes. Ideal for stateless apps.
  • HA Clustering — Group of servers that monitor each other and pass resources on failover. Required for stateful apps such as databases.
  • RAID 1 — Mirroring: identical copy on 2 disks. Survives failure of 1 disk. Not a substitute for backup.
  • RAID 5 — Striping with distributed parity (min. 3 disks). Survives failure of 1 disk, balances performance and redundancy.
  • Hot Site — Fully operational DR site, synchronized data. RTO: minutes. Cost: maximum.
  • Warm Site — DR site with hardware ready but data partially updated. RTO: hours/days. Cost: medium.
  • Cold Site — Physical structure only, no installed systems. RTO: days/weeks. Cost: minimum.
  • RPO (Recovery Point Objective) — Maximum acceptable amount of data loss, expressed as a time interval. Determines backup frequency.
  • RTO (Recovery Time Objective) — Maximum acceptable time to restore a service after an outage.
  • 3-2-1 backup rule — 3 copies, on 2 different media, with 1 off-site. Minimum standard for data protection.
  • Incremental backup — Copies only changes since the last backup (of any type). Fast, but complex recovery (full + all incrementals).
  • Differential backup — Copies all changes since the last full. Simple recovery (full + last diff.). Progressive space growth.
  • Snapshot — Point-in-time photograph of the system. Fast to create, depends on original storage. Not an independent backup.
  • Synchronous replication — Each write is confirmed on the secondary before completing. RPO ≈ zero, but introduces latency.
  • Asynchronous replication — Writes are replicated with a delay. Better performance and tolerance for geographic latency, but RPO > 0.
  • Filesystem journaling — Records planned operations before executing them; prevents filesystem corruption on crash. Not a backup.
  • Tabletop exercise — Structured discussion of a hypothetical scenario. No systems activated. Minimal cost, verifies procedures and communication.
  • Parallel Processing Test — Both sites operational in parallel, outputs compared. Most complete and expensive test; verifies real DR functionality.
  • UPS — Provides instant battery power (milliseconds). Covers the gap during generator startup. Autonomy: minutes.
  • Generator — Extended power (hours/days) during blackout. Startup: 10-30 seconds. Requires fuel and periodic maintenance.
  • Platform Diversity — Use of heterogeneous systems (different OS, vendors, clouds) to reduce the risk that a single exploit or failure affects everything.
  • Multi-cloud — Use of at least two different cloud providers. Eliminates vendor lock-in and reduces risk of single provider outage.
  • Capacity Planning — Planning of people, technology, and infrastructure to sustain operations even in failover and peak load scenarios.
