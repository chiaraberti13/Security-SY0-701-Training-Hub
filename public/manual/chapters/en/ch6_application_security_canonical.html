Objective 4.1 — Application Security: Techniques and Secure Development

Application security is not built only by patching vulnerabilities after they are found; it must be designed from the beginning. Objective 4.1 requires you to understand the concrete techniques used to integrate security into every development phase and every implementation choice, from development environments to production controls. The key message is that security is a continuous process, not a one-time activity performed at the end of a project.

SDLC and code deployment environments

The Software Development Life Cycle (SDLC) describes the entire life of an application: from the first idea to retirement. For security professionals, the value of the SDLC is that it shows where specific controls should be introduced. Adding a security requirement during analysis is far cheaper than fixing an architectural vulnerability in production.

Planning
Feasibility analysis, comparison of alternatives and cost estimation. Security contributes by defining regulatory constraints, data-classification requirements and high-level risks.
Requirements definition
Collection of functional and non-functional requirements. Security requirements belong here: authentication, authorization, error handling, logging and encryption for data at rest and in transit must be documented before the architecture is finalized.
Design
Definition of architecture, data flows, integrations and structural choices. Threat models are created in this phase to identify attack surfaces, critical components and preventive controls.
Development
Code implementation. Developers apply secure coding practices such as input sanitization, safe APIs and proper error handling. Code review and unit testing can begin here.
Testing
Formal testing, including integration, regression and User Acceptance Testing (UAT). Security tests such as SAST, DAST and penetration testing belong here or are distributed across the CI/CD pipeline.
Training and transition
User training and deployment into production. This includes operational security documentation and handover procedures.
Operations and maintenance
The longest phase: patching, updates and changes. Vulnerabilities discovered after release must re-enter the cycle as new requirements. Formal change management protects production from uncontrolled modifications.
End-of-life
Application retirement. This requires secure data deletion, credential revocation, dependency decommissioning and final documentation.
EXAM TIP 4.1-A — When the exam asks in which SDLC phase security should be included, the correct answer is from the beginning, especially during requirements definition. “Only during testing” is almost always wrong.

Code deployment environments

Organizations separate code environments so development and testing activities do not directly affect production systems.

EnvironmentPurposeSecurity relevance
DevelopmentActive code developmentRestricted to developers; should not contain real production data.
Test / QAFunctional and quality validationIsolated environment for security testing; anonymized or synthetic data.
StagingPre-production environmentShould mirror production; final checkpoint before release; change management required.
ProductionReal operational environmentMost restricted access; continuous monitoring; every change requires formal approval.

DevOps, DevSecOps and CI/CD pipelines

DevOps combines software development and IT operations to accelerate releases through automation, collaboration and continuous feedback. DevSecOps extends this model by making security a shared responsibility across developers, operations and security teams, distributed throughout the pipeline instead of concentrated in a final checkpoint.

In a DevSecOps pipeline, security controls are automated: SAST runs on commits, DAST runs against builds, dependencies and containers are scanned before deployment, and risky changes can fail the pipeline automatically.

Continuous Integration (CI)
Developers integrate code frequently into a shared repository. Every integration triggers automated builds and tests to detect defects early.
Continuous Delivery / Deployment (CD)
Code that passes tests is automatically made ready for release, or deployed to production automatically depending on the organization’s release model.
Practical example — DevSecOps pipeline: A team uses GitHub Actions. On every push to the main branch, static code analysis runs, dependencies are scanned for known CVEs, a container image is built and checked, and deployment occurs only if all security gates pass.
WARNING — CI/CD and security: automation accelerates delivery, but it can also accelerate the distribution of vulnerabilities. A compromised dependency or malicious update can reach production quickly if the pipeline does not verify packages, signatures and dependency integrity.

Secure coding practices — OWASP Proactive Controls

OWASP (Open Web Application Security Project) publishes free standards, guides and tools for application security. The OWASP Proactive Controls are practices developers should integrate actively into software projects, not reactive fixes added after incidents.

Define security requirements
Security belongs in requirements analysis. Authentication, authorization, sensitive-data handling and logging must be documented before code is written.
Use secure frameworks and libraries
Do not reimplement cryptography, hashing or session management. Use mature, reviewed libraries.
Secure database access
Use parameterized queries and stored procedures to prevent SQL injection. Apply least privilege to database accounts.
Encode and escape data
Data sent to HTML, SQL, shell or LDAP contexts must be encoded for that specific context to prevent injection.
Validate all inputs
Every external value must be considered untrusted. The safest approach is an allow list: explicitly define what is permitted and block everything else.
Implement identity and sessions correctly
Use strong authentication, safe password storage with salted hashes, random session tokens and session expiration.
Use default-deny access control
If a user is not explicitly allowed to perform an action, the action is denied.
Protect data at rest and in transit
Use encryption for sensitive data in storage and TLS for communications. Do not transmit sensitive information in cleartext.
Security logging and monitoring
Record relevant events in centralized, tamper-resistant logs and monitor them actively.
Handle errors safely
User-facing errors should be generic. Stack traces, SQL queries and system paths belong in internal logs, not client responses.

API Security

APIs (Application Programming Interfaces) allow clients, servers and applications to exchange structured requests and responses. APIs power microservices, mobile applications, third-party integrations and cloud platforms, so exposing an API without proper controls can expose the entire business logic of an application.

API authentication and authorization
Every API request must come from a verified identity, such as an API key, OAuth 2.0 token or signed JWT. Authentication identifies the caller; authorization determines what the caller may do.
Rate limiting
Limits the number of requests per unit of time to reduce scraping, brute force attempts and service abuse.
Limited data exposure
An API should return only the fields required for the requested function, not entire database objects.
Input and output validation
Unvalidated input can lead to injection; improperly encoded output can lead to XSS or information disclosure.
Channel and server protection
TLS is mandatory for API communications. The server hosting the API must be hardened like any production server.
EXAM TIP 4.1-B — API security scenarios usually test authentication, authorization and rate limiting. If an API is reachable without credentials or returns too much data, focus on those controls first.

Input validation

Input validation is one of the most fundamental application security controls. Every value coming from outside the application — user input, another API, files or databases — is untrusted until proven otherwise. The application must verify that the input matches the expected format before processing it, passing it to another system or storing it.

Client-side validation improves user experience but is completely bypassable. Security validation must always happen server-side.

Allow list
The strongest approach: explicitly define permitted format, length, characters and value range. Everything else is rejected.
Deny list
Blocks known dangerous patterns. It is weaker because attackers can use encodings, case changes and alternative syntax to bypass it.
Parameter pollution
An attack technique that sends the same parameter multiple times, such as ?id=1&id=2. If the application validates one value but uses another, the filter can be bypassed.
Example — Allow list vs Deny list: A deny list blocks <script>, DROP TABLE and OR 1=1, but attackers can change casing or encoding. An allow list that accepts only [a-zA-Z0-9_] for usernames is much stronger.

Secure cookies

Cookies often store session identifiers. If a session cookie is stolen, an attacker can impersonate the user without knowing the password. Secure cookie attributes reduce this risk.

AttributePurpose
SecureCookie is sent only over HTTPS.
HttpOnlyJavaScript cannot read the cookie, reducing the impact of XSS.
SameSiteControls whether cookies are sent with cross-site requests, helping reduce CSRF.
Expires / Max-AgeLimits cookie lifetime.

Code analysis: SAST, DAST and Fuzzing

SAST — Static Application Security Testing
Analyzes source code or compiled code without running the application. It is useful early in development and can find insecure functions, injection patterns and hard-coded secrets.
DAST — Dynamic Application Security Testing
Tests a running application from the outside. It behaves more like an attacker and can find runtime issues, authentication problems and misconfigurations.
Fuzzing
Sends large volumes of unexpected, malformed or random input to discover crashes, memory errors and validation weaknesses.
TechniqueWhen usedStrengthLimitation
SASTBefore executionFinds issues early in codeMay produce false positives
DASTAgainst running appShows real exploitable behaviorCannot see source code
FuzzingDuring testingFinds unexpected crashesNeeds good instrumentation and time

Code signing

Code signing uses a digital signature to prove that software came from a trusted publisher and has not been modified since signing. It protects against tampered installers, malicious updates and unauthorized scripts. Code signing provides integrity and publisher authentication, but it does not prove that the signed code is bug-free.

EXAM TIP 4.1-C — If the scenario asks how to verify that software has not been altered and comes from the expected vendor, choose code signing.

Sandboxing

A sandbox executes code in an isolated environment with restricted access to the host system. It limits what an application can read, write, execute or communicate with. Sandboxing is used by browsers, mobile operating systems, malware analysis labs and application isolation tools.

The main benefit is containment: if the code behaves maliciously, its ability to affect the rest of the system is limited.

Application monitoring

Application monitoring collects logs, metrics and events to detect abnormal behavior and support incident response. Relevant events include authentication failures, privilege changes, API abuse, unusual error rates, access to sensitive records and unexpected outbound connections.

Monitoring is effective only when logs are centralized, protected from tampering and actively reviewed through alerting or SIEM correlation.

Objective 2.3 (partial) — Application and Web Vulnerabilities

Application vulnerabilities arise from coding errors, weak design, unsafe dependencies or insecure configuration. Web applications are especially exposed because they accept input from untrusted users and often connect directly to sensitive databases and identity systems.

Memory injection

Memory injection inserts malicious code into the memory space of a legitimate process. Because the malicious code runs inside a trusted process, it can bypass simple allow lists or signature-based controls. Mitigations include EDR monitoring, least privilege, code integrity controls and memory protections such as ASLR and DEP.

Buffer overflow

A buffer overflow occurs when an application writes more data into a buffer than it can hold, overwriting adjacent memory. Attackers may use this to change program execution flow and run malicious code. Mitigations include bounds checking, safe languages, stack canaries, ASLR and DEP/NX.

EXAM TIP 2.3-A — If the scenario says a process writes beyond allocated memory, the answer is usually buffer overflow.

Race conditions and TOCTTOU

A race condition happens when system behavior depends on the timing of events. TOCTTOU means Time-of-Check to Time-of-Use: the application checks a resource, but an attacker changes it before it is used. Defenses include atomic operations, file locking, safe temporary-file handling and revalidating permissions immediately before use.

Malicious update

A malicious update is software distributed through an update mechanism that appears legitimate but contains malicious code. It may come from a compromised vendor, poisoned dependency or hijacked distribution channel. Defenses include signed updates, package integrity verification, dependency review and supply-chain monitoring.

SQL Injection (SQLi)

SQL injection occurs when user input is concatenated into a database query and executed as SQL code. An attacker can read, modify or delete data, bypass authentication or sometimes execute operating-system commands through the database layer. The root cause is failure to separate data from code.

Defenses against SQLi

  • Parameterized queries / prepared statements: treat user input as data, never as SQL instructions.
  • Stored procedures: safe when they use parameters correctly and do not concatenate SQL internally.
  • Input validation: reject values that do not match expected formats.
  • Least privilege: the application database account should not have unnecessary administrative rights.

Cross-Site Scripting (XSS)

XSS allows an attacker to inject script into a page viewed by another user. The script executes in the victim’s browser under the trust of the vulnerable site. XSS can steal session tokens, modify page content, perform actions as the user or redirect the victim to malicious pages.

Stored XSS
The malicious script is stored on the server, for example in a comment or profile field, and affects every user who views it.
Reflected XSS
The script is reflected immediately in the response, often through a malicious link.
DOM-based XSS
The vulnerability is in client-side JavaScript that writes untrusted data into the page DOM.

Defenses against XSS

  • Context-aware output encoding for HTML, JavaScript, URLs and attributes.
  • Input validation using allow lists where possible.
  • Content Security Policy (CSP) to restrict where scripts can load from.
  • HttpOnly cookies to prevent JavaScript from reading session cookies.

Other related application attacks

Cross-Site Request Forgery (CSRF)
Tricks a logged-in user’s browser into sending an unwanted request to a trusted site. Defenses include anti-CSRF tokens and SameSite cookies.
Directory traversal
Uses sequences such as ../ to access files outside the intended directory. Defenses include canonical path validation and allow-listed file access.
Command injection
Injects operating-system commands into application input. Defenses include avoiding shell execution, parameterized APIs and strict input validation.
Insecure deserialization
Occurs when untrusted serialized objects are processed and can trigger code execution or logic abuse. Defenses include signed objects, safe formats and strict type validation.
Server-Side Request Forgery (SSRF)
Forces the server to send requests to internal or external resources chosen by the attacker. Defenses include egress filtering, metadata-service protections and URL allow lists.

Review flashcards

Where should security enter the SDLC?
At the beginning, especially during requirements and design.
SAST vs DAST?
SAST analyzes code without running the application; DAST tests a running application from the outside.
Best defense against SQL injection?
Parameterized queries / prepared statements.
Best defense against XSS?
Context-aware output encoding, supported by CSP and secure cookie flags.
Why use code signing?
To verify software integrity and publisher authenticity.
Why use sandboxing?
To isolate code execution and limit damage if the code behaves maliciously.
What is TOCTTOU?
A race condition where a resource changes between permission check and actual use.
What does HttpOnly protect?
It prevents JavaScript from reading a cookie, reducing session theft through XSS.
