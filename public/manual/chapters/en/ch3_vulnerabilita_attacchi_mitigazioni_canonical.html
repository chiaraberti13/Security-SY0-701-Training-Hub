Objective 2.3 — Types of vulnerabilities

A vulnerability is a weakness in a system, its design, or its configuration that an attacker can exploit to compromise confidentiality, integrity, or availability. For the exam, the category of the vulnerability matters because each category points to different mitigations. A threat is the actor or event, risk is likelihood multiplied by impact, and a vulnerability is the weakness that exists before anyone exploits it.

Application vulnerabilities

Application vulnerabilities come from mistakes in software code, architecture, or input handling. They are common because applications are frequently exposed to users and networks, and they often change faster than operating systems.

Memory Injection
A technique in which malicious code is inserted into the memory space of an already running legitimate process. Because the injected code runs under the identity of the host process, it may bypass signature-based controls, allow lists, and sandbox rules. It is commonly used for persistence, privilege escalation, and credential theft. Address Space Layout Randomization (ASLR) helps mitigate it by making process memory locations harder to predict.
Buffer Overflow
A condition in which an application writes more data into a buffer than the buffer can hold, overwriting adjacent memory such as a stack return address. An attacker may redirect execution toward malicious shellcode. Mitigations include input validation, bounds checking, stack canaries, DEP/NX, ASLR, and memory-safe languages such as Rust or Go.
Race Condition — Time-of-Check (TOC) and Time-of-Use (TOU)
A race condition occurs when system behavior depends on the sequence or timing of events. In a TOC/TOU scenario, the application checks a condition, such as file permissions, and later uses the resource after an attacker has changed it. This pattern is common with temporary files, symbolic links, and shared resources.
Malicious Update
A software update that appears legitimate but contains malicious code. It can result from a compromised vendor update server, a poisoned build process, or a tampered package. Mitigations include code signing, package integrity verification, trusted repositories, and supply-chain monitoring.
Practical example — Buffer Overflow: SQL Slammer exploited a buffer overflow in Microsoft SQL Server. A single small UDP packet was enough to compromise a vulnerable host and allow the worm to spread extremely quickly across the Internet.
Exam tip 2.3-A — If the scenario describes a process writing past the limits of an allocated buffer, the answer is buffer overflow. If it describes a permission check followed later by use of a changed resource, the answer is TOC/TOU.
Warning — Memory Injection vs Buffer Overflow: Buffer overflow overwrites adjacent memory to redirect execution. Memory injection places code inside an existing process and does not necessarily require an overflow.

OS-based vulnerabilities

OS-based vulnerabilities affect the kernel, device drivers, system services, and exposed APIs. They are dangerous because successful exploitation often grants high privileges, making application-level controls less effective. The primary mitigation is disciplined patch management, removal of unnecessary services, least privilege for system processes, and secure baseline enforcement.

Exam tip 2.3-B — If the scenario mentions a kernel flaw, vulnerable driver, outdated OS, or unsupported system service, the category is OS-based and the primary mitigation is timely patching or compensating isolation when patching is not possible.

Web vulnerabilities (SQLi and XSS)

SQL Injection (SQLi)
An attacker inserts SQL statements into user input that the application concatenates into a database query. If executed, the attacker may read, modify, or delete data, bypass authentication, or sometimes execute commands on the host. The root cause is failure to separate data from code. The primary mitigation is prepared statements or parameterized queries.
Cross-Site Scripting (XSS)
An attacker injects script, usually JavaScript, into a page viewed by other users. In reflected XSS, the payload is delivered through a crafted URL. In stored XSS, the payload is saved in the application database and affects every user who views the infected content. Primary mitigations are output encoding, input validation, secure frameworks, and Content Security Policy (CSP).

Hardware vulnerabilities

Firmware
Low-level software embedded in devices such as routers, switches, printers, BIOS/UEFI, and controllers. Firmware vulnerabilities are serious because they operate below the OS and may survive OS reinstallation. Mitigations include Secure Boot, firmware signing, trusted updates, and hardware inventory management.
End-of-Life (EOL)
A component that no longer receives security support from the vendor. It may still function, but newly discovered vulnerabilities remain unpatched. Mitigation is replacement; when replacement is not immediately possible, use compensating controls such as segmentation, strict ACLs, and increased monitoring.
Legacy
Old hardware or software still in use because replacement is costly, complex, or operationally risky. Unlike EOL, a legacy system may still receive limited support, but its design may not meet modern security expectations. It often requires isolation and compensating controls.
Warning — EOL vs Legacy: EOL means official support and patches have ended. Legacy means outdated but still operational, and sometimes still supported. The exam often uses EOL when no vendor patch exists.

Virtualization vulnerabilities

VM Escape
An attack in which malicious code inside a guest virtual machine breaks isolation and interacts with the hypervisor or host operating system. If successful, the attacker may affect other VMs on the same physical host. Mitigations include hypervisor patching, administrative access control, and separating sensitive workloads.
Resource Reuse
A risk in virtualized or cloud environments where memory, storage, or CPU cache used by one tenant is reassigned to another without being properly sanitized. The next tenant may read residual data. Mitigations include secure wiping, memory zeroing, and provider controls for multi-tenant isolation.
Exam tip 2.3-C — In a multi-tenant cloud scenario, concern that another customer may read leftover memory or disk data points to resource reuse, not VM escape. VM escape requires an active process breaking out of the VM.

Cloud-specific vulnerabilities

Cloud deployments introduce risks that are less common in traditional on-premises environments. Misconfiguration is the most common cloud breach cause, but other concerns include poorly protected management APIs, excessive IAM permissions, public object storage, lack of logging, insecure key management, and dependence on third-party SaaS/PaaS providers. The shared responsibility model clarifies who owns each control, but organizations often misunderstand the boundary between provider and customer responsibilities.

Supply chain vulnerabilities

The security supply chain includes the components, vendors, libraries, build systems, and processes that make IT systems work. A weakness introduced anywhere in the chain can affect downstream customers. SolarWinds is the classic example: a legitimate software update was compromised during the build process and delivered a backdoor to many organizations.

Service Provider
A managed service provider or IT vendor often has privileged access to customer networks. If the provider is compromised, the attacker may pivot into many customer environments. Mitigations include vendor risk assessment, contract security clauses, least privilege, and monitoring of third-party access.
Hardware Provider
Hardware components may contain backdoors or vulnerabilities introduced during manufacturing or distribution. Mitigations include trusted suppliers, hardware integrity checks, secure procurement, and tamper-evident processes.
Software Provider
Open-source libraries, frameworks, and third-party software components may contain vulnerabilities such as Log4Shell in Log4j. A Software Bill of Materials (SBOM) helps track dependencies and identify exposure quickly.

Cryptographic vulnerabilities

Cryptographic vulnerabilities may come from weak algorithms, short keys, poor certificate validation, predictable random numbers, insecure modes, fixed initialization vectors, padding oracle weaknesses, or flawed implementations. Strong algorithms used incorrectly can be as dangerous as weak algorithms. Mitigation requires using modern standards, following NIST and vendor guidance, validating implementations, and disabling deprecated protocols and ciphers.

Misconfiguration

Misconfiguration is one of the most common real-world vulnerability categories. Examples include unchanged default credentials, unnecessary open ports, excessive permissions, overprivileged service accounts, incorrect firewall rules, disabled logs, and encryption not enabled for data at rest or in transit. Misconfiguration does not require a software bug; it is a management or operational error. Mitigations include configuration management, secure baselines, CIS Benchmarks, and periodic configuration scanning.

Practical example — Misconfiguration: An AWS S3 bucket accidentally configured for public access exposes files to the Internet without authentication. The resource may be technically working, but the security configuration is wrong.

Mobile device vulnerabilities

Side Loading
Installing mobile applications outside the official app store, usually through APK or IPA files. It bypasses store review and code checks. Sideloaded apps may contain malware, spyware, or trojanized versions of legitimate apps.
Jailbreaking / Rooting
Removing vendor restrictions on iOS or Android to gain root-level control. This weakens protections such as sandboxing, secure storage, integrity attestation, and MDM enforcement. A rooted or jailbroken device should not be trusted for sensitive enterprise access.
Warning — Side Loading vs Jailbreaking: Side loading is the method used to install an app outside the official store. Jailbreaking or rooting changes the device security model. Android can allow sideloading without root, while iOS usually requires stronger bypasses.

Zero-day

A zero-day is a vulnerability unknown to the affected vendor, or one for which no official patch is available. The vendor has had zero days to prepare. Zero-days are valuable because they can compromise fully updated systems. Since patching is not available at first, mitigation relies on compensating controls such as segmentation, application allow lists, behavioral EDR, least privilege, and anomaly monitoring.

Exam tip 2.3-D — If no vendor patch exists yet, do not choose patching as the immediate mitigation. Choose compensating controls such as segmentation, monitoring, EDR, or application allow listing.

Objective 2.4 — Indicators of malicious activity

This objective requires analyzing scenarios and recognizing which attack category matches the observed indicators. Categories include malware, physical attacks, network attacks, application attacks, cryptographic attacks, password attacks, and general indicators of compromise (IOCs). The exam often provides narrative clues and asks you to classify the attack.

Malware attacks

Ransomware
Malware that encrypts files or locks access to a system and demands payment, usually in cryptocurrency. Modern double extortion combines encryption with the threat of publishing stolen data. The best mitigation is immutable offline backups tested regularly.
Trojan
Malware disguised as legitimate software, such as a game, utility, or fake update. Unlike worms, trojans do not self-propagate; the victim runs them voluntarily. They may install backdoors, steal credentials, download payloads, or join a botnet.
Worm
Self-propagating malware that spreads across networks by exploiting exposed services without requiring user interaction. Worms are dangerous in poorly segmented networks because one infected host can compromise many others very quickly.
Spyware
Software designed to collect user information without consent, such as browsing history, credentials, screenshots, and usage patterns. In enterprises, spyware can quietly exfiltrate customer data or trade secrets.
Bloatware
Unwanted software preinstalled by a manufacturer or software distributor. It is not necessarily malicious, but it consumes resources, increases attack surface, may contain vulnerabilities, and can violate enterprise application policies.
Virus
Malicious code that replicates by attaching to legitimate executable files or boot sectors. It requires execution of an infected host file, unlike a worm that spreads by itself across the network.
Keylogger
Software or hardware that records keystrokes to capture passwords, PINs, and sensitive messages. MFA is a strong mitigation because a stolen password alone is not enough to authenticate.
Logic Bomb
Dormant malicious code that activates when a condition is met, such as a date, event, or user removal. It is often associated with insider threats and requires code review and change monitoring to detect.
Rootkit
Malware that hides its presence while maintaining privileged access. Kernel-level rootkits may hide processes, files, registry keys, and network connections. Removal often requires booting from clean media or reimaging the system.

Practical example — Logic Bomb: A system administrator adds a scheduled script that deletes backups every Saturday at 3:00 a.m. only if the administrator's Active Directory account is no longer in the Employees group. After termination, the condition becomes true and the backups are silently deleted.
Exam tip 2.4-A — A worm moves by itself through the network. A virus needs an infected host file. A trojan is the deceptive vehicle. A rootkit hides by manipulating the OS, while spyware focuses on collecting data.

Physical attacks

Physical Brute Force
Unauthorized physical access through force, such as breaking locks, forcing doors, or removing panels. Mitigations include hardened locks, layered access control, barriers, guards, cameras, and intrusion sensors.
RFID Cloning
An attacker reads an employee badge with a portable RFID reader and clones it onto another card. Weak RFID cards transmit static identifiers without challenge-response protection. Mitigations include smart cards, encrypted RFID, anti-passback, and MFA at sensitive areas.
Environmental
Attacks or failures involving power, temperature, humidity, water, fire, or physical destruction. Security controls include UPS systems, generators, fire suppression, HVAC monitoring, leak detection, and environmental sensors.
Exam tip 2.4-B — Physical brute force is material force against a facility or device, not password guessing. RFID cloning points to badge duplication and weak proximity-card controls.

Network attacks

DDoS — Distributed Denial-of-Service
A coordinated attack using many systems to overwhelm a target with traffic, consuming bandwidth, CPU, memory, or application resources. Mitigations include CDN protection, rate limiting, scrubbing centers, autoscaling, and upstream filtering.
DDoS Amplified
An attacker sends small spoofed requests to services such as DNS or NTP, causing much larger responses to be sent to the victim. The attacker abuses the difference between request size and response size.
DDoS Reflected
Traffic is reflected through third-party servers so the victim sees responses from legitimate systems rather than the real attacker. Reflected and amplified DDoS often occur together.
DNS Attacks
Attacks against name resolution, including DNS spoofing, cache poisoning, zone transfer abuse, tunneling, domain hijacking, and DDoS against DNS infrastructure. Mitigations include DNSSEC, restricted zone transfers, monitoring, and redundant DNS providers.
On-Path Attack (Man-in-the-Middle)
An attacker positions themselves between two communicating parties to intercept or modify traffic. Examples include ARP poisoning, rogue access points, SSL stripping, and compromised proxies. Mitigations include TLS, HSTS, certificate validation, VPNs, and secure Wi-Fi.
Credential Replay
An attacker captures valid authentication material and reuses it later. MFA, nonce-based protocols, short-lived tokens, and replay protection mitigate this risk.
Malicious Code (via network)
Malware delivered or executed through network services, shared folders, malicious downloads, drive-by compromise, or remote exploitation. Controls include IDS/IPS, secure web gateways, EDR, segmentation, and patching.

Application attacks

Injection
Untrusted input is interpreted as code or commands. SQL injection, command injection, LDAP injection, and XML injection all share the same root cause: failure to separate data from instructions.
Application Buffer Overflow
Application code writes beyond allocated memory and may allow code execution or crash the process. Memory-safe languages and runtime protections reduce exposure.
Replay Attack
An attacker captures a legitimate request and resubmits it. Replay attacks are mitigated with nonces, timestamps, sequence numbers, token binding, and short-lived sessions.
Privilege Escalation
An attacker gains higher privileges than originally granted. Vertical escalation means gaining administrative rights; horizontal escalation means accessing another user's resources at the same privilege level.
Forgery — CSRF and SSRF
Cross-Site Request Forgery tricks an authenticated user's browser into sending unwanted requests. Server-Side Request Forgery forces a server to make requests to internal or external resources chosen by the attacker.
Directory Traversal (Path Traversal)
An attacker uses sequences such as ../ to access files outside the intended directory. Mitigations include canonical path validation, allow lists, and avoiding direct user-controlled file paths.
Practical example — Directory Traversal: If a vulnerable download endpoint accepts ?file=../../../../etc/passwd, the attacker may read system files outside the intended document directory.

Cryptographic attacks

Downgrade Attack
An attacker forces two systems to use an older, weaker protocol or cipher suite. Mitigation is to disable deprecated protocols and enforce strong TLS versions and cipher suites.
Collision Attack
An attacker finds two different inputs that produce the same hash value. Weak hashes such as MD5 and SHA-1 are vulnerable and should not be used for integrity-sensitive purposes.
Birthday Attack
A statistical attack against hash functions based on the birthday paradox. It shows why a hash with n bits offers roughly n/2 bits of collision resistance.

Password attacks

Password Spraying
Trying a small number of common passwords across many accounts to avoid account lockout. The indicator is many failed logins across many users, often with the same password attempt.
Brute Force (password)
Trying many password guesses against one or a few accounts until the correct password is found. It often triggers lockouts and is mitigated by MFA, lockout thresholds, rate limiting, and strong passwords.
Warning — Password Spraying vs Brute Force: Spraying uses a few passwords against many accounts. Brute force uses many passwords against one account.
Exam tip 2.4-C — If many accounts fail authentication once or twice with the same password, think password spraying. If one account has hundreds or thousands of failures, think brute force.

Indicators of compromise (IOC)

Indicators of compromise are observable signs that malicious activity may be occurring or has already occurred. They do not prove an incident by themselves, but they provide clues for investigation and correlation.

Account Lockout
Repeated failed logins causing an account to lock. It may indicate brute force, password guessing, or credential stuffing.
Concurrent Session Usage
The same account is active in multiple locations or devices at the same time. It may indicate credential sharing or compromise.
Blocked Content
Security tools block malicious downloads, web content, email attachments, or scripts. Blocks are useful signals even when prevention succeeds.
Impossible Travel
A user logs in from two geographically distant locations within an impossible time window. This strongly suggests credential theft or proxy abuse.
Resource Consumption
Unexpected CPU, memory, disk, or bandwidth usage may indicate malware, cryptomining, data exfiltration, or runaway processes.
Resource Inaccessibility
Files, applications, or network resources become inaccessible. This may indicate ransomware, DoS, storage failure, or permission tampering.
Out-of-Cycle Logging
Logs appear at unusual times or outside expected maintenance windows. This may indicate unauthorized activity or automated attack behavior.
Published/Documented
Compromise evidence appears in threat intelligence, paste sites, dark web listings, breach notifications, or vendor advisories.
Missing Logs
Expected logs are absent. This may indicate log deletion or tampering by an attacker with elevated privileges. Centralized immutable logging helps prevent it.
Exam tip 2.4-D — Missing logs are suspicious. They may be more important than noisy logs because attackers often delete evidence after gaining privileged access.

Objective 2.5 — Enterprise mitigation techniques

Mitigation techniques reduce the likelihood or impact of attacks. The exam expects you to choose the control that directly addresses the weakness described in the scenario rather than a generic security tool.

Segmentation

Segmentation separates networks, systems, and data into controlled zones using VLANs, subnets, firewalls, ACLs, and microsegmentation. It limits lateral movement and reduces the blast radius of a compromise.

Practical example — Segmentation: Guest Wi-Fi, payment systems, administrative workstations, and production servers should not share the same unrestricted network. A firewall or access control layer should define exactly which flows are allowed.

Access Control — ACLs and Permissions

Access Control List (ACL)
A rule set that permits or denies access to resources, files, network flows, or objects. Network ACLs filter traffic; file ACLs define who can read, write, or execute a file.
Permissions
Specific rights assigned to users, groups, processes, or systems. Permission design should follow least privilege and should be reviewed periodically to prevent permission creep.

Application Allow List

An application allow list permits only approved software to run and blocks everything else by default. It is highly effective against unknown malware and zero-days because unapproved code cannot execute even if it has no known signature.

Exam tip 2.5-A — Against unknown malware or zero-day executables, application allow listing is usually stronger than traditional signature-based antivirus.

Isolation and Patching

Isolation
Separating a system from other systems or networks to reduce risk. It is especially useful for infected hosts, legacy systems, EOL devices, or systems that cannot be patched immediately.
Patching
Applying vendor updates to fix known vulnerabilities. Patch management should define maintenance windows, testing, prioritization by severity, rollback plans, and reporting.

Encryption and Monitoring

Encryption
Protects confidentiality for data at rest and data in transit. It limits breach impact because stolen encrypted data is unusable without the key. Backups should also be encrypted.
Monitoring
Continuous observation of endpoints, networks, identities, and cloud services. SIEM, EDR, NDR, and UEBA help detect indicators of compromise that would otherwise remain invisible.

Least Privilege, Configuration Enforcement, and Decommissioning

Least Privilege
Users, processes, and systems should have only the permissions needed for legitimate tasks. This reduces the damage caused by a compromised account or service.
Configuration Enforcement
Automatically maintaining secure baselines across systems with tools such as Group Policy, Ansible, Puppet, Chef, or Desired State Configuration. It prevents configuration drift and unauthorized changes.
Decommissioning
The secure retirement of systems, hardware, software, certificates, and credentials that are no longer needed. It includes data sanitization, credential revocation, inventory updates, and formal documentation.

Hardening Techniques

Hardening reduces attack surface by removing, disabling, or strengthening everything that is not strictly required. A hardened system exposes fewer services, runs less unnecessary software, uses stronger configuration, and is easier to monitor.

Encryption (on systems)
Includes disk encryption, encrypted backups, encrypted internal service communication, and disabling protocols that transmit credentials in cleartext such as Telnet, FTP, and HTTP basic authentication.
Installation of Endpoint Protection
Deploying endpoint security tools such as EDR to provide behavioral telemetry, anomaly detection, remote response, host isolation, process termination, and visibility into endpoint activity.
Host-Based Firewall
A software firewall on an individual endpoint that controls inbound and outbound traffic independently from the network. It provides host-level protection even on untrusted networks.
HIPS — Host-Based Intrusion Prevention System
Monitors OS and application activity on a host and actively blocks suspicious behavior such as system-file modification, process injection, registry tampering, and privilege escalation attempts.
Disabling Ports/Protocols
Closing unused TCP/UDP ports and disabling insecure protocols reduces exposed services and removes unnecessary attack vectors.
Default Password Changes
Changing vendor default credentials during deployment prevents easy unauthorized access using publicly known passwords.
Removal of Unnecessary Software
Removing software that is not required for the system's function reduces vulnerabilities, patch burden, and attack surface.

Practical example — Combined hardening: A Linux Apache web server is hardened by removing unnecessary modules, exposing only port 443, disabling Telnet and FTP, changing root credentials, enforcing SELinux, using Fail2Ban, installing EDR, encrypting disks with LUKS, and removing unnecessary packages.
Exam tip 2.5-B — To reduce exploitation of unnecessary services, choose disabling ports/protocols or removing unnecessary software. For an EOL system that cannot be patched, choose isolation and compensating controls such as ACLs and segmentation.

Summary flashcards

  • Buffer Overflow vs Memory Injection — Buffer overflow overwrites adjacent memory; memory injection inserts code into an existing process.
  • TOC vs TOU — Time-of-Check is when a condition is verified; Time-of-Use is when the resource is used. The gap enables the race condition.
  • SQLi primary mitigation — Prepared statements and parameterized queries separate data from SQL instructions.
  • Reflected vs Stored XSS — Reflected payloads are delivered through a link; stored payloads persist in the application and affect many users.
  • VM Escape vs Resource Reuse — VM escape breaks out of a guest VM; resource reuse exposes residual tenant data.
  • Zero-day mitigation — Use compensating controls: segmentation, allow listing, behavioral EDR, monitoring, and least privilege.
  • Worm vs Virus vs Trojan — Worms self-propagate; viruses need host files; trojans disguise themselves as legitimate software.
  • Logic Bomb — Dormant code triggered by a date, event, or condition, often associated with insider threats.
  • Rootkit detection — Tools inside the compromised OS may be unreliable; use clean boot media or reimage.
  • Password Spraying vs Brute Force — Spraying uses few passwords against many accounts; brute force uses many passwords against one account.
  • DDoS Amplified — Small spoofed requests cause much larger responses to the victim.
  • On-Path / MitM — The attacker intercepts traffic between parties; mitigate with TLS, HSTS, VPNs, and certificate validation.
  • Impossible Travel — Logins from geographically impossible locations indicate likely credential compromise.
  • Missing Logs — Missing logs may indicate tampering; centralized immutable logging helps prevent evidence loss.
  • Application Allow List vs Antivirus — Allow lists block everything not approved; antivirus mainly blocks known malicious signatures.
  • Least Privilege — Limits blast radius by restricting accounts and services to only needed permissions.
  • HIPS vs EDR — HIPS actively blocks host behavior; EDR focuses on detection, visibility, and response.
