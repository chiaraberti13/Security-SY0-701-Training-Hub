Objective 1.4 — Appropriate Cryptographic Solutions

Cryptography is the discipline that transforms readable information into a form that is unintelligible to anyone who does not possess the correct key. Its value is not limited to confidentiality: a modern cryptographic system can support confidentiality, integrity, authentication, and non-repudiation. These goals are not always provided by the same mechanism, so the exam focuses on choosing the correct tool for the exact security requirement.

A core principle, known as Kerckhoffs's principle, states that the security of a cryptographic system should depend on the secrecy of the key, not on the secrecy of the algorithm. Modern systems such as AES, RSA and TLS follow this model: the algorithms are public and reviewed by the security community. Hiding the algorithm, often called security through obscurity, is weak and not acceptable as a primary protection strategy.

EXAM TIP 1.4: Scenario questions usually describe a need such as protecting sensitive data on a stolen laptop, proving that a contract was signed, or detecting whether a file was modified. Answer by matching the mechanism to the requirement: hashing for integrity, encryption for confidentiality, digital signatures for integrity plus non-repudiation, and PKI certificates for trusted identity binding.

Symmetric Cryptography

In symmetric cryptography, the same secret key is used to encrypt and decrypt data. This approach is computationally efficient and is therefore ideal for protecting large volumes of data such as backups, encrypted file systems, VPN sessions and databases.

The main weakness is key distribution: before two parties can communicate securely, they must already share the same secret through a protected channel. In a group of n users, the number of required shared keys grows as n(n−1)/2, which makes the model difficult to scale. Because both parties possess the same key, symmetric encryption also cannot provide non-repudiation.

CharacteristicSymmetricAsymmetric
SpeedVery highLower, often much slower
Key distributionDifficult; a secure channel is required firstEasier; the public key can be distributed
ScalabilityPoor in large groupsHigh; one key pair per user
Non-repudiationNoYes, when used for digital signatures

EXAM TIP — Symmetric vs asymmetric: If the scenario asks for the fastest way to encrypt a 1 TB disk, choose symmetric encryption such as AES. If it asks how to securely send a symmetric key to another party, use asymmetric encryption to protect the key. In practice, TLS uses both: asymmetric mechanisms for authentication and key exchange, then symmetric encryption for the session.

Asymmetric Cryptography

Asymmetric cryptography assigns each entity a mathematically related key pair: a public key, which can be distributed freely, and a private key, which must remain secret. What one key encrypts can only be decrypted by the other key. This model solves the key distribution problem because parties do not need to share a secret in advance.

Public key
A key that can be shared openly, published in a certificate, or placed on a public server. It is used to encrypt data for the owner or to verify a digital signature created by the owner.
Private key
A key that must never be shared. It is used to decrypt data encrypted with the matching public key or to create digital signatures. If the private key is compromised, the trust relationship is broken.
Key pair
The public and private keys together. The keys are mathematically linked, but the private key cannot be feasibly derived from the public key when a secure algorithm and adequate key length are used.
Practical example — secure email: Alice wants Bob to read a confidential message. She encrypts the message with Bob's public key. Only Bob's private key can decrypt it. If Alice also signs the message with her private key, Bob can verify the signature using Alice's public key.

RSA and ECC

RSA is one of the most widely known asymmetric algorithms and is based on the difficulty of factoring very large numbers. It can be used for encryption, key exchange and digital signatures, but it requires large keys to remain secure and is slower than symmetric algorithms.

Elliptic Curve Cryptography (ECC) provides comparable security with much smaller key sizes. This makes ECC especially useful for mobile devices, IoT systems and environments where processing power, memory or bandwidth is limited. For the exam, remember the general comparison: RSA is common and mature; ECC is efficient and uses shorter keys for equivalent strength.

Key Exchange

Key exchange mechanisms allow two parties to establish a shared secret over an untrusted network. The shared secret is then used as the symmetric session key. This is essential because symmetric encryption is fast, but the secret key still needs to be established safely.

Diffie-Hellman
A method that allows two parties to derive the same shared secret without transmitting the secret itself. A passive listener can observe the exchange but cannot feasibly calculate the resulting key.
ECDHE
Elliptic Curve Diffie-Hellman Ephemeral. It uses temporary keys for each session and provides perfect forward secrecy: compromise of a long-term private key does not decrypt past sessions.
Perfect Forward Secrecy
A property that protects previous encrypted sessions even if a server's private key is later compromised. It is achieved through ephemeral session keys.
PITFALL — encryption vs key exchange: Diffie-Hellman does not encrypt the application data itself. It establishes the shared secret that will later be used by a symmetric encryption algorithm.

Algorithms and Key Length

Algorithm selection and key length determine how resistant a cryptographic implementation is to brute force and cryptanalytic attacks. A strong algorithm with a weak key length is still weak; a long key used in a deprecated algorithm can also be unacceptable.

EXAM TIP — Deprecated algorithms: DES, MD5 and SHA-1 are red flags in exam scenarios. They may appear in legacy systems, but they should not be selected for new secure implementations.

Encryption at Rest

Encryption at rest protects data stored on a device or storage medium: laptops, mobile devices, databases, backups, file shares, cloud object storage and removable drives. Its primary goal is confidentiality if the storage is lost, stolen or accessed without authorization.

Full-disk encryption
Encrypts the entire storage device, including the operating system and user files. It is appropriate for laptops and mobile endpoints that may be lost or stolen.
File-level encryption
Encrypts individual files or folders. It is more granular than full-disk encryption and can remain effective even when files are copied elsewhere.
Database encryption
Protects stored records or tables. It is commonly combined with strict access control, key management and auditing.
Practical example — stolen laptop: If a company laptop containing customer data is stolen, full-disk encryption prevents the thief from reading the data by removing the drive and mounting it on another computer.

Encryption in Transit — Transport and Communication

Encryption in transit protects data while it is moving across a network. It prevents eavesdropping and tampering by attackers positioned between the sender and the receiver. Typical protocols include TLS for web traffic, SSH for remote administration, IPSec for VPNs and S/MIME for secure email.

ProtocolTypical useSecurity purpose
TLSHTTPS and application trafficConfidentiality, integrity and server authentication
SSHRemote administrationSecure command-line access and file transfer
IPSecVPN tunnelsNetwork-layer encryption and authentication
S/MIMEEmail securityEmail encryption and digital signatures
PITFALL — at rest vs in transit: BitLocker or FileVault protect data at rest. TLS and IPSec protect data in transit. Choose based on where the data is when the risk occurs.

Hashing

A hash function converts input data of any size into a fixed-length value called a hash, digest or message digest. Hashing is one-way: the original data should not be recoverable from the hash. Hashing is used to verify integrity, store password representations and support digital signatures.

Integrity check
If even one bit of the original file changes, a secure hash should produce a very different digest. This makes hashing useful for verifying downloaded files, forensic images and configuration baselines.
Collision resistance
A secure hash function should make it computationally infeasible to find two different inputs that produce the same hash.
Preimage resistance
Given a hash value, it should be infeasible to reconstruct an input that produces that hash.
EXAM TIP — Hashing: If the scenario asks for integrity without confidentiality, hashing is enough. If it asks to keep data secret, hashing is not encryption and is not the right answer.

Salting and Key Stretching

A salt is a random value added to a password before hashing. Salting prevents attackers from using precomputed rainbow tables and ensures that two users with the same password still have different password hashes.

Key stretching deliberately slows down password hashing by applying many iterations or using memory-hard functions. This makes brute-force attacks more expensive. Common examples include PBKDF2, bcrypt, scrypt and Argon2.

Practical example — password storage: A secure application does not store plaintext passwords. It stores a salted and stretched password hash. During login, the submitted password is salted and hashed in the same way, and the result is compared with the stored value.

Digital Signatures

A digital signature proves that data was signed by the holder of a private key and that the signed content has not changed. The sender hashes the data and signs the hash with the private key. The receiver verifies the signature using the sender's public key.

Integrity
If the signed content changes, signature verification fails.
Authentication
The signature proves that the private key corresponding to the public key was used.
Non-repudiation
When the private key is properly protected and bound to an identity, the signer cannot reasonably deny having signed the data.
EXAM TIP — Signature vs encryption: Encrypting with a public key protects confidentiality. Signing with a private key provides integrity, authentication and non-repudiation. These are different goals.

HMAC

A Hash-based Message Authentication Code (HMAC) combines a cryptographic hash function with a shared secret key. It provides integrity and authentication between parties that already share a secret. Unlike a digital signature, HMAC does not provide non-repudiation because both parties know the same secret key.

PITFALL — HMAC vs digital signature: HMAC is symmetric and proves that someone with the shared secret produced the value. A digital signature is asymmetric and can support non-repudiation.

Public Key Infrastructure (PKI)

A Public Key Infrastructure is the system of technologies, policies, roles and procedures used to create, distribute, validate and revoke digital certificates. PKI solves a trust problem: how can a user know that a public key really belongs to the claimed entity?

Certificate
A signed data structure that binds a public key to an identity such as a domain name, person, device or organization.
Certificate Authority (CA)
A trusted entity that issues and signs certificates. Clients trust certificates because they trust the CA or a chain leading to a trusted root CA.
Registration Authority (RA)
An entity that verifies the identity of certificate requesters before the CA issues the certificate.
Chain of trust
The certificate path from an end-entity certificate through one or more intermediate CAs to a trusted root CA.

Key Escrow

Key escrow is the practice of storing a copy of a cryptographic key with a trusted third party or internal recovery system. It is used when an organization must be able to recover encrypted data if a user leaves, loses a key or becomes unavailable.

Escrow introduces a trade-off: it improves recoverability but creates a highly sensitive target. Escrowed keys must be protected with strong access controls, separation of duties, logging and approval workflows.

Digital Certificates

A digital certificate contains identity information, a public key, validity dates, usage constraints, issuer information and the issuer's digital signature. In TLS, a server presents its certificate so the client can verify that it is communicating with the legitimate server.

FieldPurpose
SubjectThe entity the certificate identifies.
Subject Alternative NameDNS names or other identities covered by the certificate.
Public keyThe key bound to the subject identity.
IssuerThe CA that issued and signed the certificate.
Validity periodThe start and expiration dates.
Key usageDefines permitted uses such as digital signature or key encipherment.

Certificate Authorities and Root of Trust

A root CA is the top of a trust hierarchy. Root certificates are self-signed and are trusted because they are preinstalled in operating systems, browsers or enterprise trust stores. Intermediate CAs are signed by the root CA and are commonly used to issue end-entity certificates, reducing exposure of the root private key.

EXAM TIP — Root CA: The root CA's private key is one of the most sensitive assets in a PKI. It should be kept offline or protected with very strong controls. Day-to-day certificate issuance is normally delegated to intermediate CAs.

Certificate Revocation List (CRL) and OCSP

Certificates may need to be revoked before their expiration date if a private key is compromised, the subject changes, a certificate was issued incorrectly or the certificate should no longer be trusted.

CRL — Certificate Revocation List
A periodically published list of revoked certificates. Clients download or query the list to check whether a certificate has been revoked. CRLs can become large and may not provide real-time status.
OCSP — Online Certificate Status Protocol
A protocol that allows clients to query the revocation status of a specific certificate. OCSP is more targeted than downloading an entire CRL.
OCSP stapling
A method where the server includes a recent OCSP response during the TLS handshake, improving privacy and performance.

Certificate Signing Request (CSR) Generation

A Certificate Signing Request is generated by the entity requesting a certificate. The CSR contains the public key and identifying information. The private key is created locally and must never be sent to the CA. The CA validates the request and signs a certificate if the request is approved.

  1. Generate a private key locally.
  2. Create a CSR containing the public key and subject information.
  3. Submit the CSR to a CA or internal PKI.
  4. Complete identity or domain validation.
  5. Install the issued certificate with the matching private key.
PITFALL — CSR and private key: A CSR does not include the private key. If the private key is uploaded or emailed, the certificate's security is compromised.

Tools — TPM, HSM, Key Management System and Secure Enclave

TPM — Trusted Platform Module
A hardware component used to store keys securely, support measured boot and protect full-disk encryption keys. TPMs are common in laptops and desktops.
HSM — Hardware Security Module
A dedicated, tamper-resistant hardware device used to generate, store and use cryptographic keys. HSMs are common in enterprise PKI, payment systems and high-assurance environments.
KMS — Key Management System
A service or platform used to create, rotate, store, control and audit cryptographic keys. Cloud providers offer managed KMS services.
Secure Enclave
A protected execution environment designed to isolate sensitive operations and secrets from the main operating system.
EXAM TIP — HSM vs TPM: TPM is normally endpoint-oriented and tied to a device. HSM is enterprise-grade dedicated hardware for protecting high-value keys and cryptographic operations.

Obfuscation — Steganography, Tokenization and Data Masking

Obfuscation makes information harder to understand or locate, but it is not the same as encryption. It may reduce exposure or hide meaning, yet it should not be treated as a substitute for strong cryptographic controls when confidentiality is required.

Steganography
Hides data inside another object, such as concealing a message inside an image or audio file. The goal is to hide the existence of the message.
Tokenization
Replaces sensitive data with a non-sensitive token. The original value is stored separately in a protected token vault. It is common in payment processing.
Data masking
Displays only part of a value, such as showing the last four digits of a credit card number. It reduces exposure for users who do not need to see the full value.

Blockchain and Open Public Ledger

A blockchain is a distributed ledger made of blocks linked by cryptographic hashes. Each block contains a hash of the previous block, which makes tampering evident: changing an earlier block changes its hash and breaks the chain. Public blockchains replicate the ledger across many participants and use consensus mechanisms to agree on the valid state.

For Security+, focus on the security concepts rather than cryptocurrency speculation: hashing provides integrity, digital signatures authorize transactions, distributed consensus reduces reliance on a single central authority, and immutability makes historical tampering difficult. Blockchain does not automatically provide confidentiality; data written to a public ledger may be visible to everyone.

Summary Flashcards — Objective 1.4

Fastest method for encrypting large data volumes?
Symmetric encryption, typically AES.
Mechanism for integrity without confidentiality?
Hashing.
Mechanism for integrity, authentication and non-repudiation?
Digital signature.
Why use salting?
To make identical passwords produce different hashes and defeat rainbow tables.
What does PKI solve?
It binds public keys to trusted identities through certificates and certificate authorities.
CRL vs OCSP?
CRL is a published list of revoked certificates; OCSP checks the status of a specific certificate.
TPM vs HSM?
TPM protects keys on an endpoint; HSM is dedicated enterprise hardware for high-value keys.
Does blockchain guarantee confidentiality?
No. It mainly provides integrity, distributed validation and tamper evidence.
