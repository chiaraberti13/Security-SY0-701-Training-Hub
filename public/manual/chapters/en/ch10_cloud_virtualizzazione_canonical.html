Cloud service models (3.1)

Cloud computing is a model for delivering IT resources over the Internet based on five essential characteristics defined by NIST: on-demand self-service, broad network access, resource pooling, rapid elasticity, and measured service. These characteristics radically change how infrastructure is purchased, sized, and protected: provisioning that once took weeks now happens in minutes, but this speed brings with it risks of misconfiguration, uncontrolled sprawl, and loss of visibility over where data resides.

Service models are distinguished by how much infrastructure the provider manages and how much remains under the customer's control. Precise understanding of this boundary is fundamental not only for designing a secure architecture, but also for correctly answering exam questions that ask "who is responsible for what."

IaaS — Infrastructure as a Service
The provider offers the basic infrastructure: physical servers, storage, networking, and hypervisor. The customer is responsible for everything else — operating system, OS patches, middleware, runtime, applications, and data. It is the model with maximum flexibility and maximum responsibility burden for the customer. Examples: AWS EC2, Azure Virtual Machines, Google Compute Engine. The main risk is misconfiguration: a poorly configured security group exposes instances directly to the internet.
PaaS — Platform as a Service
The provider also manages the operating system, runtime, and middleware; the customer handles only application code and data. PaaS accelerates development by eliminating server management, but reduces visibility and control. Examples: Azure App Service, Google App Engine, Heroku. The main risk is that vulnerabilities in the platform provided by the provider cannot be patched independently by the customer.
SaaS — Software as a Service
The provider delivers a complete, fully managed application. The customer controls nothing of the underlying infrastructure and has limited visibility even over application configurations. Examples: Microsoft 365, Google Workspace, Salesforce. The main risk concerns identity management, granular permissions, and data residency.
FaaS / Serverless Computing
The customer runs individual code functions in response to events, without managing servers. The provider automatically allocates and deallocates resources. Examples: AWS Lambda, Azure Functions, Google Cloud Functions. See the dedicated serverless section for security details.

PITFALL — IaaS vs PaaS
In IaaS the customer must patch the operating system. In PaaS the provider manages it, but the customer cannot control the timing. The exam often asks: "who is responsible for patching the OS?" — the answer depends on the service model. Do not generically answer "the provider."

Responsibility Matrix — Shared Responsibility Model

The Shared Responsibility Model formally establishes the division of security duties between the Cloud Service Provider (CSP) and the customer. The underlying principle is that the provider is always responsible for security of the cloud (hardware, hypervisor, physical datacenters), while the customer is always responsible for security in the cloud (data, identities, configurations of the resources they use). The exact boundary shifts based on the service model adopted.

The Responsibility Matrix is a practical tool — often in the form of a table or checklist — that lists every component or security area and indicates who is responsible. Frameworks such as the Cloud Controls Matrix (CCM) from the Cloud Security Alliance (CSA) provide a standardized mapping of cloud security controls organized by domain, allowing comparison of responsibilities across multiple providers.

Practical example — Exposed AWS S3 bucket
A company uses Amazon S3 (SaaS-like for storage, but technically IaaS). AWS guarantees the availability and physical security of the storage. However, the configuration of the bucket's Access Control List is the customer's responsibility. If the customer leaves the bucket publicly accessible, AWS does not intervene: the breach is the customer's fault. This error, known as "S3 bucket misconfiguration," is one of the most common causes of cloud data breaches.
Exam Tip — Responsibility Matrix
Typical question: "An organization uses a SaaS application. Who is responsible for patching the underlying operating system?" — Answer: the provider. "Who is responsible for ensuring that users only have the permissions they need?" — Answer: the customer (identity management is always the customer's responsibility in all models). When you see "data" and "identity" in the options, these are almost always the customer's responsibility, regardless of the model.

Cloud distribution models

Public Cloud
Infrastructure shared among multiple customers (multi-tenant), entirely managed by the provider and accessible via the internet. Offers reduced costs, immediate scalability, and zero hardware investment. The main risk is multi-tenancy: a misconfiguration could lead to data leakage between tenants. Examples: AWS, Azure, Google Cloud Platform.
Private Cloud
Cloud infrastructure dedicated to a single customer's use, managed internally or by a dedicated provider. Offers greater control, physical isolation, and the ability to meet stringent regulatory requirements (e.g., healthcare, defense). Cost is higher because resources are not shared. Can be hosted on-premises or in a provider's datacenter.
Community Cloud
Infrastructure shared only among organizations that belong to a community with common requirements (same regulation, same sector, same mission). Examples: cloud for government entities, cloud for hospitals subject to HIPAA. Balances the savings of sharing with the need for control and compliance.
Hybrid Cloud
Coherent and coordinated combination of two or more cloud environments (public, private, community) or of cloud and on-premises, linked by technologies that allow portability of data and applications. See the "Hybrid Considerations" section for security details.
Exam Tip — Distribution models
The exam may ask which model to choose given a requirement. Quick rule: if the question mentions "regulation" or "sensitive data" → Private Cloud. If it mentions "multiple organizations with shared requirements" → Community Cloud. If it mentions "flexibility + existing on-premises resources" → Hybrid Cloud. If none of these conditions apply → Public Cloud.

Hybrid Considerations

A hybrid environment introduces specific security complexities that do not exist in purely on-premises or purely cloud models. The main areas of attention are:

Secure connectivity between environments
Traffic between the on-premises environment and the public cloud must be encrypted and authenticated. Typical solutions are site-to-site VPN (encrypted connection over the internet) or Direct Connect / ExpressRoute (dedicated private connection that does not pass through the internet). The second option offers predictable latency and greater security, but has a higher cost.
Federated identity management
In a hybrid environment, users must authenticate to both on-premises resources (typically via Active Directory) and cloud resources. Identity federation via protocols such as SAML, OAuth 2.0, or OIDC allows using a single authentication system. Without federation, a second identity silo is created that is difficult to manage and protect.
Consistent security posture
One of the main risks of hybrid environments is that security policies defined on-premises do not automatically apply to cloud resources. Every control — firewall, logging, patch management, encryption — must be explicitly extended or reconfigured in the cloud environment. Cloud Security Posture Management (CSPM) tools help detect cloud configurations that do not comply with organizational policies.
Latency and data availability
Applications that ping between on-premises and cloud at high frequency may suffer from excessive latency. It is important to define during the design phase which workloads remain on-premises and which migrate to the cloud, avoiding "chatty" architectures that generate constant traffic between the two environments.
Practical example — Hybrid Active Directory with Azure AD
A company has on-premises Active Directory and uses Microsoft 365 (SaaS) for email. With Azure AD Connect, identities are synchronized: users authenticate once with their corporate credentials and access both internal resources and Microsoft 365. If an account is disabled in AD, the disabling automatically propagates to Azure AD within the next synchronization cycle.

Third-Party Vendors and Cloud Governance

Adopting the cloud almost always involves relying on third-party vendors for critical parts of the infrastructure or application. This does not eliminate the organization's responsibility: in case of a breach, the organization is still accountable to its users and to regulatory authorities.

Vendor assessment and due diligence
Before entrusting data or workloads to a provider, the organization must assess their security posture through standardized questionnaires (e.g., CSA's CAIQ), review of certifications (ISO 27001, SOC 2, FedRAMP), and, where contractually possible, a direct audit. SOC 2 Type II certification is particularly significant because it attests that the declared controls have been operational for a period of at least six months.
Contracts and SLA
The contract with the provider must specify service levels (SLA), incident notification procedures, audit rights, geographic data residency, and responsibilities in case of a breach. An SLA that guarantees 99.9% uptime means approximately 8.7 hours of tolerated downtime per year; 99.99% falls to 52 minutes annually.
CASB — Cloud Access Security Broker
A CASB is an intermediary — software or service — positioned between corporate users and the cloud services they use. It can operate in inline mode (in the traffic path, capable of blocking requests in real time) or in API-based mode (connected to the provider via API, oriented toward monitoring, auditing, and after-the-fact remediation). A CASB allows applying corporate policies — DLP, encryption, access control — to cloud services that would otherwise be outside the corporate control perimeter.
MSP and MSSP
A Managed Service Provider (MSP) manages technology services on behalf of the customer (servers, networks, backup). A Managed Security Service Provider (MSSP) specifically manages security services: SIEM, SOC, firewall management, incident response. MSPs and MSSPs are also third-party vendors and must be assessed with the same due diligence.
PITFALL — CASB inline vs API-based
An inline CASB can block requests before they reach the cloud service, but introduces latency and depends on the CASB's own availability. An API-based CASB cannot block in real time, but detects and corrects issues after the fact. The exam may ask you which to choose based on the requirement: "real-time blocking" → inline; "monitoring and auditing" → API-based.

Infrastructure as Code (IaC)

Infrastructure as Code is the approach of defining and managing IT infrastructure — servers, networks, databases, configurations — through declarative or imperative code files instead of manual processes. Tools such as Terraform, AWS CloudFormation, Azure ARM Templates, or Ansible allow creating, modifying, and destroying infrastructure resources in an automated and repeatable manner.

From a security perspective, IaC brings new advantages and risks. The main advantage is consistency: the same template applied a hundred times produces a hundred identical environments, eliminating the configuration drift that accumulates in manually managed environments. The main risk is that a security error in the template automatically propagates to all environments where it is deployed.

Security advantages of IaC
  • Code review: IaC templates can be subjected to code review and static analysis (e.g., tfsec, Checkov) before deployment, detecting misconfigurations before they reach production.
  • Versioning: infrastructure changes are tracked in a version control system (Git), enabling complete audits of who changed what and when.
  • Idempotency: applying the same template multiple times always produces the same result, making the infrastructure state predictable.
  • Fast rollback: in case of a problem, it is possible to revert to a previous version of the template and apply it immediately.
Security risks of IaC
  • Secrets in code: the most common risk is embedding credentials, API keys, or certificates directly in IaC templates, making them accessible to anyone with access to the repository.
  • Error propagation: a misconfiguration in the template replicates across all environments.
  • Excessive privilege: IAM roles used by IaC often have very broad permissions to be able to create any resource.
Practical example — Secrets in a Terraform template
A developer writes a Terraform template that includes the database password in plaintext as a variable. When the template is committed to GitHub, the password becomes accessible to all collaborators and potentially the public if the repository is public. The correct solution is to use a secret manager (e.g., AWS Secrets Manager, HashiCorp Vault) and have IaC retrieve the secret at runtime rather than embedding it in the code.
Exam Tip — IaC and security
The exam associates IaC with: (1) risk of secrets hardcoded in the code, (2) advantage of consistency and reproducibility, (3) need for static scanning of templates before deployment. If the question asks "how to ensure that cloud configuration is always compliant with policies?" → IaC + CI/CD pipeline with policy checks is the correct answer.

Serverless Computing

The term serverless does not mean that servers do not exist: it means that the customer does not manage the servers. The provider automatically allocates compute resources to run the customer's code and deallocates them when execution ends. The customer pays only for the actual execution time of functions, not for constantly active resources.

The fundamental execution unit in the serverless model is the function (FaaS — Function as a Service): a piece of stateless code that is executed in response to an event (an HTTP request, a queued message, a file change). Each execution is isolated from the others and typically lasts milliseconds or a few seconds.

Security implications of serverless
  • Reduced attack surface: no servers to patch, no permanently open ports, no OS to manage. The provider keeps the runtime updated.
  • Granular IAM permissions: each function should have an IAM role with minimum necessary permissions. A function that reads from S3 should not be able to write to DynamoDB.
  • Injection: serverless functions that process user input are vulnerable to the same injection attacks as traditional applications (SQL injection, command injection).
  • Dependencies: functions often import third-party libraries. If these libraries contain vulnerabilities, the function is vulnerable.
  • Inability to patch manually: the customer cannot choose when to apply patches to the runtime; the provider applies them automatically, which is generally positive but removes control.
PITFALL — Serverless and "inability to patch"
In a serverless model the customer cannot choose if or when to patch the language runtime (e.g., Python 3.11, Node.js 20). The provider applies patches according to its own schedule. This is considered an advantage from a security perspective (the runtime is always updated), but is also a loss of control that can create compatibility problems. The exam may present it as either a pro or a con depending on the context.

Microservices

The microservices architecture is an approach to application design in which a single application is broken down into many small, independent, and separately deployable services, each responsible for a single business function. Services communicate with each other via APIs (typically REST or gRPC) or via message queues.

In contrast, a monolithic architecture concentrates all functionality in a single process. Microservices offer independent scalability, resilience (a crashing service does not bring down the entire application), and ease of continuous deployment, but introduce a distributed attack surface and security complexity.

Aspect Monolith Microservices
Attack surface Concentrated Distributed across many APIs
Impact of a vulnerability Potentially the entire app Limited to the individual service
Inter-service authentication Not required (same process) Required (mutual TLS, JWT tokens)
Logging and monitoring Centralized Distributed, requires correlation
Patches and updates Requires redeployment of entire app Individual service can be updated
Security in APIs between microservices
Every call between services is a potential attack surface. Best practices include: mutual authentication via certificates (mTLS), use of a service mesh (e.g., Istio) to apply network policies and encryption automatically, and short-lived authorization tokens (short-lived JWT) for each call. A service mesh centralizes security management without requiring each individual service to implement its own logic.
API Gateway
An API Gateway is a component that acts as a single entry point for all external requests to microservices. It centralizes authentication, authorization, rate limiting, logging, and message transformation at a single control point, simplifying security management compared to implementing these controls in every service.
Exam Tip — Microservices and security
The exam associates microservices with the need to protect internal communication APIs and manage authentication between services. If the question describes an application where "each component communicates via API" and asks how to protect internal traffic, the correct answer includes mTLS or a service mesh. Do not confuse with security of APIs exposed to end users, which is the domain of WAF (Web Application Firewall).

Containerization

Containerization is a form of operating system-level virtualization that packages an application together with all its dependencies (libraries, runtime, configuration files) into a portable and isolated unit called a container. Unlike virtual machines, containers do not include an entire operating system: they share the host operating system's kernel, making startup much faster and the footprint much lighter.

Feature Virtual Machine (VM) Container
Operating system Full guest OS per VM Shares the host kernel
Size Gigabytes Megabytes
Startup time Minutes Seconds / milliseconds
Isolation Strong (hypervisor) Weaker (kernel namespaces)
Kernel attack surface Reduced (separate guest OS) Larger (shared kernel)
Container security
The main security risks of containers include: images with known vulnerabilities (a Docker image downloaded from a public registry may contain libraries with known CVEs), containers running as root (a process running as root in the container has elevated privileges if it manages to break out of isolation), and kernel sharing (a vulnerability in the host kernel can potentially be exploited by any container).
Container Escape
Analogous to VM escape, container escape is an attack in which a process inside a container exploits a vulnerability to break out of the container's namespace and access the host operating system or other containers. It is less common than VM escape because it requires kernel vulnerabilities, but is considered the most severe security risk in containerized deployments.
Orchestration — Kubernetes
Kubernetes is the most widely used container orchestration platform. It automates the deployment, scaling, and management of containers in clusters. It introduces security components such as RBAC (Role-Based Access Control) to control who can do what in the cluster, Network Policies to segment traffic between pods, and Secrets to manage credentials securely.
Practical example — Vulnerable Docker image
A development team uses the node:14-alpine image as the base for their application. This version of Node.js has reached end-of-life and no longer receives security patches. An image scanner (e.g., Trivy, Snyk) detects 47 CVEs in the image, of which 3 are critical. The solution is to upgrade to the node:20-alpine image, re-test the application, and integrate automatic scanning into the CI/CD pipeline so it runs on every build.
PITFALL — VM vs Container: isolation
A VM offers stronger isolation because each VM has its own complete kernel, and a vulnerability in the guest OS does not directly compromise the host system or other guests. A container shares the kernel with the host: if the kernel has a vulnerability, all containers running on that host could be at risk. The exam may ask you which technology to choose to maximize isolation → answer: VM (not containers).

Virtualization and Hypervisors

Virtualization is the technology that enables multiple operating systems (guests) to run on a single physical hardware by abstracting and sharing hardware resources. The key component that makes virtualization possible is the hypervisor, also called Virtual Machine Monitor (VMM), which mediates access to hardware resources between the different virtual machines and guarantees their mutual isolation.

Type I Hypervisor (Bare-Metal)
Installed directly on physical hardware, without an intermediate host operating system. It has direct access to hardware resources, making it more performant and with a smaller attack surface compared to Type II. It is the standard choice in enterprise environments and cloud providers. Examples: VMware ESXi, Microsoft Hyper-V (standalone installation), Citrix Hypervisor, KVM (in the Linux kernel).
Type II Hypervisor (Hosted)
Installed on top of an existing host operating system, like a regular application. It is easier to install and use, but has lower performance (passes through two layers before reaching hardware) and a larger attack surface (vulnerabilities in the host OS can compromise the hypervisor). It is used primarily for development, test, and lab environments. Examples: VMware Workstation, Oracle VirtualBox, Parallels Desktop.
VM Sprawl
The uncontrolled proliferation of virtual machines that have been created but are no longer needed, continuing to consume resources and, most importantly, expanding the attack surface. A forgotten VM often does not receive security patches, may have permissive configurations, and becomes easy prey for attackers. Managing the VM lifecycle (creation, use, decommissioning) is an important security control.
Feature Hypervisor Type I Hypervisor Type II
Installed on Bare-metal hardware Host operating system
Performance Higher Lower (host OS overhead)
Attack surface Reduced Larger (includes host OS)
Typical use Datacenter, cloud, production Development, test, personal lab
Examples ESXi, Hyper-V, KVM VirtualBox, VMware Workstation
Exam Tip — Type I vs Type II
The exam often asks to distinguish the two hypervisor types. The main discriminator is the presence or absence of a host operating system: Type I runs directly on hardware (more secure, more performant), Type II runs on top of a host OS (simpler, less secure). If the question mentions "datacenter" or "production" → Type I. If it mentions "a developer's laptop" or "test environment" → Type II.
PITFALL — VM Sprawl vs VM Escape
These two terms are often confused. VM Sprawl is a management problem: too many active and unmonitored VMs. VM Escape is a critical security problem: an attacker escaping VM isolation to access the hypervisor or other VMs. VM Sprawl increases the attack surface; VM Escape is the actual exploit.

Software-Defined Networking (SDN) and Cloud Networking

Software-Defined Networking (SDN) is an architecture that separates the control plane (decisions about where to route traffic) from the data plane (the actual forwarding of packets). In a traditional network, these two functions are integrated in the individual network device (switch, router). With SDN, the control plane is centralized in a software component (SDN controller) that manages the entire network via APIs, while physical devices become simple elements that follow instructions.

Virtual Private Cloud (VPC)
A VPC is a logically isolated virtual network within a public cloud, dedicated to a single customer. It allows defining subnets, routing tables, internet gateways, and VPN connections just like in a private physical network. Segmentation via VPC is the fundamental network security control in the cloud: workloads with different trust levels must reside in different VPCs or in different subnets with controlled routing.
Security Group
A security group is a stateful virtual firewall that controls inbound and outbound traffic to a cloud resource (e.g., an EC2 instance). Security group rules specify protocol, port, and source/destination. Unlike physical firewalls, security groups are applied directly to the individual resource, not to the network perimeter. A misconfiguration (e.g., allowing SSH from 0.0.0.0/0) is one of the most common vulnerabilities in cloud deployments.
Software-Defined Wide Area Network (SD-WAN)
SD-WAN applies SDN principles to wide area networks (WAN), allowing centralized management of connectivity between distributed sites, optimizing traffic routing and applying security policies programmatically. SD-WAN can route critical traffic over MPLS and less sensitive traffic over the internet, with transparent encryption.
Secure Access Service Edge (SASE)
SASE is a framework that combines network capabilities (SD-WAN) with security capabilities (CASB, ZTNA, SWG, Firewall-as-a-Service) in a single cloud service. The idea is that security follows the user wherever they are, instead of being concentrated in a fixed network perimeter. SASE is particularly relevant for organizations with a distributed or fully remote workforce.
Practical example — Segmentation with VPC and Security Group
A three-tier web application (frontend, backend API, database) is deployed on AWS. A single VPC is created with three subnets: public (frontend), application private (backend), data private (database). Security groups are configured as follows: the frontend accepts traffic on port 443 from the internet; the backend accepts traffic only from the frontend's security group on application ports; the database accepts connections only from the backend's security group on the database port. The internet can never directly reach the database.
Exam Tip — SDN and security
The exam associates SDN with the following security advantages: centralized management of network policies, automation of incident response (e.g., automatic blocking of a compromised host via API), microsegmentation (each workload has its own virtual network perimeter). The main risk of SDN is that the centralized controller becomes a single point of failure and a critical target for attackers.

High Availability in the Cloud

High availability (HA) is the property of a system to remain operational and accessible for most of the time, even in the presence of hardware, software, or network failures. In the cloud, high availability is not automatically guaranteed just because the cloud is used: it must be explicitly designed through specific architectural choices.

Availability Zone (AZ)
An Availability Zone is a physically separate datacenter within a cloud region. The AZs of the same region are connected by low-latency networks but are isolated from each other's physical failures (power, networking, local natural events). Distributing resources across multiple AZs (multi-AZ deployment) is the primary way to achieve high availability in the cloud.
Region
A region is a geographic area that contains multiple Availability Zones. Regions are completely independent from one another. Distributing resources across multiple regions (multi-region deployment) offers protection from catastrophic events affecting an entire geographic area (natural disasters, regulatory issues), but introduces latency and data synchronization complexity.
Load Balancer
A load balancer distributes incoming traffic among multiple instances of a service. In the context of high availability, it also serves to detect non-functioning instances (health check) and automatically redirect traffic to healthy instances. It is a fundamental component in any high-availability cloud architecture.
Auto Scaling
Auto scaling is the cloud's ability to automatically add or remove instances of a service based on demand (horizontal scalability) or to increase/decrease the resources of a single instance (vertical scalability). It contributes to high availability by ensuring there are always enough resources to handle the load.
Exam Tip — HA and Scalability
The exam distinguishes between availability (the system is up), scalability (the system handles more load), and elasticity (the system adapts automatically to load). Availability concerns resistance to failures. Scalability is the capacity to grow. Elasticity is the ability to grow and shrink automatically. A system can be highly available without being elastic (e.g., two physical servers in failover).

Considerations — Evaluation factors for architectures

Objective 3.1 requires evaluating the security implications of different cloud architectural models not only from a technical perspective, but also considering a number of operational and strategic factors. These factors influence design choices and must be balanced against each other.

Availability
Measures the time a system is operational and accessible. In the cloud, availability is typically expressed as a percentage (SLA). An SLA of 99.9% (three nines) tolerates approximately 8.7 hours of annual downtime; 99.99% (four nines) only 52 minutes.
Resilience
A system's ability to withstand failures and recover quickly. Resilience goes beyond simple high availability: a resilient system not only stays up during a failure, but adapts and continues to function in a degraded mode if necessary, instead of failing completely (concept of graceful degradation).
Cost
The cloud transforms costs from CapEx (Capital Expenditure, hardware purchases) to OpEx (Operational Expenditure, pay-per-use). This transformation has security implications: resource sprawl (forgotten VMs or containers) generates unexpected costs, and pressure to reduce costs can lead to reducing security controls.
Responsiveness
The speed with which a system responds to requests. In the cloud context, responsiveness influences the choice between serverless architectures (fast response for variable loads), containers (startup in milliseconds), and VMs (startup in minutes). Geographically distributed architectures (CDN, multi-region) improve responsiveness by reducing latency.
Scalability
The ability to increase resources to sustain higher loads. Vertical scalability adds resources to a single instance (more CPU, more RAM). Horizontal scalability adds instances. Horizontal scalability is preferable in the cloud because it has no physical limits and integrates better with high availability.
Ease of deployment
The ease with which new versions of an application or infrastructure can be deployed. IaC, containers, and serverless increase deployment ease but require secure CI/CD pipelines and approval processes to prevent unauthorized deployments.
Risk transference
The cloud shifts some operational risks to the provider (hardware failures, OS patching in PaaS and SaaS, physical security of datacenters), but does not transfer legal and reputational responsibility toward end customers in case of a breach. The organization can purchase cyber insurance as an additional form of risk transference.
Ease of recovery
The ease of restoring after a failure or incident. The cloud simplifies recovery through snapshots, automatic backups, failover to a secondary AZ, and the ability to recreate the entire infrastructure via IaC in a short time. However, the recovery plan must be tested regularly: having a backup is not sufficient, one must verify that the restore works.
Patch availability vs Inability to patch
In IaaS models the customer controls operating system patches. In PaaS and SaaS models the provider applies patches, which guarantees regular updates but may cause unexpected incompatibilities. In some embedded or legacy systems (IoT, ICS/SCADA) it may be impossible to apply patches without prolonged operational interruptions: this "inability to patch" is a significant risk that requires compensating controls (network segmentation, anomaly monitoring, application whitelisting).
Power and Compute
In the cloud, the organization does not manage the physical infrastructure. Energy consumption and compute power are variables abstracted by the provider. However, in on-premises or hybrid environments the responsibility remains to ensure redundant power (UPS, generators) and sufficient compute capacity for critical workloads.
Practical example — Risk Transference in the cloud
A healthcare company migrates its electronic health records system to a HIPAA-certified cloud provider. The provider assumes responsibility for physical security of datacenters, infrastructure availability, and OS patching (PaaS). However, if a customer employee misconfigures access permissions and a third party accesses patient data, the HIPAA violation falls on the healthcare company, not the provider. The Business Associate Agreement (BAA) signed with the provider defines exactly this boundary.

Cloud and Virtualization Vulnerabilities (2.3)

Objective 2.3 introduces a subset of vulnerabilities specific to virtualized and cloud environments, which differ in mechanism and impact from those in traditional environments.

VM Escape
VM escape is a critical vulnerability in which an attacker who controls a process inside a virtual machine manages to cross the boundaries imposed by the hypervisor and access the hypervisor itself, the host operating system, or other virtual machines on the same physical hardware. In a multi-tenant cloud environment, a VM escape would allow an attacker to move from their own tenant to other tenants. It is considered one of the most severe risks of virtualization, even if relatively rare thanks to the robustness of mature hypervisors. The main mitigation is to keep the hypervisor updated with all security patches.
Resource Reuse
When a customer releases a VM or storage in the cloud, the provider typically reallocates those physical resources (RAM, disk, CPU) to the next customer. If the provider does not correctly "clean" the memory or storage before reallocation, the new customer could potentially access residual data from the previous one. Mature cloud providers implement memory and storage "zeroing" procedures, but this remains a theoretically relevant vulnerability, especially for highly sensitive data.
Cloud-Specific Vulnerabilities
Cloud environments introduce categories of vulnerabilities absent on-premises:
  • Misconfiguration: errors in configuring security groups, storage buckets, IAM permissions — the most frequent cause of cloud breaches.
  • Insecure API: cloud management APIs (management plane) are a critical target; compromised API credentials give complete access to the infrastructure.
  • Data sovereignty violations: data automatically replicated to geographic regions not authorized by regulation.
  • Insufficient logging: cloud environments with disabled or non-centralized logging make incident detection and analysis difficult.
Data Sovereignty
The data sovereignty principle establishes that data is subject to the laws and regulations of the country or jurisdiction in which it is stored or processed. If a cloud provider automatically replicates data to a different geographic region from the one contractually agreed upon, the organization may violate regulations such as GDPR (which requires that EU citizens' data remain within the EU or in countries with adequate protection) or specific sector regulations.
Practical example — VM Escape in a multi-tenant environment
In 2015 the VENOM vulnerability (CVE-2015-3456) was discovered in the virtual floppy disk driver of QEMU/KVM. An attacker with access to a guest VM could exploit this vulnerability to execute arbitrary code on the hypervisor host, potentially gaining access to all VMs running on the same physical hardware. The patch was released within a few days by major cloud providers, but the incident concretely demonstrated the theoretical risk of VM escape.
Exam Tip — Cloud vulnerabilities (2.3)
For the exam, remember the distinction: VM Escape = breaking out of VM isolation (attack); VM Sprawl = too many unmanaged VMs (operational/governance problem); Resource Reuse = residual data on reallocated resources. Misconfiguration is the most common cloud vulnerability on the exam: if a question describes a storage bucket with exposed data or an instance reachable from the internet, the cause is almost always a misconfiguration, not a technical exploit.
PITFALL — VM Escape vs Container Escape
VM Escape exploits vulnerabilities in the hypervisor to escape the VM. Container Escape exploits vulnerabilities in the host OS kernel to escape from the container's namespace. The mechanisms are different, but the effect is analogous: an attacker crosses the isolation boundaries. VM Escape is generally considered harder to accomplish because mature hypervisors have few attack vectors; Container Escape is more common because the kernel is a larger attack surface.

Key terms

Cloud Computing
Model for delivering IT resources over the internet, on-demand, with automatic elasticity and pay-per-use billing.
IaaS / PaaS / SaaS
The three main cloud service models, distinguished by how much infrastructure the provider manages. In IaaS the customer manages the OS; in PaaS they manage only the code; in SaaS they manage only the data.
FaaS (Function as a Service)
Execution of individual code functions in response to events, without server management. Practical synonym for serverless from a computational standpoint.
Shared Responsibility Model
Model that defines the division of security responsibilities between provider and customer. The provider is responsible for security of the cloud; the customer for security in the cloud.
Cloud Controls Matrix (CCM)
Framework from the Cloud Security Alliance (CSA) that maps security controls specific to cloud environments, organized by domain and aligned to major standards.
Public / Private / Community / Hybrid Cloud
The four cloud distribution models. They differ by who has access to the infrastructure and the level of resource sharing.
CASB (Cloud Access Security Broker)
Intermediary between users and cloud services that applies corporate security policies. Can operate inline (real-time blocking) or via API (monitoring and remediation).
Hypervisor Type I / Type II
Type I (bare-metal) runs directly on hardware, more secure and performant. Type II runs on top of a host OS, simpler but with a larger attack surface.
VM Escape
Attack in which a process in a VM crosses the hypervisor's boundaries to access the host system or other VMs. One of the most serious attacks in virtualized environments.
VM Sprawl
Uncontrolled proliferation of VMs that are no longer managed, increasing attack surface and costs without operational value.
Resource Reuse
Risk that cloud resources reallocated to a new customer contain residual data from the previous customer if not correctly zeroed.
Containerization
OS-level virtualization that packages applications with their own dependencies. Lighter than VMs but with weaker isolation (shared kernel).
Infrastructure as Code (IaC)
Infrastructure management through versioned and automated code. Ensures consistency but requires secrets protection and template scanning.
Serverless
Model in which the provider fully manages the servers; the customer deploys only code functions. Reduces attack surface but removes control over runtime patching.
Microservices
Architecture in which an application is decomposed into small, independent services that communicate via API. Reduces the impact of individual vulnerabilities but requires inter-service API security.
VPC (Virtual Private Cloud)
Isolated virtual network within a public cloud, with subnets, routing, and security controls configurable by the customer.
Security Group
Stateful virtual firewall applied to individual cloud resources, controlling inbound and outbound traffic via rules on protocol, port, and source/destination IP.
SDN (Software-Defined Networking)
Architecture that separates the control plane from the data plane, centralizing network management via software and APIs.
SASE (Secure Access Service Edge)
Framework that combines network capabilities (SD-WAN) and security (CASB, ZTNA, SWG, FWaaS) into a unified cloud service, following the user wherever they are.
Data Sovereignty
Principle by which data is subject to the laws of the country where it resides or is processed. Relevant for GDPR and sector regulations.
HSM (Hardware Security Module)
High-security device or service for generating, storing, and using cryptographic keys, keeping them inaccessible in plaintext even to administrators.

Summary flashcards

  • Who manages the OS in IaaS? — The customer; in PaaS and SaaS the provider manages it.
  • Shared Responsibility Model — The provider is responsible for security OF the cloud; the customer for security IN the cloud.
  • Hypervisor Type I vs Type II — Type I runs bare-metal (more secure); Type II runs on top of a host OS (less secure, used in labs).
  • VM Escape — Attack in which a process in a VM escapes the hypervisor's isolation and accesses the host or other VMs.
  • VM Sprawl — Uncontrolled proliferation of decommissioned but still active VMs; increases attack surface and costs.
  • Resource Reuse — Risk of residual data on cloud resources reallocated to the next customer without proper zeroing.
  • Container vs VM — isolation — Containers share the host kernel (weaker isolation); VMs have a separate kernel (stronger isolation).
  • IaC — main risk — Secrets hardcoded in the code (credentials, API keys) committed to accessible repositories.
  • Serverless — inability to patch — The runtime is patched by the provider; the customer does not control the timing and manner of updates.
  • CASB inline vs API-based — Inline: real-time blocking in the traffic path. API-based: monitoring and after-the-fact remediation via provider API.
  • VPC — Isolated virtual network within a public cloud; provides segmentation and control similar to a physical private network.
  • Security Group — Stateful virtual firewall applied to individual cloud resources; misconfiguration (e.g., SSH 0.0.0.0/0) is a common cause of breaches.
  • Data Sovereignty — Data is subject to the laws of the country where it resides; relevant for GDPR and sector regulations.
  • SASE — Framework that combines SD-WAN + cloud security capabilities (CASB, ZTNA, SWG) in a unified service that follows the user everywhere.
  • Risk transference in the cloud — The cloud shifts operational risks to the provider, but not legal responsibility toward customers and regulators.
  • Most common cloud-specific vulnerability — Misconfiguration (security groups, public S3 buckets, excessive IAM permissions); not advanced technical exploits.
