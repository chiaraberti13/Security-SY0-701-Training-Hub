Objective 4.1 — Security techniques on computing resources

Protecting computing resources means addressing the entire attack surface: installing an antivirus on a workstation is not enough. Objective 4.1 covers the systematic approach to system hardening, starting from the definition of secure reference configurations through to specific hardening for each device category — workstations, servers, network devices, cloud, embedded systems, ICS/SCADA, IoT, and mobile devices — concluding with application security and continuous monitoring.

The guiding principle is attack surface reduction: every enabled service, open port, active account, or installed software that is not strictly necessary represents an opportunity for an attacker. Hardening means systematically removing everything unnecessary, strengthening what remains, and verifying over time that the configuration does not drift from its target.

Secure Baselines — Establish, Deploy, Maintain

A secure baseline is a documented set of minimum settings that a system must comply with to be considered acceptably secure in the organizational environment. The baseline is not an unattainable ideal state, but a realistic operational standard, derived from frameworks such as CIS Benchmarks, DISA STIGs, or NIST SP 800-70, and adapted to the company's specific context.

Defining a baseline without deploying or maintaining it is useless: the complete cycle consists of three mandatory phases that the exam specifically tests.

Establish
The establishment phase produces the reference configuration. Security settings appropriate to the system type (Windows workstation, Linux server, network switch…) are selected, required parameters are documented (enabled accounts, active services, password policies, encryption, logging), and the baseline is approved through the change management process. The critical point is that the baseline must be realistic: settings that are too restrictive and are systematically bypassed by users protect no one.
Deploy
The deployment phase applies the baseline to systems in production. In enterprise environments this is done through automated configuration management tools: Group Policy Objects (GPO) for Windows systems, Ansible/Puppet/Chef for mixed environments, gold-certified images for provisioning new systems, or MDM for mobile devices. The deployer must verify that each system correctly accepts the configuration and there are no conflicts with existing applications.
Maintain
The maintenance phase is the most operational and continuous. It includes monitoring configuration drift — the gradual deviation of systems from the baseline due to patches, application updates, support interventions, or unauthorized changes — and automated or manual remediation when a deviation is detected. It also includes periodic review of the baseline itself: a configuration defined two years ago may no longer be adequate against current threats.
Practical example — Baseline for corporate Windows workstations: the security team defines that all workstations must have: SMBv1 disabled, PowerShell in ConstrainedLanguageMode, AutoRun disabled, BitLocker active, Windows Defender with cloud protection, Guest account disabled, minimum password length of 12 characters, screensaver with lock after 5 minutes. The baseline is deployed via GPO and verified weekly with Microsoft Defender for Endpoint. When a system shows a deviation (e.g., AutoRun enabled after a driver update), the alert is generated automatically and the policy is reapplied.
EXAM TIP 4.1-A — In scenario questions, the exam often asks which phase of the baseline cycle was omitted. If a company has a well-written policy but systems don't comply → Deploy is missing. If systems initially complied with the policy but drifted over time → Maintain is missing. If no reference document exists → Establish is missing.
PITFALL — Baseline vs Benchmark: a benchmark (e.g., CIS Benchmark) is a document of general recommendations produced by a third-party organization. A baseline is the specific configuration the organization has chosen to adopt, derived from the benchmark but adapted to its own context. The exam uses both terms: the benchmark is the source, the baseline is the internal result.

Hardening — Workstations and Servers

Workstations and servers are the most classic hardening targets because they run general-purpose operating systems with a broad attack surface. The difference between the two is not just processing power: servers host critical services accessible over the network, while workstations are managed by end users and are therefore exposed to attack vectors related to human behavior (phishing, downloads, USB devices). The hardening process must take these differences into account.

Workstation hardening

Removal of unnecessary software
Every installed application is a potential vulnerability. The principle is simple: if it is not necessary for the user's work role, it should be removed. This includes pre-installed games, unauthorized third-party applications, development tools on non-technical users' machines, browser add-ons. Less software means fewer patches to manage, fewer attack vectors, and less likelihood of misconfigurations.
Disabling unnecessary services
Modern operating systems start numerous background services for features rarely used in business environments (e.g., Routing and Remote Access, Telnet Client, SNMP Service, Remote Registry on Windows). Every active service is a potential gateway into the system. Hardening requires a systematic review of running services and disabling those not justified by documented operational needs.
Local account management
The local Administrator account on Windows and the root account on Linux are targets of brute force and pass-the-hash attacks. Best practices include: disabling or renaming the built-in Administrator account, using tools like Microsoft LAPS (Local Administrator Password Solution) to randomize local passwords, and applying the principle of least privilege to all user accounts.
Group Policy and SELinux
In Windows environments, Group Policy Objects (GPO) allow centralized distribution of security settings: password policies, Registry restrictions, software restrictions, host-based firewall configuration, and policies for removable devices. On Linux systems, SELinux (Security-Enhanced Linux) implements MAC (Mandatory Access Control): even if a process is compromised, it cannot access resources outside its SELinux domain, drastically limiting lateral damage.
Disk encryption
Full-Disk Encryption (FDE) protects data in the event of physical theft or loss of the device. On Windows, BitLocker is used, integrated with TPM for transparent protection to the user. Without FDE, an attacker with physical access can simply boot the system from an external medium and read the entire disk. With FDE active, the disk is unreadable without the cryptographic key.

Server hardening

Servers require an even more rigorous approach because they host network-accessible services and high-value data. The operational principle is that a server should do one thing and do it securely.

Role minimization
A web server should not also run a database, an email server, and an FTP server. Separation of roles limits the blast radius of a compromise: if a service is breached, the attacker does not automatically find other critical services on the same host.
Patch management
Servers are high-value targets and their OS is updated less frequently than workstations due to fear of service interruptions. This delay creates known vulnerability windows. The correct process includes: testing in a staging environment, scheduled maintenance windows, rollback plans, and prioritization based on CVSS scores of critical vulnerabilities.
Secure remote access
Remote access to servers should occur exclusively through secure protocols such as SSH (with public key authentication, not passwords), and never through Telnet, RDP exposed directly to the Internet, or direct root access. The ideal is access via a jump server (bastion host), so that the production server is never directly reachable from the outside.
Practical example — Hardening a Linux server: after installing Ubuntu Server 22.04, the hardening process includes: immediate package update, removal of openssh-server and reinstallation with custom configuration (PermitRootLogin no, PasswordAuthentication no, AllowUsers only the technical accounts), enabling UFW (host firewall) with allow rules only on necessary ports, installing fail2ban to block SSH brute force, enabling auditd for logging privileged actions, installing aide for file integrity monitoring, and enabling SELinux in enforcing mode.
EXAM TIP 4.1-B — In workstation hardening questions, the exam often associates SELinux with Linux systems and Group Policy with Windows systems. If the question mentions "Mandatory Access Control on Linux endpoints", the correct answer is SELinux. If it mentions "centralized distribution of Windows security policies", the answer is GPO/Group Policy.

Hardening — Switches and Routers

Network devices — switches and routers — are often overlooked in the hardening process because they "just route traffic." In reality they are high-value targets: whoever controls the network infrastructure controls all the traffic passing through it, with possibilities for sniffing, man-in-the-middle, and routing manipulation. Their hardening follows principles similar to those for servers, adapted to the firmware/CLI context.

Changing default credentials
The first action on any network device is changing the default credentials. Factory passwords (often "admin/admin" or variants) are publicly known and indexed in online databases. This applies to all devices: managed switches, routers, access points, firewalls, network printers. The management account password must follow the same policies as system passwords: adequate length, complexity, no reuse.
Disabling insecure protocols
Many network devices still support legacy protocols such as Telnet, HTTP (for web management), SNMPv1/v2c, or CDP/LLDP indiscriminately. Telnet transmits credentials in cleartext; SNMP v1/v2 uses unencrypted community strings; HTTP exposes the management session to interception. Hardening includes: exclusive enablement of SSH v2 for remote CLI access, HTTPS for web management, SNMPv3 with authentication and encryption if SNMP is necessary, and disabling CDP/LLDP on ports facing untrusted networks.
Port Security (switches)
Port security on managed switches limits the number of MAC addresses that can be learned on a single port and defines the behavior in case of violation (shutdown, restrict, protect). It prevents attacks like MAC flooding (which saturates the switch's MAC table, degrading it to a hub) and limits the ability to connect unauthorized devices to the physical network.
VLAN and segmentation
Segmentation via VLAN separates traffic from different groups of users or systems at Layer 2, even on shared physical infrastructure. This limits the broadcast domain, reduces lateral propagation of malicious traffic, and allows different security policies to be applied to each segment. A separate IoT VLAN from the workstation network, for example, prevents a compromised IoT device from directly accessing file servers.
Access Control Lists (ACL) on routers
Routers filter traffic between network segments via ACL. ACLs define which traffic flows (based on source/destination IP, port, protocol) are permitted or denied. A border router with correctly configured ACLs prevents unauthorized traffic from entering or leaving the corporate network, acting as the first perimeter filter before the dedicated firewall.
Disabling unnecessary services on the router
Cisco routers and similar devices run numerous services by default that are rarely needed in a modern enterprise environment: Finger service, IP small servers (echo, chargen, discard), HTTP server, CDP on external interfaces. Every disabled service reduces the device's own attack surface.
Practical example — Hardening a Cisco switch: after initial configuration, apply: no cdp run on ports facing external networks, service password-encryption, no service telnet, transport input ssh on VTY lines, setting a legal banner, ip dhcp snooping to prevent rogue DHCP servers, ip arp inspection to prevent ARP spoofing, and spanning-tree portfast bpduguard enable on access ports to protect against STP attacks.
EXAM TIP 4.1-C — The exam often associates MAC flooding with switch attacks. The countermeasure is port security. ARP spoofing is prevented by Dynamic ARP Inspection (DAI). DHCP spoofing is prevented by DHCP Snooping. Knowing the attack-countermeasure pair is the correct answer pattern.

Hardening — Cloud Infrastructure

Hardening in cloud environments cannot ignore the shared responsibility model: the cloud provider is responsible for security of the infrastructure (hypervisor, physical network, data center), while the customer is responsible for security in the cloud (instance configuration, access management, data encryption, application security). Neglecting one's own share of responsibility is the most frequent cause of data breaches in cloud environments.

Cloud Security Posture Management (CSPM)
CSPM tools continuously analyze the configuration of cloud resources (S3 buckets, security groups, IAM roles, databases) and flag deviations from the secure baseline. Without CSPM, a single misconfiguration — such as an S3 bucket with public access — can expose sensitive data for weeks before being discovered.
Cloud Identity and Access Management (IAM)
In cloud, identities are the perimeter. IAM policies define who can do what on which resources. The principle of least privilege is even more critical in cloud because a compromised account with excessive permissions can exfiltrate data, delete resources, or create backdoors in seconds via API. Best practices include: mandatory MFA for root/admin accounts, no hardcoded access keys in code, periodic credential rotation, and regular audit of permissions.
Security groups and network ACL
Security Groups (AWS) and Network Security Groups (Azure) are virtual firewalls at the instance or subnet level. The most common error is opening management ports (SSH, RDP) to "0.0.0.0/0" (all of Internet) instead of limiting them to specific IPs or a bastion host. Cloud hardening requires reviewing all security groups to eliminate overly permissive rules.
Encryption at rest and in transit
Every sensitive piece of data in the cloud must be encrypted both at rest (using the provider's KMS or own keys — BYOK) and in transit (TLS 1.2+ for all inter-service communications). Many cloud services offer automatic encryption, but it is the customer's responsibility to verify that it is actually enabled and configured with appropriate keys.
Secure Infrastructure as Code (IaC)
In modern cloud environments, infrastructure is defined via code (Terraform, CloudFormation, Bicep). This introduces the risk that insecure configurations are "written" into code and automatically deployed. IaC hardening includes: static scanning of IaC templates before deployment (tools like Checkov, tfsec), policy-as-code to block non-compliant resources, and version control with code review before merging.
EXAM TIP 4.1-D — In cloud security questions, if the scenario describes data exposed by misconfiguration (e.g., public bucket, port 22 open to the Internet), the problem is in the Deploy phase of the cloud baseline (security group rules not applied) or in the Maintain phase (no CSPM to detect the deviation). The answer rarely points to the cloud provider: the shared responsibility model assigns this responsibility to the customer.

Hardening — Embedded Systems and RTOS

Embedded systems are devices with hardware and software designed to perform specific, dedicated functions: routers, printers, IP cameras, medical devices, POS terminals, automation systems. Unlike workstations and servers, they are not designed to be easily upgradeable and often have severe hardware constraints that limit security options.

Embedded System
An embedded system combines dedicated hardware (processor, memory, storage) and specific software (firmware) in a device with a predefined function. The critical characteristic from a security perspective is that the software is often written with little emphasis on security, updates are rare or complex, and the support lifecycle is often shorter than the operational life of the device itself. An end-of-life embedded device will continue to function physically, but will no longer receive security patches.
Real-Time Operating System (RTOS)
An RTOS is an operating system designed to respond to events within precise and predictable time deadlines (temporal determinism). It is used in applications where latency is critical: industrial control systems, medical devices, avionics systems, automotive. Temporal predictability is the absolute priority; security is traditionally secondary. Many classic RTOSes (VxWorks, FreeRTOS, QNX) had critical vulnerabilities discovered only after years of use because they were not designed with an external threat model.
Typical security constraints of embedded systems
Embedded systems have structural limitations that make it difficult to apply traditional security controls: limited RAM (impossible to run complex security software), reduced storage (impossible to install monitoring agents), absence of user interface (difficult multi-factor authentication), difficulty with patching (often requires physical intervention or scheduled downtime), and in some cases prohibitive replacement costs that incentivize maintaining obsolete systems in production.
Practical example — Security assessment of an embedded system: a healthcare company needs to assess the security of a patient monitor with embedded software. The process includes: identifying the manufacturer and verifying active support, requesting the list of open network ports and active services, verifying the firmware update process, checking whether it uses standard or proprietary communication protocols, assessing the possibility of segmenting it in a dedicated VLAN, and requesting an SBoM (Software Bill of Materials) to identify components with known vulnerabilities.
EXAM TIP 4.1-E — The exam typically associates RTOS systems with scenarios involving medical devices, industrial systems, or automotive where availability and temporal determinism are priorities. When a question asks for the most appropriate security control for an embedded device that cannot be updated, the typical answer is network segmentation (isolating it) as a compensating control.

Hardening — ICS and SCADA

Industrial Control Systems (ICS) are computer systems that control physical processes in industrial, energy, hydraulic, or manufacturing environments. The term SCADA (Supervisory Control and Data Acquisition) specifically refers to the supervisory control and data acquisition architecture distributed over wide geographic areas. The security of these systems is critical because an attack can cause real physical damage: interruption of power supply, water contamination, blocking of production lines, or even damage to machinery and infrastructure.

Components of an ICS/SCADA environment
A typical ICS/SCADA environment includes: engineering workstations (HMI — Human Machine Interface) for supervision, historian servers for data collection, PLCs (Programmable Logic Controllers) that directly control actuators and motors, RTUs (Remote Telemetry Units) that collect data from remote sensors, and communication networks both standard (Ethernet, TCP/IP) and industrial (Modbus, DNP3, PROFINET). The IT/OT (Operational Technology) convergence has moved these systems from isolated networks to being increasingly connected, increasing the risk of cyber attacks.
Specific security challenges
ICS/SCADA systems present unique challenges: many use obsolete operating systems (Windows XP, Windows 2003) that no longer receive patches; update cycles are long because machine downtime for maintenance is costly; industrial protocols (Modbus, DNP3) were designed for reliability, not security; patching may require vendor validation to avoid invalidating certifications; and continuous availability is often more critical than security.
Air gap and segmentation
Historically, ICS security was based on the air gap: the industrial network was completely separate from the Internet and the corporate IT network. Today the air gap is often impractical due to remote monitoring needs, vendor maintenance, integration with ERP systems. The modern strategy is segmentation with an industrial DMZ: a separate demilitarized zone with bidirectional firewalls, which acts as a buffer between the IT and OT networks, using jump servers and unidirectional historians (data diodes) to minimize direct traffic.
Purdue Model and IEC 62443
The Purdue Reference Model (or ISA-95) is the reference architectural framework for ICS network segmentation, dividing systems into levels from 0 (physical field, sensors/actuators) to 5 (enterprise/Internet network). Separation between levels must be enforced with technical controls. The IEC 62443 standard is the international security framework for industrial environments, equivalent to what ISO 27001 is for general IT security.
Practical example — Stuxnet attack and lessons learned: Stuxnet (2010) is the most well-known example of a cyberattack on an ICS system. It targeted uranium enrichment centrifuges in Iran by subtly altering the rotation speed while displaying normal values to operators. The key lesson for Security+: (1) the air gap is not an absolute guarantee (it was introduced via USB); (2) PLCs without authentication can be reprogrammed by anyone who gains network access; (3) segmentation and process anomaly monitoring are essential even in industrial networks.
EXAM TIP 4.1-F — The exam often uses ICS/SCADA in scenarios asking for the most appropriate control for systems that cannot be updated or shut down. The answer almost always involves a combination of network segmentation (isolating the OT from the IT network) and compensating controls (anomaly monitoring, communication whitelisting). A total air gap is rarely the correct answer in modern contexts.

Hardening — IoT (Internet of Things)

IoT devices (sensors, IP cameras, smart thermostats, wearables, smart meters, connected locks, voice assistants) have grown enormously in corporate and home networks, introducing a distributed and difficult-to-manage attack surface. The fundamental problem is that many IoT devices are designed with priority given to cost and ease of use, relegating security to a secondary consideration — or ignoring it completely.

Typical structural vulnerabilities of IoT devices
Recurring problems in the IoT ecosystem include: default credentials identical for all devices of the same model (change not required or made difficult); firmware rarely updated with a short support cycle; web management interfaces with XSS, CSRF, injection vulnerabilities; unnecessary services active by default (Telnet, UPnP, FTP); absence of encryption in communications between device and vendor cloud; and collection of personal data with opaque privacy policies.
IoT network segmentation
The most effective and practical security measure for corporate IoT devices is their isolation in a dedicated VLAN, separate from the workstation, server, and guest network. This limits the blast radius in the event of a compromise: a breached IoT device can at most communicate with other IoT devices in the same VLAN, not with file servers or user workstations. Access from the IoT network to other networks must be explicitly permitted by a firewall, not allowed by default.
IoT communication protocols
IoT devices often use protocols different from traditional ones: MQTT (lightweight pub/sub for telemetry), CoAP (HTTP-like for low-power devices), Zigbee and Z-Wave (low-power wireless for home automation), LoRaWAN (long range, low power for smart cities). Each has its own security characteristics: some support authentication and encryption, others do not. The choice of protocol and its secure configuration fall within the responsibilities of the integration.
Privacy and collected data
IoT devices continuously collect data about the environment and users: cameras recording physical spaces, wearables tracking health, thermostats revealing lifestyle habits. This introduces privacy risks that go beyond traditional cybersecurity. Organizations must evaluate what data is collected by corporate IoT devices, where it is sent (vendor cloud, to which countries), how long it is retained, and whether vendor contracts offer adequate guarantees.
PITFALL — IoT vs ICS/SCADA: both use embedded systems and have similar patching constraints, but the context is different. IoT typically refers to consumer or building automation devices (cameras, sensors, smart meters); ICS/SCADA refers to industrial control systems critical for infrastructure (energy, water, manufacturing). IoT risks are primarily data theft and botnets; ICS/SCADA risks include physical damage and interruption of critical services. Security measures overlap (segmentation, strong credentials) but the criticality is very different.
EXAM TIP 4.1-G — In IoT questions, the typical pattern is: device with default credentials + connected to the main corporate network = high risk. The correct answer almost always includes changing default credentials + segmentation in a dedicated VLAN. If the device cannot be updated and has known vulnerabilities, the compensating control is complete isolation.

Hardening — Mobile Devices, MDM and Deployment Models

Mobile devices (smartphones, tablets) have become primary work tools in many organizations, but their personal nature, the variety of installable applications, constant connectivity, and ease of loss make them a complex security challenge. Control of mobile devices rests on three pillars: the management platform (MDM), the ownership model adopted by the organization (BYOD/COPE/CYOD), and the applied technical policies.

Mobile Device Management (MDM)
An MDM system is a centralized platform that allows the organization to manage, monitor, and protect corporate mobile devices or employee devices used for work. Typical features include: device enrollment, distribution of applications and certificates, policy enforcement (mandatory PIN/biometrics, encryption, blocking root/jailbreak access), separation between personal and work profiles (containerization), and complete or selective remote wipe in case of loss or theft. Examples of enterprise MDM: Microsoft Intune, VMware Workspace ONE, Jamf (for Apple).

The term EMM (Enterprise Mobility Management) is broader and includes MDM + application management (MAM) + content management (MCM). The term UEM (Unified Endpoint Management) unifies the management of PCs, Macs, mobile devices, and IoT in a single platform.
Bring Your Own Device (BYOD)
In the BYOD model the employee uses their own personal device to access corporate resources. The advantage for the organization is zero hardware cost; for the employee it is using a familiar device. The risks are significant: the organization has limited control over the device, unverified personal applications coexist with corporate data, incident management is complicated (remote wipe cannot delete personal data without consent), and the employee may resist installing the MDM agent for privacy reasons. BYOD requires strong containerization: corporate data must reside in an encrypted, separate container, not accessible to personal apps.
Corporate-Owned, Personally Enabled (COPE)
In the COPE model the organization purchases and provides the device, but allows personal use. The organization maintains full ownership and can apply complete security policies, including a full remote wipe. The employee can install personal apps in a separate space. The balance between control (full, on the IT side) and privacy (partial, on the employee side) makes COPE the most common model in large regulated organizations.
Choose Your Own Device (CYOD)
In the CYOD model the organization defines a catalog of approved devices and the employee chooses which they prefer. The device is corporate-owned (like COPE), but the personalized choice increases employee satisfaction. From a security perspective it is similar to COPE: full IT control, uniform MDM management, remote wipe without limitations.
Practical example — Remote wipe in a BYOD scenario: an employee reports the loss of their smartphone on which they had configured corporate email via Outlook and access to SharePoint. With a correctly configured MDM in BYOD mode, IT can perform a selective wipe: it deletes only the corporate profile (email data, certificates, VPN configuration, corporate apps) leaving personal photos, personal contacts, and private apps intact. Without MDM, the only option is to ask the employee to activate "Erase iPhone" from their own accounts, losing personal data too — and without any certainty that they will do so.

EXAM TIP 4.1-H — The exam distinguishes models primarily based on who owns the device and how much control IT has. BYOD = employee owns, IT controls little. COPE/CYOD = company owns, IT controls everything. If the question mentions "maximum control with personal use allowed" → COPE. If it mentions "minimum hardware cost" → BYOD. If it mentions "employee chooses the model from an approved catalog" → CYOD.
PITFALL — MDM vs MAM vs UEM: MDM manages the entire device; MAM (Mobile Application Management) manages only corporate apps without touching the rest of the device (useful for BYOD where installing an MDM agent is invasive); UEM manages all types of endpoints (PC, mobile, IoT) from a single console. The exam uses these terms specifically: they are not synonyms.

Application Security — Input Validation, Secure Cookies, Code Analysis, Code Signing

Application security addresses the risks introduced by the software itself: vulnerabilities in code, distribution of unverified software, insecure management of web sessions. Objective 4.1 specifically includes four fundamental techniques that the exam tests in secure development and application hardening scenarios.

Input Validation
Input validation is the fundamental defensive technique against a wide category of application attacks: SQL injection, XSS (Cross-Site Scripting), command injection, path traversal, buffer overflow. The principle is simple: any data coming from outside (users, APIs, uploaded files, URL parameters) must be validated, sanitized, and verified before being used by the application.

There are two main approaches: allowlist validation (permits only what is explicitly defined as valid — format, length, allowed characters) and denylist validation (blocks what is known to be dangerous). The allowlist is more secure because it blocks any unexpected input, including previously unknown input; the denylist is easier to implement but requires constant updates for new attack patterns.

Validation must occur server-side: client-side validation (JavaScript in the browser) is useful for the user experience but is trivially bypassable with tools like Burp Suite or simply by disabling JavaScript.
Secure Cookies
Session cookies in web applications are the primary mechanism for maintaining authentication between stateless HTTP requests. If a session cookie is stolen (via XSS, traffic interception, or physical browser access), the attacker can impersonate the authenticated user without knowing their password.

The three fundamental security properties of cookies are: HttpOnly (prevents JavaScript access to the cookie, mitigating XSS), Secure (the cookie is transmitted only over HTTPS connections, not HTTP), and SameSite (controls in which cross-site contexts the cookie is sent, mitigating CSRF). A session cookie that does not have all three of these properties set is vulnerable.
Static Code Analysis (SAST)
Static Application Security Testing (SAST) analyzes the application's source code or bytecode looking for vulnerabilities, without executing the program. It is run during development (integrated into the CI/CD pipeline) and can identify: use of deprecated or insecure functions, SQL injection or XSS patterns in code, insecure error or exception handling, hardcoded secrets (API keys, passwords in code), and dependencies with known vulnerabilities.

The advantages are that it can cover 100% of the code and identify problems before the software goes to production; the limitation is that it produces false positives and cannot detect logical or configuration vulnerabilities that only emerge at runtime.
Code Signing
Code signing is the process of cryptographically signing software with a digital certificate associated with the identity of the developer or organization. The operating system or distribution mechanism verifies the signature before executing the software: if the signature is valid, the code comes from the declared entity and has not been modified since signing; if the signature is invalid or missing, the system can block execution or warn the user.

Code signing protects against two distinct risks: software tampering (modification of the binary after signing, detected by the invalid signature) and malware distribution under a false name (a malware cannot sign itself with Microsoft's or Adobe's certificate without stealing those private keys). On modern operating systems (Windows with Secure Boot + driver signing, macOS with Gatekeeper and Notarization, iOS with App Store review), execution of unsigned code is blocked by default.
Practical example — SQL Injection and input validation: a web application has a search field that constructs the SQL query as: SELECT * FROM users WHERE name = '" + userInput + "'. An attacker enters: ' OR '1'='1. The query becomes: SELECT * FROM users WHERE name = '' OR '1'='1', which returns all users. With correct input validation (allowlist: only alphanumeric characters, maximum length 50 characters, no single quotes) or with parameterized queries, the input is rejected or treated as a literal string, making the injection impossible.
EXAM TIP 4.1-I — The exam distinguishes SAST (static analysis, source code, during development) from DAST (Dynamic Application Security Testing: dynamic analysis, running application, simulates attacks). SCA (Software Composition Analysis) is also relevant: it analyzes third-party dependencies for known vulnerabilities. Typical scenario: "before deploying the application, which technique identifies vulnerabilities in the source code without running the application?" → SAST.
PITFALL — Code signing vs Code review: code signing guarantees authenticity and integrity (who signed it and that the code has not been modified since signing), but does not guarantee that the code is secure or free of vulnerabilities. A developer can correctly sign software that contains serious bugs. Code review and SAST concern the quality and security of the code, not its authenticity. They are complementary, not alternatives.

Sandboxing and Monitoring

Sandboxing

A sandbox is an isolated execution environment, separated from the host operating system and other processes, in which untrusted or potentially malicious code can be executed without risk of real damage to the system. The principle is that even if the code in the sandbox is malicious, its actions are confined: it cannot modify the host filesystem, cannot access the corporate network, cannot communicate with other processes.

Sandboxing for malware analysis
Advanced antimalware systems and EDR solutions use the sandbox for behavioral analysis: a suspicious file (email attachment, download) is executed in an isolated virtual environment and monitored for 30-60 seconds. If it shows malicious behaviors (connection to C2, registry modification, file encryption, creation of suspicious child process), it is classified as malware even if its signature is unknown. This enables detection of zero-days and polymorphic variants.
Sandboxing in applications
Modern browsers (Chrome, Firefox, Edge) run each tab in a separate sandbox: if a web page exploits a browser vulnerability, the damage is limited to the sandboxed process and does not propagate to the operating system. The same principle applies to iOS and Android applications: each app runs in its own sandbox and cannot access another app's data without the user's explicit permission. Docker containers implement a similar concept at the system level.
Sandboxing in email systems
Advanced email gateways (Secure Email Gateway) use the sandbox to analyze attachments before delivery: a PDF or Office file is opened in a controlled environment and monitored for behaviors such as macro execution, network connections, or attempts to write to disk. This technique intercepts threats that would bypass analysis based only on static signatures.

Monitoring

Endpoint monitoring is the continuous process of collecting, analyzing, and correlating security events to detect anomalies, suspicious activity, and indicators of compromise. Monitoring is not a passive function: it requires a clear definition of what to monitor, how to correlate events, and how to respond to alerts.

Log collection and SIEM
Endpoints generate continuous logs: authentication events, file access, process execution, network activity, registry modifications. These logs must be centralized in a SIEM for correlation: a single failed authentication log is not significant; 500 attempts in 5 minutes on the same account from different IP addresses is a brute force pattern that the SIEM must detect and report.
Endpoint Detection and Response (EDR)
An EDR system is an agent installed on the endpoint that collects detailed telemetry (started processes, network connections, file system modifications, registry access) and sends it to a central platform for behavioral analysis. Unlike traditional antivirus, EDR is not limited to detecting known malware, but also identifies anomalous behaviors of legitimate software being abused (living-off-the-land attacks, use of PowerShell to exfiltrate data).
File Integrity Monitoring (FIM)
FIM monitors in real time modifications to critical system files (system binaries, configuration files, startup scripts). Every unauthorized modification is reported. It is particularly useful for detecting system compromise: an attacker who modifies a configuration file to maintain persistence is detected by FIM even if they use techniques to evade antivirus.
User Behavior Analytics (UBA)
UBA (or UEBA — User and Entity Behavior Analytics) builds a baseline of each user's normal behavior and flags statistically significant deviations: a user who accesses at 3 AM from a country they have never been in, who suddenly downloads 50 GB of data, or who accesses folders never visited before. This anomaly-based approach detects insider threats, compromised accounts, and lateral movement even when no known attack tools are used.
Practical example — Sandbox and zero-day: an employee receives an email with an Excel file attachment with macros. The email gateway sends the file to the sandbox before delivery. In the isolated environment, opening the file activates a macro that attempts to run PowerShell downloading a payload from the Internet. The sandbox detects: (1) execution of an unapproved macro, (2) child process powershell.exe created by excel.exe, (3) outbound connection to a domain registered 2 days ago. The attachment is blocked and the email is delivered without the attachment with a notification to the employee. The malware was a zero-day: no traditional AV would have detected it by signature.
EXAM TIP 4.1-J — The exam distinguishes sandboxing (isolated execution for behavioral analysis) from virtualization (general isolation of working environments). Although technically similar, in security contexts sandboxing specifically refers to the analysis of untrusted code. If the scenario mentions "analyzing a suspicious file without risking the production system" → sandboxing. If it mentions "isolating a test environment from production" → virtualization.

Objective 4.2 — Security implications of asset management

The management of hardware, software, and data assets is not just an operational or accounting matter: it has direct security implications. An uninventoried asset cannot be protected, patched, or monitored. A device decommissioned without proper sanitization can expose sensitive data. Software acquired without a security evaluation can introduce vulnerabilities from outside. Objective 4.2 covers the entire asset lifecycle from a security perspective: from acquisition to disposal.

Acquisition and Procurement Process

The security of an asset begins before it arrives in the company. The acquisition and procurement process is the moment when the organization can assess the security risks of a product, negotiate contractual requirements with the vendor, and ensure that the asset integrates into the existing security framework. Ignoring security during the purchasing phase often means having to remediate after the fact with higher costs and non-eliminable residual risks.

Vendor and supply chain evaluation
Before purchasing hardware or software, the organization should evaluate: the vendor's reputation in terms of vulnerability response (bug bounty, timely patches, responsible disclosure), the provenance of the hardware supply chain (components from countries with geopolitical risks), support and end-of-life policy (end of support), and the availability of security documentation (SBoM, penetration test reports, certifications such as Common Criteria or FIPS 140-2 for cryptographic products).
Security contracts and SLAs
The contract with the vendor is the legal tool for defining security expectations. Security-relevant elements include: SLA for critical vulnerability response, notification obligations in case of a data breach involving the customer's data, audit rights, data deletion obligations at the end of the contract, and compliance with regulatory standards (GDPR, HIPAA, PCI DSS) if the vendor processes sensitive data on behalf of the organization (subprocessor).
Shadow IT and unapproved software
The main risk of unmanaged procurement is shadow IT: employees who install software or use cloud services not approved by IT because they find the approval processes too slow. Shadow IT exposes corporate data to systems that have not been evaluated, are not configured according to corporate policies, and are not monitored. Solutions include: fast approval processes for low-risk software, an easily accessible catalog of approved software, and network traffic monitoring to detect the use of unauthorized cloud services.
EXAM TIP 4.2-A — In supply chain risk questions, the exam tests the understanding that security must be evaluated before acquisition, not only afterward. If the question asks how to reduce the risk of a hardware component of unknown origin, typical answers include: vendor evaluation, requesting SBoM, testing in an isolated environment before production deployment.

Assignment, Accounting — Ownership and Classification

Every asset must have an identifiable owner and a classification that determines the required level of protection. Without clear ownership no one assumes responsibility for keeping the asset secure and updated. Without classification it is not possible to apply controls proportionate to the value and sensitivity of the asset.

Asset ownership
The owner of an asset (asset owner) is the person or organizational unit responsible for its security over time: deciding who can access it, ensuring it gets updated, approving changes to its configuration, and requesting its retirement when it is no longer needed. Ownership does not necessarily coincide with technical custody: the owner of a customer database (data owner) may be the marketing director, while the DBA is the custodian who operationally manages its security.
Asset classification
Classification assigns each asset a sensitivity level that determines the applicable security controls. Typical classification in a commercial context includes four levels: Public (freely distributable data), Internal (data for unclassified internal use), Confidential (sensitive data with limited access), Restricted/Secret (highly sensitive data with strictly controlled access). In government contexts: Unclassified, Sensitive But Unclassified, Confidential, Secret, Top Secret. Classification guides decisions on encryption, access control, logging, and disposal.
Data owner vs Custodian vs Processor
The data owner determines the classification and access policies. The data custodian technically implements the controls defined by the data owner (backup, encryption, access control lists). The data processor (relevant in GDPR) is the entity that processes data on behalf of the data controller. Confusing these roles leads to accountability gaps: if no one feels they are the owner of a dataset, that dataset tends not to be adequately protected.
EXAM TIP 4.2-B — The exam frequently distinguishes data owner, custodian, and processor. Owner = decides access policy and classification. Custodian = technically implements. In GDPR scenarios: controller = decides the purpose of processing; processor = processes on behalf of the controller. Typical scenario: "Who is responsible for determining the classification of a dataset?" → Data owner (not IT, not the custodian).

Monitoring and Asset Tracking — Inventory and Enumeration

You cannot protect what you do not know you have. Asset inventory is the foundation of any security program: without an updated list of all the organization's systems, devices, applications, and data, it is impossible to apply patches, monitor access, respond to incidents, or plan risk management.

Asset Inventory
The asset inventory is the updated register of all the organization's assets: hardware (servers, workstations, network devices, mobile, printers), software (installed applications, licenses, versions), data (datasets, databases, archives, where they reside), and cloud services. The inventory must include for each asset: unique identifier, type, owner, classification, physical or logical location, software/firmware version, date of last update, and support status. Inventory management is supported by CMDB (Configuration Management Database) platforms or dedicated tools such as ServiceNow, Lansweeper, or Qualys Asset Management.
Asset Enumeration
Asset enumeration is the process of actively discovering resources, often through network scanning. Unlike inventory (which is a database maintained by the organization), enumeration "discovers" what is actually connected to the network at a given moment, including uncatalogued or unauthorized devices (shadow IT hardware). Typical tools: Nmap for host and service discovery, vulnerability scanners like Qualys or Tenable for collecting detailed information, and network discovery features of SIEMs. Periodic comparison between declared inventory and actual enumeration reveals unauthorized devices, forgotten systems, and assets never formally decommissioned.
Continuous asset monitoring
Static inventory is insufficient in dynamic environments where systems are frequently created and destroyed (cloud, containers, VMs). Continuous monitoring keeps the inventory synchronized in real time using cloud APIs, endpoint agents, and scheduled scans. Deviation alerts flag: appearance of new unexpected devices, configuration changes from the baseline, systems that have not connected to the network for unexpected periods.
Practical example — Shadow IT discovered through enumeration: the security team runs an Nmap scan on the internal network and finds a host with IP 10.10.5.44 that responds on port 22 (SSH) and 5432 (PostgreSQL). The host is not in the CMDB inventory. The investigation reveals that a development team installed a mini-PC to run a "temporary" project database six months ago, then forgot it was connected to the network. The PostgreSQL database uses version 10.5 with 47 known vulnerabilities, has default credentials, and contains production exports of customer data. Without active enumeration, this risk would have remained invisible.
EXAM TIP 4.2-C — The exam distinguishes inventory (register maintained by the organization, updated manually or via agents) from enumeration (active discovery via scanning). A typical scenario: "An organization wants to verify that there are no unauthorized devices connected to the network" → enumeration (active scanning), not inventory (which might not include unauthorized ones by definition).

Disposal and Decommissioning — Sanitization, Destruction, Certification, Data Retention

Asset disposal and retirement is the final phase of the lifecycle and, if handled poorly, can nullify years of data protection. A hard drive disposed of without proper sanitization can expose all the data it has ever contained. A "retired" system left connected to the network is an active attack vector. The decommissioning process must be formal, documented, and verifiable.

Sanitization
Sanitization is the process of securely removing data from a medium before it is reused, transferred, or disposed of. It is not simply a matter of deleting files or formatting the disk: standard deletion only removes pointers to the files, not the data itself, which remains recoverable with forensic tools.

Recognized sanitization methods include: overwriting (multiple overwrites with random patterns or standards such as DoD 5220.22-M), crypto-erase (deletion of the encryption key to render data unrecoverable, applicable when the drive uses FDE), and for Flash/SSD devices the firmware-level Secure Erase method (necessary because standard overwriting is less effective due to SSD wear leveling).
Destruction (Physical destruction)
When software sanitization is insufficient or impractical (damaged media, media handling classified data, high regulatory requirements), physical destruction is used. Methods include: shredding (shredding of the medium), degaussing (exposure to an intense magnetic field for magnetic media — not effective on SSDs or optical media), incineration, and pulverizing. Degaussing renders the medium unusable as well as unreadable and must be combined with physical destruction for more modern hard drives.
Certification
The Certificate of Destruction is the document attesting that sanitization or destruction has occurred correctly, when, by whom, and with which method. It is essential when disposal is entrusted to a third party (companies specializing in ITAD — IT Asset Disposition): the contract must include delivery of the certificate for each disposed device. Without certification, the organization has no documentary evidence that the data was destroyed — with legal implications in case of audit or breach.
Data Retention
Data retention defines how long data must be kept before it can be deleted. This is not a discretionary choice: regulations (GDPR, HIPAA, PCI DSS, tax laws, legal retention obligations) define minimum retention periods for different data categories. But there is also the opposite side: data should not be retained longer than necessary, because every piece of data kept beyond necessity is data at risk in case of a breach.

The data retention policy must balance two opposing needs: retaining long enough to meet legal and business obligations, and deleting promptly to reduce risk and storage costs. Retention applies both to the physical disposal of media and to the deletion of digital data from active systems (e.g., logs that must be kept for 12 months but then deleted).
Practical example — Data recovery from a "deleted" disk: a company sells used laptops on eBay after formatting the disks with Windows quick format. A security researcher buys three of these laptops and, using Recuva (a free data recovery tool), manages to recover: Word documents with customer data, Excel spreadsheets with employee salaries, and PDF files of contracts. The quick format had not deleted the data, only marked the space as available. Proper sanitization would have required a complete overwrite or, better yet, disk encryption before formatting followed by deletion of the key.

PITFALL — Degaussing on SSDs: a common mistake is believing that degaussing works on any media. SSDs and Flash drives are not magnetic: degaussing has no effect on them. For SSDs the correct sanitization is firmware Secure Erase or crypto-erase (if the drive uses FDE). If neither is available, physical destruction is the only reliable option.
EXAM TIP 4.2-D — The exam often tests the correct method for specific media. Typical pattern: "What is the most secure method for disposing of an SSD containing classified data?" → physical destruction (shredding), because degaussing does not work and Secure Erase may have limitations on enterprise SSDs. If the question asks for the method to render data unrecoverable AND reuse the media → crypto-erase (if FDE was active) or firmware Secure Erase.

Summary flashcard

  • Secure baseline — three phases — Establish (define), Deploy (distribute), Maintain (monitor and maintain over time)
  • Configuration drift — Gradual deviation of a system from its secure baseline due to patches, interventions, or unmanaged changes
  • SELinux — Implementation of Mandatory Access Control (MAC) on Linux: limits what each process can do even if compromised
  • GPO (Group Policy Object) — Windows mechanism for centrally distributing security settings to domain workstations and servers
  • Port Security (switch) — Limits allowed MAC addresses on a port; countermeasure to MAC flooding
  • BYOD — Employee-owned device; IT has limited control, mandatory containerization to separate corporate and personal data
  • COPE — Company-owned device with personal use allowed; IT has full control including full remote wipe
  • CYOD — Like COPE but the employee chooses the model from a company-approved catalog
  • MDM — Centralized platform for managing, configuring, and protecting mobile devices (enrollment, policies, remote wipe)
  • Input validation (server-side) — Verifies that every external data item meets expected format, length, and characters; countermeasure to SQL injection, XSS, command injection
  • Secure cookie (HttpOnly + Secure + SameSite) — HttpOnly blocks JS access to the cookie (anti-XSS); Secure limits transmission to HTTPS; SameSite mitigates CSRF
  • SAST vs DAST — SAST analyzes source code without executing it (during development); DAST tests the running application by simulating attacks (pre-production)
  • Code signing — Cryptographic signature of software to guarantee authenticity (who signed it) and integrity (not modified after signing); does not guarantee absence of bugs
  • Sandboxing — Execution of untrusted code in an isolated environment; used for behavioral analysis of malware and zero-day detection
  • ICS/SCADA — main compensating control — Network segmentation with industrial DMZ between IT and OT network; air gap where practicable
  • RTOS — Real-time operating system for deterministic response; used in industrial, medical, automotive systems; security often secondary to temporal determinism
  • Degaussing — Effective only on magnetic media (HDD, tapes); has no effect on SSD/Flash
  • Crypto-erase — Sanitization by deleting the encryption key; effective and fast, requires FDE to have been active on the device
  • Certificate of Destruction — Document attesting the correct destruction of a medium; required when disposal is entrusted to third parties
  • Data owner vs Custodian — Owner determines classification and access policies; Custodian technically implements the controls
  • Asset enumeration — Active discovery of devices connected to the network via scanning; reveals shadow IT and unauthorized devices not present in the inventory
  • Data retention — Defines how long data must be kept (legal obligations) and when it must be deleted (risk reduction)
