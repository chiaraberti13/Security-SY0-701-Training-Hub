Objective 1.1 — Security controls

Security controls are measures — technical, organizational, or physical — that an organization implements to reduce security risks. Each control is classified on two axes: the category (what is the nature of the control?) and the type (what is its functional objective?). Understanding this distinction is fundamental because the exam often presents scenarios where you must identify which control is most appropriate.

Control categories

The category indicates where the control operates and who manages it.

Technical
Controls implemented via software, hardware, or digital systems. They are automatic and do not require continuous human action. Examples: firewall, IDS/IPS, encryption, ACL, antivirus, MFA.
Managerial
Controls tied to governance and security planning. They define the framework within which other controls operate. Examples: security policies, risk assessment, audit plans, security awareness programs, gap analysis.
Operational
Controls carried out through daily processes and activities, often performed by people. They concern how work is done day to day. Examples: backup procedures, incident management, password rotation, physical access inspection, staff training.
Physical
Controls that protect the physical environment and physical access to facilities and hardware. They exist independently of the computer network. Examples: locks, fences, cameras, security guards, access badges, vehicle barriers.
EXAM TIP 1.1-A — The most common confusion is between Operational and Managerial: Managerial concerns planning and governance (who decides what must be done), Operational concerns daily execution (who actually does it every day).

Control types

The type indicates when the control intervenes and what its functional objective is.

Preventive
Prevents the security event from occurring. Acts before the damage. Examples: firewall blocking unauthorized traffic, biometric lock on a server door, MFA, patch closing a vulnerability.
Deterrent
Discourages a potential attacker from attempting the action, but does not physically prevent it. Operates on the perception of risk. Examples: visible camera, "video surveillance" sign, policy with explicit penalties, perimeter lighting.
Detective
Identifies and records security events after they have occurred or while they are happening. Does not prevent the event but enables analysis and response. Examples: IDS, SIEM, system logs, user activity audit.
Corrective
Minimizes damage and restores the system to a secure state after an incident. Examples: restore from backup after ransomware, patch applied post-exploit, isolation of an infected host.
Compensating
An alternative measure adopted when the primary control cannot be implemented, for example due to technical incompatibility on legacy systems. It is not a permanent substitute but an accepted solution in the context of a risk assessment. Examples: manual supervision on a system that does not support MFA, isolated network for an ICS device that cannot be patched.
Directive
Guides behavior through documented rules or obligations. Has no direct technical effect but establishes expectations and responsibilities. Examples: Acceptable Use Policy, password policy, SOP, mandatory training.
WARNING — Deterrent vs Preventive — A visible camera is deterrent: it discourages but does not stop the attacker. A biometric lock is preventive: it physically blocks access. "A guard visibly patrolling the perimeter" → deterrent, not preventive.

EXAM TIP 1.1-B — Method for scenario questions: (1) identify the category from the nature of the measure; (2) identify the type by asking yourself WHEN it intervenes — before = preventive · discouragement = deterrent · after/during = detective · restoration = corrective · alternative = compensating · written rule = directive.

Objective 1.2 — Core security concepts

CIA Triad — Confidentiality, Integrity, Availability

The CIA triad is the reference framework for evaluating and designing the security of any system. The complementary triad DAD (Disclosure, Alteration, Denial) describes the corresponding threats.

CIA Property Definition Threat (DAD) Typical attack Primary control
Confidentiality Information is accessible only to authorized subjects. Disclosure Data breach, eavesdropping, shoulder surfing Encryption, ACL, MFA, DLP, data classification
Integrity Data is not modified in an unauthorized manner. What is read is what was written by the legitimate author. Alteration Man-in-the-middle, SQL injection, log tampering Hashing, digital signatures, version control, FIM
Availability Information and services are available to legitimate users when needed. Denial DDoS, ransomware, hardware failure Redundancy, clustering, backup, DDoS mitigation, UPS
Practical example — CIA in a ransomware attack
Ransomware violates Confidentiality (pre-encryption exfiltration), Availability (files inaccessible), and Integrity (files modified without authorization). It is one of the most impactful attacks precisely because it strikes all three pillars simultaneously.

Non-repudiation

Ensures that a subject cannot deny having performed an action. Key difference from authentication: authentication proves who you are at the time of access; non-repudiation proves that the person cannot deny a specific action at a specific time. Main tools: digital signatures, audit logs with intact timestamps, digital certificates.

AAA — Authentication, Authorization, Accounting

Authentication
Identity verification: answers the question "who are you?". Distinguishes between:
  • Person authentication: password, PIN, biometrics, hardware/software token, smart card.
  • System authentication: digital certificates, API key, service account, SSH keys, device attestation.
Authorization
Defines what an authenticated subject is permitted to do: answers "what can you do?". Always presupposes a successful authentication. Main models covered in CH8: MAC, DAC, RBAC, ABAC, Rule-based.
Accounting
Tracks actions performed: answers "what did they do?". Produces the audit trail for forensic investigations, compliance, and anomaly detection. Tools: system logs, SIEM, NetFlow.
WARNING — Authentication vs Authorization — Authentication = identity verification (who you are). Authorization = permission verification (what you can do). HTTP error "403 Forbidden" is an authorization error; "401 Unauthorized" is an authentication error.

Gap Analysis

A systematic comparison between the current security level and the required or desired level, defined by frameworks (NIST, ISO 27001) or regulatory requirements (GDPR, HIPAA). The result is a list of gaps and an action plan to address them. It is a managerial tool that precedes any security improvement strategy.

Zero Trust

An architectural model based on the principle "never trust, always verify". Eliminates implicit trust based on network location: even those already inside the corporate perimeter must be explicitly authenticated and authorized for every access.

Control Plane — the decision plane

Policy Engine
Makes access decisions. Evaluates each request against policies, analyzing identity, context, risk, and subject attributes. Issues the decision: granted, denied, or conditional.
Policy Administrator
Intermediary between Policy Engine and Policy Enforcement Point. Receives the decision from the Policy Engine and communicates it to the PEP. Manages the session (creation and revocation).
Adaptive Identity
Evaluates access context beyond credentials: geographic location, device, time of day, historical behavior, real-time risk level.
Threat Scope Reduction
Each subject gains access only to strictly necessary resources. Limits potential damage in case of compromise.
Policy-Driven Access Control
Every access is based on explicit policies, not on trust assumptions based on network location.

Data Plane — the execution plane

Policy Enforcement Point (PEP)
The point where the decision is concretely applied (gateway, proxy, agent on the endpoint). Allows or blocks traffic based on the received decision.
Subject / System
The requesting subject: user, device, application, or process. In Zero Trust all are to be evaluated regardless of their network location.
Implicit Trust Zones
Implicit trust zones of the traditional model (e.g., "internal network = trusted"). Zero Trust eliminates or minimizes them, requiring verification even within the perimeter.
EXAM TIP 1.2 — Zero Trust — "Which component makes the access decision?" → Policy Engine. "Which component applies the decision?" → Policy Enforcement Point (PEP). Zero Trust does not mean blocking everything: it means not granting implicit trust based solely on network location.

Physical Security

Physical security protects people, facilities, and assets from unauthorized access. An attacker with uncontrolled physical access can bypass almost all logical controls.

ElementDescription and purposeControl type
Bollards Concrete or steel columns embedded in the ground in front of entrances. Prevent vehicles from ramming doors or reaching protected areas. Physical Preventive
Access Control Vestibule (Mantrap) Two sequential doors: the second unlocks only after the first has closed and identity has been verified. Prevents tailgating (entering behind someone without authenticating). Physical Preventive
Fencing Perimeter barrier that defines property boundaries. Height and material determine the level of protection. Physical Preventive + Deterrent
Video Surveillance Visible presence has a deterrent effect; recordings support post-event forensic investigations. Physical Deterrent + Detective
Security Guard Human personnel who can adapt to unexpected situations, perform visual checks, and respond to incidents. Physical Preventive + Deterrent
Access Badge RFID or smart card that authenticates identity for physical access. Leaves an audit trail of every access: who, when, where. Physical Preventive + Detective
Lighting Reduces anonymity in outdoor areas, discourages nighttime illicit activity, improves camera effectiveness. Physical Deterrent
Sensors — Infrared Detect body heat. Effective in low-visibility conditions. Used in alarms and night cameras. Physical Detective
Sensors — Pressure Detect pressure on surfaces (pressure-sensitive floors, mats). Trigger an alarm when the protected area is stepped on. Physical Detective
Sensors — Microwave Emit radio waves and detect variations caused by movement. Cover larger areas than infrared and work through non-metallic materials. Physical Detective
Sensors — Ultrasonic Emit high-frequency (inaudible) sound waves and detect movement by calculating variations in the reflected echo. Physical Detective

Deception and Disruption Technology

Fake assets created to detect, slow down, and study attackers. Fundamental principle: any interaction with a decoy asset is by definition suspicious, because legitimate users have no reason to access it.

TechnologyDescriptionWhat it detectsScale
Honeypot A deliberately exposed system that simulates a real target. Any traffic toward it is suspicious by definition. Scans, exploits, reconnaissance Single system
Honeynet A network of multiple honeypots that simulates an entire infrastructure. Allows observation of lateral movement post-compromise. Lateral movement, tools used Full network
Honeyfile A file with an attractive name (e.g., "passwords.xlsx") containing fake data. Alert generated when it is opened or moved. Unauthorized data access, insider threat Single file
Honeytoken A decoy credential or data item (fictitious account, API key, database record). Alert generated as soon as the token is used anywhere. Use of stolen credentials, exfiltration Single credential
Practical example — Honeytoken in production
An organization creates a fictitious account ("svc_monitoring_backup") with a password saved in the corporate password manager. If someone compromises the password manager and uses those credentials, the SIEM immediately triggers an alert. The account does not really exist: any use of it is proof of compromise.

Objective 1.3 — Change Management

A formal process by which an organization plans, approves, executes, and documents changes to its systems. Without this process, a change can introduce vulnerabilities, cause downtime, or create incompatibilities. The goal is to reduce the risk introduced by each change by ensuring it is evaluated, tested, and reversible.

Business processes

Approval Process
Every change must be formally approved before being implemented in production. In structured environments there is a Change Advisory Board (CAB) that evaluates requests including risk and impact review.
Ownership
Every system has an owner responsible for decisions about it. The owner is the reference for evaluating whether a change is appropriate and for accepting responsibility for the impact.
Stakeholders
All subjects who have an interest in the system or who will be affected by the change. They must be identified and informed before and after the change.
Impact Analysis
Assessment of impact: what can go wrong, which dependent systems will be affected, how much downtime is expected, what security risks the change introduces.
Test Results
The change must be tested in a non-production environment before implementation. The results document that it works as expected and does not introduce regressions.
Backout Plan
A rollback plan documented before implementation. Includes: how to return to the previous configuration, how long it takes, who executes it. It is not improvised during an incident.
Maintenance Window
An approved time window for executing changes — usually during low-impact hours. Changes outside the window require an emergency approval procedure.
Standard Operating Procedure (SOP)
A step-by-step document on how to execute the change. Ensures the process is repeatable and does not depend on the knowledge of a single technician.

Technical implications

Allow Lists / Deny Lists
A change may require updating allow lists (permitted entities) or deny lists (blocked entities). E.g.: a new application might require opening a firewall port or blocking an old protocol.
Restricted Activities
Actions prohibited during maintenance to avoid conflicts. E.g.: do not run backups simultaneously with a database migration.
Downtime
Expected service interruption time. Must be communicated in advance to stakeholders and minimized.
Legacy Applications
Old applications that often do not support updates or changes to the underlying operating system — a typical case for compensating controls.
Dependencies
Systems, libraries, or services that depend on the component being changed. Must be identified before the change to avoid cascading failures.

Documentation and Version Control

Updating Diagrams
Network and architecture diagrams updated after every significant change. Outdated diagrams compromise troubleshooting and incident response.
Updating Policies/Procedures
Policies and operating procedures updated if the change modifies how a process works. An outdated procedure followed by a new technician can create vulnerabilities.
Version Control
Tracks all versions of software, configurations, and scripts. Essential for the backout plan and for the audit trail (who changed what and when).
EXAM TIP 1.3 — In "which action FIRST" or "what is MISSING" questions: impact analysis comes before approval, backout plan comes before execution. Typical scenario: "the technician wants to patch immediately without approval" → answer: an emergency path exists, the process is never bypassed.

Objective 5.1 — Security governance (basics)

A system of rules, structures, and roles that guides how an organization manages information security. For complete governance, compliance, risk management, and audit → CH16 and CH17.

Governance document hierarchy

LevelDocumentKey characteristicExample
1 — broadest Guidelines Non-binding recommendations. Suggest best practices but leave discretion. "It is recommended to use passwords of at least 12 characters"
2 Policies High-level binding rules. State WHAT to do, not how. Approved by management. "All accounts must use MFA"
3 Standards Technical specifications that support policies. State HOW MUCH / WITH WHAT in a precise and measurable way. "Password ≥ 14 characters with uppercase, lowercase, number, and symbol"
4 — most specific Procedures Step-by-step instructions. Describe WHO does WHAT, WHEN, and HOW in a detailed and repeatable manner. "Step 1: create AD account. Step 2: assign RBAC role. Step 3: send encrypted credentials..."
WARNING — Policy vs Standard vs Procedure — Policy = WHAT (binding). Standard = HOW MUCH/WITH WHAT (measurable). Procedure = HOW step-by-step (executable). Guideline = non-binding.

Main policy types

Acceptable Use Policy (AUP)
Defines what users can and cannot do with IT resources. It is generally the first document signed by a new employee.
Information Security Policy
General information security framework. It is the "parent" policy from which all other security policies of the organization derive.
Business Continuity Policy
How the organization will continue to operate during a service disruption. Includes RTO and RPO objectives.
Disaster Recovery Policy
Technical recovery of IT systems after a disaster. More technical than BCP: focuses on IT infrastructure.
Incident Response Policy
How to respond to security incidents: who is notified, which phases to follow, how to document. Guides the PICERL cycle (covered in CH14).
SDLC Policy
Security requirements to be integrated at every stage of software development. Ensures that security is not an afterthought.
Change Management Policy
Governs the process of changes to systems (see section 1.3). Defines who approves, which processes to follow, and what documentation to produce.

Roles and responsibilities for systems and data

RolePrimary responsibilityWho it typically is
Owner Business owner of the data or system. Decides the classification, access policies, and appropriate use. Manager or business unit executive — not necessarily a technical person.
Controller (GDPR) Determines why and how personal data is processed. Establishes the purposes of processing. The organization itself (legal entity).
Processor (GDPR) Processes personal data on behalf of the Controller. Does not decide the purposes: follows the Controller's instructions. Vendor, cloud provider, partner that processes data on behalf of the organization.
Custodian / Steward Technically responsible for data management. Implements controls decided by the Owner: backup, encryption, access, logs. IT team, DBA, system administrator.
EXAM TIP 5.1 — Roles — Owner = business owner, decides the policy. Custodian = IT, implements technical controls. Do not confuse Owner with Administrator: the Owner is the business figure, the Administrator is the technical person (= Custodian).

Summary flashcards

  • Control category vs type — Category = nature of the measure (Technical / Managerial / Operational / Physical). Type = when it intervenes (Preventive / Deterrent / Detective / Corrective / Compensating / Directive).
  • Deterrent vs Preventive — Deterrent discourages but does not block. Preventive physically prevents the action.
  • CIA → DAD — Confidentiality violated = Disclosure. Integrity violated = Alteration. Availability violated = Denial.
  • Non-repudiation — Inability to deny a performed action. Tool: digital signature.
  • AAA in order — Authentication (who are you?) → Authorization (what can you do?) → Accounting (what did you do?).
  • 403 vs 401 — 401 = authentication failed. 403 = authorization denied (user recognized but without permission).
  • Zero Trust: who decides? — Policy Engine decides. Policy Administrator communicates. Policy Enforcement Point applies.
  • Mantrap / Vestibule — Two sequential doors to prevent tailgating.
  • Honeypot vs Honeyfile vs Honeytoken — Decoy system / Decoy file / Decoy credential. Any interaction = alert.
  • Backout Plan — A rollback plan documented before implementation.
  • Governance hierarchy — Guidelines (non-binding) → Policy (WHAT) → Standard (HOW MUCH) → Procedure (HOW).
  • Owner vs Custodian — Owner = decides (business). Custodian = implements (IT).
  • Gap Analysis — Comparison between current and desired security. Produces an action plan.
  • Compensating control — Temporary alternative measure when the primary control cannot be implemented.
  • AUP — Acceptable Use Policy: rules for using IT resources. First document signed by new employees.
