Dalla teoria alla configurazione: applicare la sicurezza all'infrastruttura

L'OBJ 3.2 è pratico: dato uno scenario reale, scegliere e applicare i controlli di sicurezza corretti all'infrastruttura enterprise. Ogni strumento ha un ruolo preciso, un posizionamento ottimale e un contesto d'uso specifico. L'esame mette alla prova la capacità di distinguere strumenti simili in scenari concreti.

Considerazioni infrastrutturali

Device placement (Posizionamento dei dispositivi)

Il posizionamento dei dispositivi (device placement) determina l'efficacia di ogni controllo. I dispositivi di sicurezza devono essere nel percorso del traffico che devono controllare. Un firewall dopo il traffico da filtrare è inutile. La regola: posizionare ogni controllo nel punto che massimizza visibilità ed efficacia con il minimo impatto sulle prestazioni.

Security zones e DMZ

Le zone di sicurezza (security zones) sono segmenti di rete con policy specifiche basate sul livello di fiducia. Il modello classico a tre zone:

Internet (Zona non fidata)
Traffico non controllato. Nessun accesso diretto alle risorse interne. Tutto il traffico deve attraversare almeno un firewall.
DMZ (Demilitarized Zone)
Zona tampone tra internet e rete interna. Ospita servizi pubblici (web, mail, DNS). Protetta da firewall su entrambi i lati. Anche se un server DMZ viene compromesso, l'attaccante non accede direttamente alla rete interna (SCREEN_020).
Rete interna (Zona fidata)
Workstation, server interni, database. Accesso da DMZ/internet regolato da policy esplicite.
Shared services zone
Zone dedicata a servizi condivisi: DNS interno, AD, jump server, monitoring. Raggiungibile da tutte le zone con policy restrittive.
EXAM TIP — Scopo primario della DMZ
"Primary purpose of implementing a DMZ?" → Isolare e aggiungere un layer di sicurezza tra rete interna ed esterna (SCREEN_020). Non per migliorare le performance, non per il backup sicuro.

Failure modes: Fail-open vs Fail-closed

Fail-open
In caso di guasto permette tutto il traffico. Priorità: disponibilità. Usato per controlli fisici di emergenza (porte antincendio) dove un blocco mette a rischio la vita.
Fail-closed
In caso di guasto blocca tutto il traffico. Priorità: sicurezza. Standard per firewall e IPS: meglio interrompere il servizio che lasciare passare traffico non controllato.

Active vs Passive / Inline vs Tap

Active / Inline
Il dispositivo è nel percorso del traffico e può intervenire (bloccare, modificare). Esempi: IPS, firewall, load balancer, WAF. Impatto diretto sulla disponibilità se si guasta.
Passive / Tap
Riceve una copia del traffico (via SPAN/mirror port). Solo osserva, non può bloccare. Nessun impatto sulle prestazioni. Esempi: IDS, packet sniffer, SIEM collector.
Esempio — IPS inline per bloccare pacchetti malevoli (SCREEN_596)
Un team vuole proteggere la rete da pacchetti malevoli esterni. Serve un IPS inline: deve essere nel percorso del traffico per poterlo bloccare prima che raggiunga la rete interna. Un IDS passivo rileva ma non blocca. Anti-malware e host-based firewall agiscono sull'endpoint, non sul flusso di rete.

Appliance di rete

Jump server / Bastion host

Un jump server (bastion host) è l'unico punto di accesso amministrativo a zone di rete protette. Gli amministratori si autenticano sul jump server e da lì raggiungono i sistemi interni. Centralizza e audita tutte le sessioni amministrative.

EXAM TIP — Jump server: cinque scenari
(1) "Admin workstations cannot access DB segment directly" → Jump server (SCREEN_116, 121, 133).
(2) "Layer of security preventing unauthorized access to internal resources" → Jump server (SCREEN_145).
(3) "Minimize admin traffic through security boundary" → Bastion host (SCREEN_148).
(4) "RDP exposed, secure setup" → VPN + jump server inside firewall (SCREEN_209).
(5) "Servers segmented, reachable only from authorized internal systems" → Internal jump server (SCREEN_706).

Proxy server

Forward proxy
Media le richieste degli utenti interni verso internet. Funzioni: web filtering, caching, logging, anonimizzazione degli IP interni. Il client è configurato per usarlo.
Reverse proxy
Media le richieste esterne verso server interni. Il client non sa che esiste. Funzioni: load balancing, TLS termination, caching, protezione identità server. Spesso implementa il WAF.

IPS e IDS

IDS (Intrusion Detection System)
Passivo/tap. Genera alert ma non blocca il traffico. Nessun impatto sulle prestazioni. Utile per visibilità e analisi forense.
IPS (Intrusion Prevention System)
Inline. Blocca attivamente il traffico malevolo. Introduce latenza e rischio di falsi positivi. Necessario per prevenire attacchi in tempo reale.
Signature-based vs Anomaly-based
Signature: alta accuratezza per attacchi noti, zero per zero-day. Anomaly/Behavioral: rileva deviazioni dalla baseline, più falsi positivi ma cattura attacchi nuovi.

Load balancer

Distribuisce il traffico tra più server backend per performance e HA (SCREEN_061, SCREEN_094, SCREEN_631). Funzionalità chiave:

Affinity scheduling (Sticky session)
Stesso utente sempre allo stesso server durante la sessione (SCREEN_043). Critico per app stateful: senza affinity, ogni richiesta va a un server diverso che non ha la sessione dell'utente.
Health checks
Monitora i server backend. Se uno cade, ridistribuisce automaticamente il traffico tra i rimanenti.
SSL offloading (TLS termination)
Il load balancer gestisce TLS, riducendo il carico computazionale sui server backend.

Sicurezza delle porte

Port security e MAC filtering

La port security controlla quali dispositivi possono connettersi a una porta fisica dello switch basandosi sul MAC address. Previene che utenti non autorizzati colleghino dispositivi non approvati alla rete.

MAC filtering: la porta accetta solo MAC address pre-approvati (SCREEN_597). Forma più semplice di port security. Agirabile con MAC spoofing (cambiare il MAC del proprio dispositivo), quindi non è robusta contro attaccanti determinati. La soluzione più robusta è 802.1X.

ATTENZIONE — Port security vs 802.1X
Port security/MAC filtering: blocca dispositivi con MAC non approvati. Semplice ma aggirabile con MAC spoofing.
802.1X: autenticazione crittograficamente forte (certificati, credenziali) prima dell'accesso alla rete. Non aggirabile con MAC spoofing. "Visitor plugs into lobby jack" → port security o 802.1X (SCREEN_290).

802.1X e EAP

IEEE 802.1X è lo standard per il controllo degli accessi basato su porta: il dispositivo deve autenticarsi prima di comunicare sulla rete. Tre componenti:

Supplicant
Il dispositivo client con software 802.1X che gestisce l'autenticazione.
Authenticator
Lo switch o AP Wi-Fi che controlla l'accesso fisico: blocca tutto finché il supplicant non si autentica.
Authentication server
Server RADIUS che verifica le credenziali e autorizza/nega l'accesso, integrando AD o certificati digitali.
EAP (Extensible Authentication Protocol)
Framework di autenticazione usato da 802.1X. Varianti: EAP-TLS (certificati, più sicuro), PEAP (password over TLS), EAP-TTLS. L'autenticazione avviene prima dell'accesso alla rete.
Esempio — Wi-Fi heat map: i primi passi (SCREEN_676)
Per sviluppare una heat map Wi-Fi: (1) ottenere/creare la pianta dell'ufficio (base cartografica per mappare i dati) e (2) girare metodicamente nell'ufficio misurando la potenza del segnale (raccolta dati sul campo). Non si inizia accedendo agli AP né si misurano i cavi.

Tipi di firewall

WAF (Web Application Firewall)
L7 HTTP/HTTPS. Filtra attacchi web: SQLi, XSS, CSRF, buffer overflow su parametri HTTP. Prima linea di difesa per applicazioni web pubbliche. Scenario attacco web app → WAF, non NGFW.
UTM (Unified Threat Management)
All-in-one: firewall + IPS + VPN + AV gateway + web filtering + anti-spam. Per PMI con risorse limitate. Semplicità di gestione ma single point of failure.
NGFW (Next-Generation Firewall)
Estende il firewall tradizionale con DPI applicativa, identificazione applicazioni (non solo porta), user identity awareness, TLS inspection, threat intelligence integration.
Layer 4 vs Layer 7
L4 (stateful): IP, porta, stato TCP. Veloce, non capisce il payload applicativo. L7 (applicativo): ispeziona il contenuto, distingue le applicazioni, policy per utente. Più lento ma molto più granulare.

Comunicazioni e accesso sicuro

VPN e Remote access

Una VPN crea un tunnel cifrato su internet per connettere dispositivi remoti alla rete aziendale (SCREEN_653). Tipi principali:

Client-to-site (Remote access VPN)
Singolo dispositivo si connette alla rete aziendale tramite client VPN. Per lavoratori remoti.
Site-to-site VPN
Connessione permanente tra due reti (es. sede + filiale). Gestita da VPN gateway, non da client software.
Split tunneling
Solo il traffico verso la rete aziendale usa la VPN; il traffico internet esce direttamente. Riduce il carico sul concentrator ma riduce la visibilità dell'IT.
EXAM TIP — VPN + Jump server per accesso sicuro
"RDP esposto su internet" → VPN + jump server inside firewall (SCREEN_209). La VPN cifra il canale; il jump server è il punto di accesso controllato. Non cambiare solo la porta RDP (security through obscurity).
"Accesso remoto sicuro minimizzando l'esposizione della rete interna" → VPN (SCREEN_653).

TLS vs IPSec

TLS (Transport Layer Security)
L5-L6. Protegge singole connessioni applicative (HTTPS, SMTP/TLS). Semplice da configurare. Non protegge l'intero stack IP.
IPSec (Internet Protocol Security)
L3. Protegge l'intero payload IP, incluse le informazioni di routing. Modalità Tunnel: cifra il pacchetto IP completo con nuovo header. Per VPN site-to-site con tutto il traffico cifrato (SCREEN_299).
EXAM TIP — IPSec per traffico inter-rete
"Third party access between two networks, encryption on ALL traffic traversing" → IPSec (SCREEN_299). EAP è per autenticazione. SD-WAN ottimizza il routing. TLS protegge singole connessioni, non tutto il traffico IP.

Protocolli sicuri vs non sicuri

Non sicuri (trasmettono in chiaro)
Telnet (23), FTP (21), HTTP (80), SNMP v1/v2 — trasmettono credenziali e dati senza cifratura (SCREEN_037).
Sostituzioni sicure
SSH ← Telnet: autenticazione con chiavi asimmetriche + connessione cifrata (SCREEN_040, SCREEN_397). SFTP/FTPS ← FTP. HTTPS ← HTTP. SNMPv3 ← SNMPv1/v2.
EXAM TIP — SSH: i due motivi vs Telnet (SCREEN_040)
"Perché passare a SSH?" → (A) chiavi asimmetriche per autenticazione + (D) connessione cifrata. Non perché permette comandi superuser (anche Telnet li permette). Non perché non richiede pre-shared key (SSH le usa).
ATTENZIONE — VoIP: i protocolli (SCREEN_403)
RTP, SIP, H.323, SRTP sono tutti protocolli VoIP. Regole firewall per questi protocolli → supporto VoIP. Non confondere con RTOS, SoC, o HVAC.

Zero Trust per la rete enterprise

"Secure zone + company-wide access control policy + reduce scope of threats" = Zero Trust (SCREEN_220). Nessuna zona implicitamente fidata, ogni accesso verificato, policy applicata consistentemente. Non è AAA (solo autenticazione), non è CIA (principi dei dati).

DMZ (De-Militarized Zone) Clicca per girare

La zona tampone isolata da firewall per l'esposizione di database o servizi pubblici, per evitare che la compromissione di un server esterno garantisca accessi diretti alla rete aziendale.

Fail-Open vs Fail-Closed Clicca per girare

Fail-Open: privilegia l'Availability (disponibilità, porte tagliafuoco che si aprono in assenza di corrente).
Fail-Closed: privilegia la Security (firewall che blocca tutto).

802.1X Network Authentication Clicca per girare

Protocollo ubiquo client-switch-RADIUS. Richiede imperativamente autenticazione formale (identità sicura) prima che la porta fisica dello switch venga abilitata sul server.

Active/Inline vs Passive/Tap Clicca per girare

Active/Inline: il sensore è collocato sul canale di traffico e può decidere di droppare pacchetti (IPS).
Passive/Tap: riceve una copia del traffico silente, senza poter mai bloccare (IDS).
