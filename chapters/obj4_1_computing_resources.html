La messa in sicurezza quotidiana delle risorse aziendali

Il Dominio 4 riguarda la quotidianità operativa della cybersecurity. Questo obiettivo si concentra su come proteggere concretamente le risorse di elaborazione e calcolo (computing resources) nell'impresa. Ciò include workstation, server, dispositivi mobili, dispositivi di rete, sistemi IoT e le reti wireless su cui tali sistemi comunicano. Copriremo inoltre le discipline applicative (come evitare l'iniezione di codice malevolo) e i controlli lato host — come l'isolamento dei dispositivi compromessi e il monitoraggio degli endpoint.

Baseline di sicurezza (Secure Baselines)

Una baseline di sicurezza (secure baseline) è una configurazione standardizzata minima che assicura che un dispositivo o un sistema operativo soddisfi i requisiti minimi di sicurezza approvati dall'azienda. Rappresenta la fondamenta necessaria per verificare la conformità e contenere i rischi.

Il ciclo di gestione delle baseline comprende tre fasi operative cruciali:

Stabilire (Establish)
Definire lo standard di sicurezza iniziale utilizzando linee guida interne e benchmark riconosciuti a livello internazionale, come i CIS Benchmarks (Center for Internet Security) o le DISA STIGs (Defense Information Systems Agency Security Technical Implementation Guides). Esempio: stabilire che tutte le workstation Windows devono avere BitLocker abilitato, porte USB disabilitate e la password di root disattivata.
Distribuire (Deploy)
Applicare lo standard di configurazione stabilito a tutti i dispositivi nuovi ed esistenti. Questa fase solitamente è automatizzata tramite l'uso di Group Policy Objects (GPO) in Active Directory, profili di configurazione di Mobile Device Management (MDM), o strumenti di Infrastructure as Code (IaC) e Configuration Management (come Ansible, Terraform o Chef) per garantire la scalabilità e ridurre gli errori di imposizione manuale.
Gestire / Mantenere (Maintain)
Monitorare costantemente i sistemi per individuare la deviazione della configurazione (configuration drift) e applicare patch aggiornate nel tempo. Il monitoraggio continuo rileva le modifiche non intenzionali o non autorizzate alla baseline stabilita.

Applicazione delle Baseline (Baseline Enforcement)

L'applicazione delle baseline (baseline enforcement) è il meccanismo con cui si verifica programmaticamente e costantemente se i sistemi sono conformi alla configurazione standard e si interviene per bloccare o correggere le violazioni.

Esempio operativo — Script di accesso utente (SCREEN_660)
Un amministratore di sistema implementa uno script personalizzato che viene eseguito automaticamente ad ogni accesso dei dipendenti. Lo script controlla la versione corrente del sistema operativo, verifica che tutte le patch critiche siano installate e che nei sistemi stiano girando solo le applicazioni concordate dall'azienda.

Questo tipo di controllo rappresenta l'esempio perfetto di Baseline Enforcement (Applicazione delle Baseline). Lo script non implementa il provisioning delle risorse, né si limita a definire le regole di autenticazione ("guardrails" o "policy enumeration"), ma le impone ed applica attivamente al login del sistema.

Hardening dei diversi sistemi aziendali (Hardening Targets)

Ogni classe di risorsa informatica presenta la propria peculiare superficie d'attacco e richiede passaggi specifici di indurimento (hardening) prima e dopo l'avvio operativo.

Hardening dei Server

Quando si immette un nuovo server in produzione, il tempismo e la priorità dei controlli sono fondamentali per evitare compromissioni repentine prima ancora che il server inizi a lavorare.

EXAM TIP — Primi passi per la sicurezza dei server
(1) Il primissimo passo (Before everything else): Quando viene installato un nuovo server, prima ancora di disattivare servizi o configurare liste di accesso (ACL), l'attività prioritaria assoluta è Aggiornare il software alla versione corrente (Update the current version of the software) (SCREEN_728). Un sistema non patchato è vulnerabile fin dal momento in cui tocca la rete.
(2) Prima di andare "Live": Un amministratore deve preparare un server web prima che esso diventi pubblicamente accessibile. Le due azioni raccomandate ed essenziali sono Disabilitare gli account di default corporativi (Disable default accounts) e Rimuovere i servizi non necessari (Remove unnecessary services) (SCREEN_353).

Oltre al sistema operativo server, è fondamentale **mettere in sicurezza l'host virtuale (Harden the virtual host)** sul quale gira la macchina virtuale del server web (SCREEN_547) prima che questa diventi visibile su internet.

Dispositivi di Rete: Switch e Router

Anche l'infrastruttura di routing e switching deve essere oggetto di hardening per impedire agli attaccanti interni od esterni di deviare o catturare il traffico di transito.

Sostituzione dei protocolli in chiaro (Telnet → SSH)
L'hardening fondamentale per i dispositivi di rete è la cifratura delle sessioni amministrative. SSH (Secure Shell - porta TCP 22) deve essere utilizzato al posto di Telnet (porta TCP 23) (SCREEN_588). Telnet espone le credenziali di root e le configurazioni in chiaro sul cavo di rete, offrendo terreno facile a sniffers locali.
Disattivazione di porte e servizi inutilizzati
Le porte fisiche dello switch non collegate devono essere administratively disabilitate e messe in una VLAN priva di connettività, e i protocolli di discovery non necessari (come CDP o LLDP) spenti se non strettamente indispensabili.

Sistemi Embedded, IoT e RTOS

Molti ecosistemi moderni usano dispositivi che non possiedono le medesime interfacce d'amministrazione o le capacità computazionali di un server standard, ma rappresentano comunque enormi criticità di sicurezza:

  • Sistemi Embedded (Sistemi integrati): Spesso soffrono di indisponibilità di patch nel lungo periodo. Richiedono schermatura di rete, disabilitazione delle porte di diagnostica (come i pin JTAG o porte seriali ad uso interno) e cambio tassativo delle credenziali di fabbrica.
  • Sistemi SCADA / ICS (Industrial Control Systems): Sovraintendono reti manifatturiere o d'utilità nazionale (acqua, elettricità). Devono essere isolati drasticamente mediante **Air-Gap (Isolamento Fisico)** o forti politiche di segmentazione logica dalla rete aziendale generica.
  • RTOS (Real-Time Operating Systems): Sistemi operativi deterministici in tempo reale dedicati a scopi medici o dell'aviazione. Richiedono rigido isolamento del traffico e minimizzazione assoluta delle app eseguibili.
  • Dispositivi IoT (Internet of Things): Sensori, caldaie, o fotocamere di sicurezza. Vediamo come installarli in sicurezza.
Scenario d'esame — Installazione di telecamere di sicurezza IP (SCREEN_284)
Un tecnico sta installando una nuova fotocamera di sicurezza IP (dispositivo IoT). Qual è l'attività da condurre preliminarmente o parallelamente al fine di assicurare l'hardware?

Risposta: Eseguire un rilievo/rilievi sul campo (Conduct a site survey). Il site survey assicura che il posizionamento fisico e radio della fotocamera e degli access point copra l'area corretta senza esporre porte fisiche a manomissioni libere e assicura che il segnale non filtri all'esterno del perimetro protetto (leakage).

Installazione e sicurezza dei dispositivi Wireless

La rete wireless estende lo spettro radio oltre le pareti fisiche dell'ufficio, rendendo indispensabili metodologie di pianificazione e autenticazione specifiche.

Site Survey e Heat Map

Rilievo RF sul campo (Site Surveys)
Consente di pianificare la disposizione fisica degli Access Point analizzando ostacoli fisici (muri in cemento, metalli) e interferenze, prima della messa in funzione.
Mappe termiche radio (Heat Maps)
Rappresentazioni grafiche dell'intensità del segnale Wi-Fi nell'edificio. Permettono agli amministratori di regolare le potenze di trasmissione degli AP in modo da limitare la dispersione radio ("leakage") nei parcheggi esterni o strade limitrofe, dove gli attaccanti potrebbero tentare attacchi offline.

Sistemi di Ospiti e Captive Portal

Nelle reti aperte destinate ai visitatori o clienti, non è possibile imporre l'autenticazione centralizzata aziendale. Per ovviare a ciò si usa un portale catturante.

Portale Captivo (Captive Portal)
Una pagina web a cui l'utente viene reindirizzato forzatamente quando tenta di navigare su internet per la prima volta. Richiede un'interazione prima di garantire l'accesso.
Scenari pratici di Captive Portal
  • Caffetteria (SCREEN_379): Se il proprietario di un bar desidera limitare l'accesso alla rete Wi-Fi esclusivamente ai clienti paganti richiedendo un codice dallo scontrino, la soluzione ideale è implementare un Captive Portal che verifichi il numero della ricevuta.
  • Ospiti aziendali (SCREEN_408): Se la squadra legale di un'impresa è preoccupata dalle responsabilità legali derivanti dalle azioni di visitatori esterni, la soluzione ottimale è configurare un **Guest Captive Portal** che obblighi i visitatori ad accettare formalmente i Termini e le Condizioni d'uso prima di concedere l'accesso a Internet.

Soluzioni e modelli per i dispositivi dei dipendenti

I dispositivi mobili contengono grandi quantità di dati aziendali e accedono sistematicamente alla intranet. Per tenerli sotto controllo, si adotta una suite dedicata di monitoraggio e gestione.

MDM: Tracciamento, Forzatura e Sicurezza

Un sistema di gestione dei dispositivi mobili (Mobile Device Management - MDM) consente il controllo completo dei telefoni e dei tablet dell'azienda. Esso permette di:

  • Mettere in sicurezza i dispositivi in caso di smarrimento forzandone l'inizializzazione e cancellando i dati (Remote Wipe) (SCREEN_367, SCREEN_501).
  • Garantire che sui dispositivi personali dei dipendenti (BYOD) possano essere installate esclusivamente applicazioni approvate dall'ufficio (SCREEN_605).
  • Gestire in modo massivo i tablet aziendali distribuendo funzionalità e applicazioni mancanti che bloccano la produttività (SCREEN_604).

Modelli di distribuzione mobile (Deployment Models)

Modello di dispiegamento Spiegazione ed Uso Politiche d'esame da definire
BYOD (Bring Your Own Device) I lavoratori utilizzano i propri telefoni personali per scopi professionali. Massima flessibilità ma difficile bilanciamento privacy/sicurezza. Richiede containerizzazione tramite MDM per separare dati aziendali da dati privati.
COPE (Corporate-Owned, Personally Enabled) I dispositivi sono di proprietà dell'azienda, ma i lavoratori sono autorizzati ad utilizzarli anche per questioni personali. La policy COPE deve obbligatoriamente includere controlli critici d'esame: Cancellazione remota (Remote wipe) e Cifratura dei dati (Data encryption) (SCREEN_544).
CYOD (Choose Your Own Device) L'azienda acquista l'hardware, ma consente ai lavoratori di selezionare il modello preferito all'interno di una lista approvata. Facilita la gestione dei profili MDM standardizzati.
ATTENZIONE — Jailbreaking vs Side-loading (SCREEN_346)
  • Jailbreaking (Sblocco): La rimozione forzata dei limiti imposti dal produttore sul sistema operativo del dispositivo (es. root su Android o Jailbreak su iOS). Permette all'utente di accedere al codice core e installare unauthorized software o custom tweaks abilitando nuove feature non previste di fabbrica.
  • Side-loading (Caricamento laterale): Il semplice atto di installare pacchetti applicativi (.apk o .ipa) bypassando il market store ufficiale, senza richiedere necessariamente lo sblocco del root del firmware.

Impostazioni e protocolli di sicurezza Wireless

Wi-Fi Protected Access 3 (WPA3)
Lo standard crittografico moderno per Wi-Fi. Sostituisce la chiave precondivisa vulnerabile (PSK) di WPA2 con il protocollo SAE (Simultaneous Authentication of Equals). SAE offre protezione robusta contro attacchi offline di brute-force sulle password "leggere", fornendo quella che viene denominata Forward Secrecy.
RADIUS (Remote Authentication Dial-In User Service) & Autenticazione AAA
Consente l'autenticazione individuale e centralizzata. Invece di condividere la stessa password Wi-Fi tra 500 dipendenti, ogni dipendente inserisce le credenziali aziendali del dominio. Lo switch/AP inoltra la richiesta al server RADIUS (AAA) che convalida l'accesso tramite protocolli come 802.1X.

Sicurezza delle Applicazioni (Application Security)

Un codice non indurito espone le organizzazioni ad attacchi devastanti finiti poi nei log e sfruttabili per rubare credenziali o dati sensibili.

Validazione dell'Input (Input Validation)

La validazione dell'input (input validation) o sanificazione dell'input (input sanitization) è il controllo prioritario assoluto per prevenire attacchi di tipo Injection (come SQLi) e Cross-Site Scripting (XSS).

Esempi d'Esame — Tecniche di validazione
  • Espressioni Regolari (SCREEN_078, SCREEN_191): Un'organizzazione aggiorna la policy di sviluppo imponendo l'uso di Regular Expression nel codice sorgente per estirpare simboli nocivi come $, |, ;, &, `, ? provenienti dai form Web dell'app. Questa configurazione rappresenta una tecnica eccitante di Input Validation (Validazione dell'Input).
  • Evitare Cross-Site Scripting (SCREEN_084, SCREEN_342): Un field di un form è aperto a payload pericolosi come <script>alert("Warning!");</script>. L'analista consiglia di implementare Input Sanitization / Validation per ripulire l'input.
  • Protezione Log4j ed User-Agent (SCREEN_483): Un analista osserva la riga di log:
    149.32.228.10 - - [28/Jan/2023:16:32:45] "GET / HTTP/1.0" User-Agent: ${/bin/sh/ id} 200 397
    L'attacco (Injection / Remote Code Execution via log injection o exploit Apache) viene mitigato in assoluto applicando Input Sanitization sullo User-Agent.

Peer Review e Prevenzione degli Insider (Peer Review and Approval)

EXAM TIP — Blocco degli Insider infiltrati
"Which of the following practices would be best to prevent an insider from introducing malicious code into a company's development process?" (SCREEN_082, SCREEN_236)
→ Peer review and approval (Revisione e approvazione tra pari).

Nessun sistema di scansione vulnerabilità automatico o test QA garantisce che un programmatore interno malevolo non inserisca intenzionalmente una backdoor astuta e "pulita" nel sorgente. Solo una meticolosa revisione manuale guidata da un secondo programmatore indipendente (Peer Review) seguita da autorizzazione formale scherma questo rischio.

Analisi del codice: Silente vs Runtime

Analisi Statica del codice (Static Code Analysis / SAST)
Analisi del codice sorgente condotta "a freddo" senza eseguire il programma. Ideale per cercare vulnerabilità strutturali, credenziali hardcoded o firme note di file malevoli (Static analysis) (SCREEN_622).
Analisi Dinamica del codice (Dynamic Analysis / DAST)
Verifica e test di sicurezza eseguiti mentre l'applicazione è in esecuzione (at runtime) (SCREEN_619). Permette di valutare le risposte reali dell'app a sollecitazioni fittizie e carichi di traffico.

Obfuscation e Code Signing

Offuscamento del codice (Code Obfuscation)
La tecnica preferita per ridurre drasticamente la probabilità che il software proprietario esportato o ceduto ai clienti venga decompilato o retroingegnerizzato (Reverse engineering), proteggendo la proprietà intellettuale (SCREEN_369).
Firma digitale del codice (Code Signing)
Garantisce l'integrità e l'autenticità del codice d'installazione. L'utente ricevente sa con certezza che l'eseguibile proviene dal fornitore dichiarato e che non è stato alterato da terze parti.

Sicurezza degli Host e degli Endpoint (Host Security & Controls)

Una volta immessi sulla rete, gli host richiedono protezioni individuali capaci di operare anche qualora il perimetro esterno crolli.

Controllo degli Accessi di Rete (NAC) e Superfici Fisiche

Il Network Access Control - NAC (Controllo degli accessi di rete) convalida l'identità, lo stato delle patch e i sistemi antivirus di qualsiasi dispositivo che tenti di accoppiarsi alla rete prima di accordare un IP.

EXAM TIP — Proteggere la superficie cablata (Wired Port)
(1) "After a security incident, a system administrator asks the company to buy a NAC platform. Which attack surface is trying to protect?" (SCREEN_495)
→ Wired (Cablata). Il NAC impedisce collegamenti abusivi sui cavi ethernet aziendali.
(2) "A penetration tester finds an unused Ethernet port during an on-site test. Upon plugging a device, the tester is assigned an IP address allowing enumeration. Which solution should an administrator implement?" (SCREEN_563)
→ Port Security (o 802.1X). La port security spegne la porta switch se rileva un indirizzo MAC non registrato o forza la richiesta di profili di sicurezza individuali.

EDR, Lateral Movement e Isolamento

Host Isolation (Isolamento dell'host / host-based isolation)
Il meccanismo in assoluto più sicuro per garantire che una risorsa infetta diventi totalmente impermeabile ed inaccessibile ad ogni altra entità connessa alla rete aziendale (SCREEN_681).
Host-based Firewall (Firewall lato host)
Fornisce la protezione più fine contro flussi e comunicazioni insicure inviate da e verso la singola workstation o server (SCREEN_499).
Endpoint Detection and Response (EDR)
Piattaforma avanzata che monitora in real-time i comportamenti malevoli su dischi, processi e registri. È la risposta per bloccare l'esecuzione laterale di ransomware e virus tra nodi della stessa rete (SCREEN_308).
Mitigare i click sui Phishing Link (SCREEN_192)
Un test aziendale di Phishing mostra tassi di errore inaccettabili da parte degli utenti, i quali cliccano con alta frequenza su link pericolosi. Al fine di circoscrivere ed abbattere l'impatto distruttivo in tempo reale di questi click, l'analista deve **configurare le politiche EDR lato endpoint** per impedire l'esecuzione automatica di codici e software scaricati a seguito del click.

Application Allow List (Lista di programmi autorizzati)

I sistemi antivirus tradizionali cercano firme note (database delle minacce). Se la minaccia è ignota (Zero-Day o ransomware mai catalogato), essi falliscono.

Application Allow List (Lista di programmi autorizzati)
Filosofia di sicurezza "Zero-Trust" in cui l'host blocca programmaticamente l'avvio di qualsiasi programma, script o eseguibile ad eccezione di quelli esplicitamente registrati e approvati in lista. Rappresenta la difesa d'oro per impedire l'esecuzione accidentale di malware da parte del personale (SCREEN_237) e bloccare programmi sconosciuti (SCREEN_180).

Gestione Endpoints, IaC e Wiping dei Laptop dismessi

Per governare parchi computer formati da migliaia di nodi, si scelgono software centralizzati di Endpoint Management.

Endpoint Management Software (Software di gestione degli endpoint)
Fornisce una singola console centralizzata atta a verificare l'insorgenza di modifiche non autorizzate del parco server e delle workstation aziendali, offrendo monitoraggio costante del software caricato (SCREEN_225).
Controllo di Versione (Version Control)
Quando i server sono configurati tramite modelli software di Infrastructure as Code (IaC), lo strumento più adatto per tracciare e tenere storicizzate le modifiche al codice di build è un applicativo di **Version Control (Controllo di Versione - es. Git)** (SCREEN_654).
Bonifica (Wiping) dei Laptop pensionati
Per impedire il furto di dati dei clienti conservati all'interno di hard disk appartenenti a laptop dismessi (pensionati/decommissionati), il metodo più efficiente per costi e scalabilità che azzera i rischi è procedere al **Wiping (Sovrascrittura sicura dei dati del disco)** (SCREEN_567) rispetto alla più costosa frammentazione o distruzione fisica del bene aziendale riutilizzabile.

Static vs Dynamic Analysis Clicca per girare

Static Analysis: scansione a codice spento interrogando firme e pattern noti.
Dynamic Analysis (DAST): monitora in tempo reale l'applicazione at runtime.

Application Allow List vs Block List Clicca per girare

Allow List: approccio protettivo massimo che inibisce preventivamente l'esecuzione di qualsiasi programma sconosciuto o non censito nei cataloghi autorizzativi.
