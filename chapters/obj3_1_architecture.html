Architettura e sicurezza: ogni modello porta le sue implicazioni

La scelta dell'architettura IT non è solo una decisione tecnica o economica: è una decisione di sicurezza. Ogni modello — cloud, on-premises, containerizzato, IoT, ICS/SCADA — porta con sé un profilo di rischio specifico, vincoli di patching, livelli di controllo e considerazioni di compliance diverse. L'obiettivo 3.1 chiede di confrontare e contrapporre queste implicazioni per scegliere e difendere architetture appropriate ai requisiti di sicurezza.

Cloud: modelli, responsabilità e rischi

Modelli di servizio cloud (IaaS · PaaS · SaaS)

Il cloud computing offre tre modelli di servizio principali, che differiscono per il livello di astrazione fornito al cliente e, di conseguenza, per la distribuzione delle responsabilità di sicurezza.

IaaS — Infrastructure as a Service
Il provider fornisce l'infrastruttura fisica: hardware, rete, storage, virtualizzazione. Il cliente gestisce sistemi operativi, middleware, runtime, applicazioni e dati. Massima flessibilità, massima responsabilità del cliente. Esempi: AWS EC2, Azure Virtual Machines, Google Compute Engine.
PaaS — Platform as a Service
Il provider fornisce IaaS + OS + runtime + middleware. Il cliente gestisce solo applicazioni e dati. Ideale per gli sviluppatori: non devono occuparsi dell'infrastruttura sottostante. Esempi: Heroku, AWS Elastic Beanstalk, Azure App Service.
SaaS — Software as a Service
Il provider fornisce l'intera stack: infrastruttura + piattaforma + applicazione. Il cliente usa solo il software. Minima responsabilità tecnica del cliente, massima dipendenza dal provider. Esempi: Microsoft 365, Salesforce, Google Workspace.
Public Cloud (Cloud pubblico)
Infrastruttura condivisa tra molti clienti (multi-tenant), gestita da un provider terzo. I clienti affittano "pezzi" della capacità totale. Il rischio principale è la condivisione fisica delle risorse con altri tenant.
Private Cloud (Cloud privato)
Infrastruttura cloud dedicata a una singola organizzazione. Maggiore controllo e sicurezza rispetto al cloud pubblico, ma costi molto più elevati. Può essere on-premises o hosted da un provider dedicato.
Hybrid Cloud (Cloud ibrido)
Combinazione di cloud pubblico e privato (o on-premises) con integrazione e orchestrazione tra i due ambienti. Permette di mantenere i dati sensibili on-premises/private cloud e usare il cloud pubblico per carichi di lavoro meno critici.
Community Cloud
Infrastruttura cloud condivisa tra organizzazioni con requisiti simili (es. settore sanitario, governo). A metà tra public e private: condivisa ma solo con un gruppo ristretto di organizzazioni con esigenze e requisiti di compliance compatibili.
EXAM TIP — Il maggior rischio del cloud computing
"Biggest risk in cloud computing?" → Lack of control. Non mancanza di disponibilità (i cloud provider offrono SLA elevati), non mancanza di accountability (i contratti definiscono le responsabilità), non mancanza di responsabilità (il modello di responsabilità condivisa la definisce). Il rischio fondamentale è che il cliente cede il controllo dell'infrastruttura sottostante al provider.

Responsibility matrix (Matrice delle responsabilità)

Il modello di responsabilità condivisa (shared responsibility model) definisce con precisione chi è responsabile di cosa in un ambiente cloud. È il documento fondamentale per capire dove finisce la responsabilità del provider e dove inizia quella del cliente. La responsibility matrix è il documento specifico che formalizza questa divisione per un contratto cloud specifico.

EXAM TIP — IaaS: il database è responsabilità del cliente
"In IaaS, chi è responsabile di proteggere il database dell'azienda?" → Il cliente (client). In IaaS il provider gestisce hardware, rete e virtualizzazione. Il cliente gestisce OS, middleware, applicazioni e dati — incluso il database. Il DBA è un ruolo interno al cliente, non un'entità del modello. Il third-party vendor non è il cloud provider.

Hybrid considerations (Considerazioni sul modello ibrido)

Un'architettura ibrida (hybrid) combina ambienti on-premises e cloud, creando sfide di sicurezza specifiche legate all'integrazione tra i due ambienti. Il principale punto di attenzione è la sicurezza dei canali di connettività tra on-premises e cloud, la gestione delle identità cross-environment (federazione delle identità), e la coerenza delle policy di sicurezza applicata in entrambi gli ambienti.

La geolocation policy è uno strumento tipico degli ambienti cloud ibridi e SaaS: limita l'accesso a risorse cloud in base alla posizione geografica dell'utente, bloccando accessi da paesi ad alto rischio. È più efficace della semplice cifratura o del data masking per prevenire accessi non autorizzati da specifiche giurisdizioni.

Third-party vendors (Fornitori terzi)

L'uso di fornitori cloud terzi introduce rischi di supply chain e di dipendenza dal vendor. Le considerazioni di sicurezza includono: la revisione del contratto (SLA, responsabilità, diritto di audit), la verifica delle certificazioni di sicurezza del provider (ISO 27001, SOC 2, FedRAMP), la gestione del vendor lock-in (difficoltà a cambiare provider), e la supervisione continua della postura di sicurezza del provider tramite audit periodici.

SASE — Secure Access Service Edge

SASE (Secure Access Service Edge) è un'architettura cloud-native che converge le funzionalità di rete (SD-WAN) e di sicurezza (CASB, FWaaS, ZTNA, SWG) in un unico servizio cloud distribuito. È progettata specificamente per organizzazioni con workforce distribuita o completamente remota: invece di fare passare tutto il traffico remoto attraverso un VPN concentrator centrale (collo di bottiglia), ogni utente si connette al nodo SASE più vicino che applica le policy di sicurezza localmente.

Componenti chiave di SASE
SD-WAN: ottimizzazione del routing del traffico. SWG (Secure Web Gateway): filtraggio web. CASB (Cloud Access Security Broker): visibilità e controllo sull'uso dei SaaS. ZTNA (Zero Trust Network Access): accesso alle applicazioni basato su identità. FWaaS (Firewall as a Service): regole firewall distribuite nel cloud.
Quando usare SASE vs VPN
SASE è la risposta quando: workforce completamente o prevalentemente remota, necessità di ridurre il carico sul VPN concentrator, necessità di web filtering integrato, scalabilità automatica. La VPN tradizionale è adatta a connettività site-to-site con traffico prevedibile.
EXAM TIP — SASE: i tre scenari chiave
(1) "Branch offices, encrypted access, web filtering, firewall capabilities, existing cloud connections" → SASE.
(2) "VPN concentrator overloaded, remote workforce, encrypted access + web monitoring" → SASE.
(3) "Fully remote workforce, redundant, VPN + integrated software firewall" → SASE.
SD-WAN ottimizza solo il routing; non include le funzioni di sicurezza integrate (web filtering, ZTNA). VPN site-to-site non si scala facilmente per workforce distribuita.

Architetture moderne

Infrastructure as Code (IaC)

Infrastructure as Code (IaC) è la pratica di gestire e provisionare l'infrastruttura IT attraverso file di configurazione leggibili dalla macchina invece di processi manuali. Con IaC, la configurazione di server, reti, database e altri componenti è definita come codice (Terraform, Ansible, CloudFormation, Puppet, Chef) che può essere versionato, testato, revisionato e riutilizzato.

Dal punto di vista della sicurezza, IaC permette di standardizzare rapidamente la configurazione di molti server, ridurre il configuration drift, e applicare automaticamente le secure baseline a ogni nuovo deployment. Un'organizzazione con grandi operazioni IT che vuole controllare, standardizzare e ridurre i tempi di provisioning dei server → IaC.

Serverless (Senza server)

L'architettura serverless permette di eseguire codice applicativo senza gestire alcuna infrastruttura server: il provider cloud gestisce automaticamente il provisioning, la scalabilità e la disponibilità dell'infrastruttura sottostante. Il cliente scrive solo le funzioni applicative (AWS Lambda, Azure Functions, Google Cloud Functions) e paga solo per il tempo di esecuzione effettivo.

È la soluzione di hosting cloud-based a basso costo: non si paga per server idle, non si gestisce l'OS, non si applicano patch all'infrastruttura. Dal punto di vista della sicurezza, riduce drasticamente la superficie di attacco infrastrutturale (nessun OS da hardenire), ma introduce nuove sfide: gestione delle autorizzazioni per le funzioni, protezione delle variabili d'ambiente (che spesso contengono credenziali), e visibilità limitata sull'infrastruttura sottostante.

EXAM TIP — Serverless: basso costo + riduzione patch OS
"Cloud-based application hosting, low cost" → Serverless framework. "Reduce code deployment time and expense" → Serverless architecture. Non confondere con IaC (che è per standardizzare il provisioning), né con containers (che riducono le patch OS ma non le eliminano).

Microservices (Microservizi)

L'architettura a microservizi (microservices) decompone un'applicazione monolitica in servizi piccoli, indipendenti e debolmente accoppiati, ognuno responsabile di una singola funzionalità di business e comunicante con gli altri tramite API. Ogni microservizio può essere sviluppato, deployato, scalato e aggiornato indipendentemente.

Vantaggi di sicurezza dei microservizi vs monolitici
Scalabilità migliorata (ogni servizio scala indipendentemente) e compartimentazione aumentata: un microservizio compromesso non compromette necessariamente l'intera applicazione, come invece avverrebbe con un monolite. Il blast radius di una vulnerabilità è limitato al singolo servizio.
Svantaggi di sicurezza
La comunicazione inter-service (via API, spesso HTTP/gRPC) aumenta la superficie di attacco. La gestione delle identità e delle autorizzazioni inter-service è più complessa. Il monitoring e il debugging sono più difficili ( esclude "easier debugging").
Microservizi e JSON/API
Quando si ha bisogno di trasferire dati velocemente via richieste JSON esternamente: microservizi con API è la risposta. SFTP è file-based e non supporta JSON natively; email e cloud storage non sono pensati per trasferimento dati strutturati ad alta velocità.

Containerization (Containerizzazione)

La containerizzazione (containerization), di cui Docker rappresenta la tecnologia e la piattaforma standard industriale di riferimento, impacchetta un'applicazione con tutte le sue dipendenze (librerie, runtime, configurazioni) in un container isolato che gira sullo stesso kernel del sistema operativo host, condividendolo con altri container. A differenza delle VM (che virtualizzano l'intero hardware), i container (come quelli gestiti tramite Docker e orchestrati con Kubernetes) virtualizzano solo a livello di OS process — sono più leggeri, più veloci da avviare e più efficienti nelle risorse.

Esempio — Container vs VM: analogia pratica
Una VM è un appartamento in un palazzo: ogni appartamento ha le sue mura, il suo impianto elettrico, il suo riscaldamento (kernel separato, OS separato). Un container è una stanza in un appartamento condiviso: ognuno ha il suo spazio privato ma condivide le infrastrutture comuni (kernel del host). I container sono più efficienti ma condividono più superficie: se il kernel host ha una vulnerabilità, potenzialmente tutti i container sono esposti.
Vantaggi per la sicurezza
Isolamento applicativo, immutabilità (un container viene ricreato, non modificato), riduzione delle patch OS necessarie: l'OS host gestisce le patch, le applicazioni nel container non devono essere patchate separatamente. Ambienti costantemente mutevoli → containers.
Svantaggi per la sicurezza
Condivisione del kernel host: una vulnerabilità a livello di kernel impatta tutti i container. Container escape (equivalente del VM escape). Il codice monolitico non può essere containerizzato direttamente: un'architettura monolitica deve essere prima decomposta in microservizi per beneficiare pienamente della containerizzazione.
ATTENZIONE — Container vs VM: differenze chiave
VM: virtualizza hardware, kernel e OS separato per ogni VM. Isolamento più forte. Più pesante (overhead maggiore). Riduce l'hardware attack surface (: virtualization per ridurre server fisici).
Container: virtualizza a livello di processo OS, kernel condiviso. Più leggero. Meglio per ambienti variabili e riduzione patch. Non adatto a codice monolitico senza refactoring.

Virtualization (Virtualizzazione)

La virtualizzazione (virtualization) crea macchine virtuali (VM) che emulano hardware completo su un sistema fisico, tramite un hypervisor. Permette di far girare più sistemi operativi sullo stesso hardware fisico, riducendo il numero di server fisici necessari e quindi la superficie di attacco hardware.

La principale minaccia specifica alla virtualizzazione è il VM escape: un attaccante che sfrutta una vulnerabilità nell'hypervisor per uscire dall'isolamento della VM e accedere all'hypervisor o ad altre VM (: patch zero-day sull'hypervisor previene VM escape/escaping). In ambienti cloud, il VM escape rompe l'isolamento multi-tenant.

EXAM TIP — Virtualizzazione per ridurre server fisici
"Reduce number of physical servers" → Virtualization. "Decrease hardware attack surface" → Virtualization. Non confondere con containers (che riducono le patch OS), né con serverless (che elimina la gestione dell'infrastruttura). La virtualizzazione riduce il numero di server fisici consolidando più workload su meno hardware.

Infrastruttura di rete

Physical isolation — Air-gapped (Isolamento fisico)

Un sistema air-gapped (con gap d'aria) è fisicamente isolato da qualsiasi rete esterna: non è connesso ad internet, non è connesso alla rete aziendale interna, e non ha interfacce di rete attive. È il massimo livello di isolamento fisico possibile, usato per i sistemi più critici e sensibili: sistemi di controllo di armi nucleari, reti classificate governative, sistemi di controllo industriale in infrastrutture critiche.

Un consulente che vuole fisicamente isolare i sistemi sicuri del cliente → air gapped. Non è SDN (che è virtualizzazione della rete), non è containerizzato (isolamento software), non è high availability (che riguarda la ridondanza).

ATTENZIONE — Air-gap ≠ isolamento totale
Anche i sistemi air-gapped hanno subito attacchi: Stuxnet ha raggiunto il sistema air-gapped delle centrifughe iraniane tramite chiavette USB infette. Un sistema air-gapped ha ancora superfici di attacco fisiche: porte USB, interfacce seriali, ingegneria sociale sul personale che ha accesso fisico. L'air-gap elimina i vettori di rete ma non tutti i vettori di attacco.

Logical segmentation (Segmentazione logica)

La segmentazione logica (logical segmentation) divide la rete in zone virtuali (VLAN, subnet, zone di sicurezza) senza richiedere separazione fisica dei cavi. Permette di applicare policy di sicurezza diverse a gruppi diversi di dispositivi sullo stesso switch fisico, e di limitare il movimento laterale degli attaccanti anche se sono riusciti a compromettere un segmento.

Software-defined networking (SDN)

Il Software-Defined Networking (SDN) separa il piano di controllo (che decide come il traffico deve essere instradato) dal piano dati (che esegue fisicamente l'instradamento). Il controllo è centralizzato in un controller software che programma i dispositivi di rete fisici tramite API. Questo permette una gestione centralizzata e programmabile della rete, con policy di sicurezza applicate dinamicamente.

La centralizzazione del controller SDN crea però un single point of failure e un bersaglio di alto valore: se il controller viene compromesso, l'attaccante può riprogrammare l'intera rete. La sicurezza del controller SDN è quindi critica.

Modelli di deployment

On-premises (In sede)

Un'architettura on-premises mantiene l'intera infrastruttura IT fisicamente all'interno dell'organizzazione: hardware, software, dati e personale IT sono tutti sotto il controllo diretto dell'organizzazione. Dal punto di vista della sicurezza, è il modello con il massimo livello di controllo: nessun terzo ha accesso fisico all'infrastruttura, non ci sono dipendenze da provider esterni per la disponibilità, e le policy di sicurezza possono essere applicate senza vincoli contrattuali.

Il costo operativo è elevato (CAPEX per hardware, personale IT specializzato, spazio fisico, energia), e la scalabilità richiede acquisti di hardware con tempi lunghi. Per espansioni internazionali che richiedono il massimo livello di sicurezza → on-premises.

Centralized vs. Decentralized (Centralizzato vs. Decentralizzato)

Architettura centralizzata
Tutte le risorse e i controlli di sicurezza sono gestiti da un punto centrale. Vantaggi: visibilità completa, policy coerenti, gestione semplificata, minor costo operativo. Svantaggi: single point of failure, latenza per utenti distanti, se il centro è compromesso è tutto compromesso.
Architettura decentralizzata
Le risorse e i controlli sono distribuiti tra più nodi o sedi. Vantaggi: resilienza (nessun single point of failure), latenza ridotta per utenti distribuiti, blast radius limitato in caso di compromissione. Svantaggi: maggiore complessità di gestione, policy meno coerenti, difficoltà nel garantire visibilità uniforme.

High availability (Alta disponibilità)

L'alta disponibilità (high availability — HA) è un'architettura ridondante progettata per eliminare i single point of failure e garantire che i sistemi rimangano operativi anche in caso di guasto di uno o più componenti. Si misura in "nines": 99,9% = ~8,7 ore/anno di downtime; 99,999% = ~5 minuti/anno.

Un'architettura cloud-native è la risposta quando si cerca ridondanza per processi di business critici: sfrutta nativamente la ridondanza geografica dei cloud provider (multi-AZ, multi-region), il load balancing automatico e il failover gestito. La migrazione a IaaS risolve il problema di scalabilità di storage con resilienza ai guasti di circuito singolo.

Esempio pratico — Architettura sicura per applicazione di pagamento
Una web application di pagamento in cloud richiede: Internet Gateway → WAF (filtra traffico malevolo prima che raggiunga l'applicazione) → Load Balancer (in subnet pubblica, distribuisce il carico e garantisce HA) → due istanze applicazione in private subnet (non esposte direttamente a internet) → database in private subnet (mai in subnet pubblica). Questo schema garantisce: disponibilità, ridondanza, filtro degli attacchi web e isolamento del database.

Sistemi specializzati (Specialized Systems)

IoT — Internet of Things

I dispositivi IoT (Internet of Things) sono dispositivi fisici connessi a internet che raccolgono e trasmettono dati: sensori industriali, telecamere IP, dispositivi medici, termostati smart, sistemi di controllo degli accessi fisici. Le loro caratteristiche intrinseche creano sfide di sicurezza significative:

Risorse computazionali limitate
CPU lenta, poca RAM, batteria limitata. Non possono eseguire agenti di sicurezza tradizionali, crittografia pesante, o aggiornamenti OS complessi. La crittografia deve essere leggera (ECC invece di RSA).
Firmware rarement aggiornato
Molti dispositivi IoT non ricevono mai aggiornamenti firmware dal produttore, o gli aggiornamenti richiedono intervento fisico. Vulnerabilità note rimangono aperte indefinitamente.
Comunicazione spesso non cifrata
Molti dispositivi IoT a basso costo trasmettono dati in chiaro (: IoT è dove si trova più probabilmente traffico non cifrato). La connessione cifrata è il requisito principale per IoT in cloud.
Data storage insicuro
I dispositivi IoT a basso costo spesso memorizzano dati localmente senza cifratura (: concern principale = storage of data). Credenziali predefinite hardcoded nel firmware.
EXAM TIP — IoT: traffico non cifrato e storage non sicuro
"Where is unencrypted traffic most likely found?" → IoT. Non in SDN (che è gestione centralizzata della rete), non in VPN (cifrata per definizione), non in SCADA (che usa protocolli industriali spesso proprietari).
"Security concern for low-cost IoT in infrastructure?" → Storage of data. Non il paese di origine, non la reattività, non la facilità di deployment.

ICS / SCADA (Sistemi di controllo industriale)

I sistemi ICS (Industrial Control Systems) e SCADA (Supervisory Control and Data Acquisition) controllano infrastrutture fisiche critiche: reti elettriche, acquedotti, impianti chimici, reti del gas, sistemi di controllo del traffico, impianti di produzione. A differenza dei sistemi IT tradizionali, in questi sistemi la sicurezza fisica ha priorità sulla sicurezza informatica: un guasto può avere conseguenze fisiche catastrofiche (esplosioni, blackout, contaminazione dell'acqua).

Sfide di sicurezza ICS/SCADA
Sistemi legacy con cicli di vita di decenni, spesso impossibili da patchare senza fermare la produzione. Protocolli industriali proprietari (Modbus, DNP3, OPC) non progettati con la sicurezza in mente. Connettività OT-IT sempre più comune (convergenza IT/OT) che espone sistemi precedentemente air-gapped.
Priorità di sicurezza invertite rispetto all'IT
In IT: Confidentiality > Integrity > Availability. In ICS/SCADA: Availability > Integrity > Confidentiality. Un sistema SCADA che controlla una rete elettrica non può permettersi downtime per applicare patch: la disponibilità è la priorità assoluta.

RTOS — Real-Time Operating System

Un RTOS (Real-Time Operating System) è un sistema operativo progettato per rispondere a eventi entro un tempo garantito (deadline). È usato in sistemi dove il timing è critico: sistemi di controllo di volo, dispositivi medici (pacemaker, pompe insulina), automotive (ABS, airbag), robotica industriale. Non ha filesystem standard, non supporta software convenzionale, e spesso non può essere aggiornato facilmente. La mitigazione più efficace per attacchi a RTOS è la memory injection (OBJ 2.3), che non richiede filesystem.

Embedded systems (Sistemi embedded)

I sistemi embedded (embedded systems) sono sistemi informatici integrati in dispositivi fisici con una funzione specifica e dedicata: router, switch, stampanti, firewall hardware, ATM, sistemi di navigazione, elettrodomestici smart. Le implicazioni di sicurezza sono simili all'IoT: firmware raramente aggiornato, risorse limitate, spesso con credenziali predefinite hardcoded. La differenza principale è che i sistemi embedded sono tipicamente dispositivi singolo-funzione in infrastrutture specifiche, non sensori IoT distribuiti.

Considerazioni architetturali (Architecture Considerations)

Ogni scelta architetturale implica trade-off tra questi fattori. Non esiste un'architettura universalmente superiore: la scelta dipende dai requisiti specifici dell'organizzazione.

Availability (Disponibilità)
La capacità del sistema di essere operativo quando necessario. Alta disponibilità richiede ridondanza: load balancer, failover, multi-region deployment. Trade-off con il costo (ridondanza è cara) e la complessità.
Resilience (Resilienza)
La capacità di recuperare rapidamente da un guasto o da un attacco. Include backup e DR plan, architetture fault-tolerant, capacità di degradarsi in modo controllato (graceful degradation).
Cost (Costo)
Il costo totale di proprietà (TCO): non solo il costo iniziale ma anche operatività, personale, licenze. Il cloud riduce il CAPEX ma aumenta l'OPEX; on-premises ha alti CAPEX ma OPEX prevedibile.
Responsiveness (Reattività)
La velocità di risposta del sistema agli utenti. La latenza aumenta con la distanza geografica (risposta al cloud lontano vs on-premises locale) e con la complessità dell'architettura (molti hop di rete).
Scalability (Scalabilità)
La capacità di espandere le risorse per soddisfare la domanda crescente. Il cloud offre scalabilità elastica automatica; on-premises richiede acquisti hardware anticipatori. I microservizi scalano i singoli componenti indipendentemente.
Ease of deployment (Facilità di deployment)
Quanto è semplice e veloce deployare nuove istanze o aggiornamenti. IaC e container massimizzano la facilità di deployment; on-premises la minimizza.
Risk transference (Trasferimento del rischio)
La capacità di spostare il rischio su terzi: cyber insurance, cloud provider (che assumono la responsabilità dell'infrastruttura), MSP. Non elimina il rischio ma lo redistribuisce.
Ease of recovery (Facilità di recovery)
Quanto velocemente si può recuperare da un guasto o da un attacco. Cloud-native e IaC massimizzano la facilità di recovery (si ri-deploya da zero in minuti); on-premises richiede restore da backup con tempi più lunghi.
Patch availability (Disponibilità di patch)
Quanto velocemente il vendor rilascia patch per vulnerabilità note. I sistemi moderni (cloud, container) ricevono patch frequenti; IoT, RTOS e sistemi EOL spesso non ne ricevono.
Inability to patch (Impossibilità di patchare)
Sistemi che non possono essere patchati: IoT EOL, RTOS, ICS/SCADA durante operazioni critiche. Richiedono controlli compensativi: segmentazione, monitoring rinforzato, deny list.
Power (Energia)
Il consumo energetico dell'architettura. I data center on-premises richiedono sistemi UPS, generatori, HVAC. Il cloud trasferisce questo onere al provider. Critico per dispositivi IoT a batteria: influenza la scelta degli algoritmi crittografici.
Compute (Potenza di calcolo)
La capacità di elaborazione disponibile. Limita la scelta degli algoritmi di sicurezza per IoT e sistemi embedded. Il cloud offre potenza di calcolo virtualmente illimitata su richiesta (elasticità).

Jump server / Jump host

Un jump server (jump host o bastion host) è un sistema dedicato posizionato in una zona di sicurezza specifica (tipicamente una shared services security zone) che funge da unico punto di accesso per amministrare sistemi in zone più protette. Gli amministratori si connettono prima al jump server (autenticato), e da lì raggiungono i sistemi interni. Il jump server registra tutte le sessioni amministrative fornendo audit trail completo.

Quando un amministratore configura l'accesso VPN a un ambiente cloud e vuole facilitare l'amministrazione remota sicura → jump host nella shared services security zone. Non un SSH server nella LAN (che non sarebbe nella zona corretta per l'accesso cloud), non un reverse proxy sul firewall (non è progettato per sessioni amministrative).

Shared Responsibility: IaaS vs SaaS Clicca per girare

IaaS: il cliente gestisce OS, app, dati e sicurezza database virtualizzati.
SaaS: il provider cloud gestisce praticamente tutto eccetto dati inseriti e policy d'accesso.

Definizione di SASE in cloud Clicca per girare

Convergenza integrata di SD-WAN con funzionalità di sicurezza avanzate (SWG, CASB, ZTNA, FWaaS) erogata interamente tramite architetture di rete cloud-native.

Infrastructure as Code (IaC) Clicca per girare

Definizione e provisioning di infrastrutture IT tramite codice strutturato (Terraform, Ansible), azzerando l'errore umano e prevenendo il configuration drift.

Serverless: principali vantaggi Clicca per girare

Nessun server da manutenere, pagamenti basati su esecuzione delle sole chiamate attive (pay-per-use), e tempi di deployment del codice drasticamente ridotti.

VM Escape vs Container Escape Clicca per girare

VM Escape: evasione logica da VM verso l'hypervisor fisico.
Container Escape: evasione logica da container verso il kernel host condiviso.

Priorità IT vs Priorità ICS/SCADA Clicca per girare

IT Tradizionale: Confidentiality (Primo) → Integrity → Availability (Ultimo).
SCADA/ICS: Availability (Primo) → Integrity → Confidentiality (Ultimo).
