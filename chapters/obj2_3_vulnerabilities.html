Vulnerabilità applicative (Application Vulnerabilities)

Le vulnerabilità applicative nascono da errori nel codice software: logica difettosa, mancata validazione dell'input, gestione scorretta della memoria. Sono le più diffuse e le più sfruttate dagli attaccanti perché il software è ovunque e il codice sicuro è difficile da scrivere. OWASP (Open Web Application Security Project) è la risorsa di riferimento per le tecniche di exploitation applicativo più comuni.

Memory injection (Iniezione di memoria)

La memory injection (iniezione di memoria) è una tecnica in cui l'attaccante inserisce codice malevolo nello spazio di memoria allocato di un processo già in esecuzione. Il codice malevolo viene eseguito nel contesto del processo legittimo, il che permette di bypassare i controlli di sicurezza del sistema operativo che si basano sull'identità del processo. Poiché il malware non esiste come file su disco ma solo in memoria, può eludere molti antivirus tradizionali basati su firma di file.

Process injection
La forma più comune: il codice malevolo viene iniettato in un processo legittimo (es. svchost.exe su Windows, o il browser) e viene eseguito nel suo contesto, ereditandone i privilegi e le autorizzazioni.
DLL injection
Un attaccante forza un processo a caricare una Dynamic Link Library (DLL) malevola. La DLL viene eseguita nel contesto del processo, con i suoi permessi.
Shellcode injection
Codice macchina compatto (shellcode) viene iniettato in memoria e poi l'esecuzione viene reindirizzata ad esso tramite tecniche come buffer overflow o return-oriented programming (ROP).
Esempio pratico — Memory injection su RTOS
Un Real-Time Operating System (RTOS) è un sistema operativo utilizzato in dispositivi embedded (PLC industriali, dispositivi medici, sistemi di controllo). Un attaccante che vuole compromettere un RTOS non può installare malware tradizionale — non c'è filesystem standard. La soluzione è la memory injection: il codice malevolo viene iniettato direttamente nella memoria del sistema durante l'esecuzione, senza mai toccare il disco.

Buffer overflow (Overflow del buffer)

Il buffer overflow si verifica quando un programma scrive più dati di quanti un buffer (area di memoria allocata) possa contenere, sovrascrivendo aree di memoria adiacenti. Se il programma non verifica i limiti dell'input (bounds checking), un attaccante può deliberatamente fornire un input troppo grande per sovrascrivere: l'indirizzo di ritorno di una funzione (puntando a codice malevolo), variabili critiche del programma, o registri del processore con indirizzi malevoli.

È una delle vulnerabilità più antiche e ancora diffuse, specialmente in codice C/C++ che gestisce direttamente la memoria. Linguaggi moderni come Java, Python e Rust includono controlli automatici dei limiti che prevengono questi overflow.

ATTENZIONE — Buffer overflow vs Memory injection
Buffer overflow: il programma riceve dati di dimensione inaspettata che sovrascrivono memoria adiacente → l'attacco sfrutta l'assenza di bounds checking.
Memory injection: codice malevolo viene copiato nello spazio di memoria allocato di un processo già in esecuzione → il processo esiste già, l'attaccante vi "inietta" il codice dall'esterno.
Il buffer overflow può essere il vettore attraverso cui si realizza una memory injection.

Race condition — TOC/TOU

Una race condition (condizione di gara) si verifica quando il comportamento di un sistema dipende dalla sequenza temporale di eventi concorrenti che non è garantita. In sicurezza, la race condition più rilevante è il pattern TOC/TOU (Time-of-Check / Time-of-Use):

Time-of-Check (TOC)
Il momento in cui il sistema verifica una condizione (es. "questo utente ha i permessi per accedere a questo file?").
Time-of-Use (TOU)
Il momento in cui il sistema usa il risultato di quella verifica (es. "apri il file").
L'attacco
Se un attaccante riesce a modificare le condizioni tra il TOC e il TOU — nella finestra temporale tra la verifica e l'uso — può sfruttare il sistema. Esempio classico: la verifica (TOC) controlla che un file sia legittimo; l'attaccante lo sostituisce con un file malevolo prima che venga usato (TOU). Il sistema usa il file malevolo credendolo legittimo.
Esempio pratico — Race condition in un database
Durante un aggiornamento SQL del database, viene creato un campo temporaneo per la transazione. Un attaccante sfrutta la finestra temporale tra la creazione del campo (TOC) e il suo utilizzo (TOU) per sostituire il valore del campo con uno malevolo che permette l'accesso al sistema. Il database usa il valore modificato credendolo legittimo. Questo scenario è classificato come "malicious update" con la tecnica della race condition.

Malicious update (Aggiornamento malevolo)

Un malicious update (aggiornamento malevolo) è un aggiornamento software apparentemente legittimo che contiene o introduce codice malevolo. Può avvenire a diversi livelli: un attaccante che compromette l'infrastruttura di update di un vendor (supply chain attack), un aggiornamento distribuito da un vendor compromesso, o un aggiornamento falso che imita quello del vendor reale. È distinto dal supply chain attack nel senso che riguarda specificamente il meccanismo di aggiornamento, non l'intera catena di fornitura.

Vulnerabilità web-based (Web-based Vulnerabilities)

SQL injection (SQLi)

La SQL injection (SQLi) è un attacco in cui l'attaccante inserisce comandi SQL malevoli in un campo di input di un'applicazione web che non esegue la validazione dell'input. Se il backend costruisce le query SQL concatenando direttamente l'input dell'utente, l'attaccante può modificare la query originale per: visualizzare dati non autorizzati, modificare o cancellare dati, bypassare l'autenticazione, ed eseguire comandi sul sistema operativo sottostante.

Esempio pratico — SQLi classico
Un form di login costruisce la query: SELECT * FROM users WHERE username='[input]' AND password='[input]'
L'attaccante inserisce come username: ' OR '1'='1
La query diventa: SELECT * FROM users WHERE username='' OR '1'='1' AND password='...'
Poiché '1'='1' è sempre vero, la query restituisce tutti gli utenti. L'attaccante accede senza password valida.
Mitigazione: prepared statements (query parametrizzate), input validation, principio del least privilege per l'account DB.
EXAM TIP — SQLi: le parole chiave dello scenario
"Input field to run commands that view or manipulate data" → SQL injection. "Unexpected characters to gain unauthorized access to underlying systems" → SQL injection. "Database misconfigurations" + exploitation → SQL injection. L'applicazione web che permette l'upload di immagini con un processo che genera traffico outbound verso IP a bassa reputazione → web shell deploiata via vulnerabilità di upload, non direttamente SQLi.

Cross-site scripting (XSS)

Il Cross-site scripting (XSS) è una vulnerabilità che permette a un attaccante di iniettare script malevoli (tipicamente JavaScript) in pagine web visualizzate da altri utenti. A differenza della SQLi (che attacca il database), la XSS attacca il client: il browser della vittima esegue il codice malevolo credendo che provenga dal sito legittimo, permettendo all'attaccante di rubare cookie di sessione, reindirizzare l'utente, modificare il contenuto della pagina, o eseguire azioni per conto della vittima.

Stored XSS (Persistente)
Lo script malevolo viene salvato nel database del server (es. in un commento di un forum). Ogni utente che carica la pagina esegue lo script. È la forma più pericolosa perché non richiede ingegneria sociale per ogni vittima.
Reflected XSS (Non persistente)
Lo script malevolo è nell'URL della richiesta. Il server lo "riflette" nella risposta. La vittima deve essere indotta a cliccare su un link appositamente costruito (es. via phishing).
DOM-based XSS
L'attacco avviene interamente nel browser della vittima: JavaScript legittimo del sito usa parametri dell'URL o dell'ambiente DOM per scrivere contenuto nella pagina, e l'attaccante manipola quei parametri.
ATTENZIONE — XSS vs SQLi: qual è il bersaglio?
SQLi: bersaglio = database del server → l'attaccante manipola le query SQL per leggere/modificare i dati nel DB.
XSS: bersaglio = browser del client → l'attaccante inietta script che vengono eseguiti nel browser degli utenti.
Nello scenario vedi <script>function(send_info)</script> nei log → è XSS, non SQLi: è JavaScript iniettato, non comandi SQL. Mitigazione XSS: input validation.

Directory traversal (Path traversal)

Il directory traversal (anche noto come path traversal) permette a un attaccante di accedere a file e directory al di fuori della cartella web root del server, navigando verso l'alto nella gerarchia del filesystem usando sequenze come ../. Se l'applicazione non sanitizza i percorsi nei parametri di input, l'attaccante può leggere file sensibili del sistema operativo.

Il pattern classico è: GET ../../../../etc/passwd — quattro livelli su dalla web root per raggiungere /etc/passwd, il file delle password di sistema Unix/Linux. Ogni ../ sale di un livello nella gerarchia del filesystem.

Vulnerabilità OS e Hardware

OS-based (Vulnerabilità del sistema operativo)

Le vulnerabilità OS-based esistono nel kernel o nei componenti core del sistema operativo. Sono particolarmente critiche perché il sistema operativo ha il controllo totale sull'hardware e sulle applicazioni: una vulnerabilità nel kernel può permettere a un attaccante di ottenere privilegi elevati (privilege escalation), bypassare tutti i controlli di sicurezza applicativi, o compromettere l'intero sistema. Le patch del sistema operativo sono tra le più urgenti da applicare.

Le vulnerabilità OS-based includono: difetti nel kernel, nel gestore della memoria, nel sistema di controllo degli accessi, nei driver di dispositivo, e nei servizi di sistema. Molte vulnerabilità famose come EternalBlue (usato da WannaCry) erano vulnerabilità OS-based nel protocollo SMBv1 di Windows.

Firmware (Vulnerabilità del firmware)

Il firmware è il software embedded nell'hardware: BIOS/UEFI dei computer, firmware dei router e switch, firmware delle stampanti, dei dispositivi IoT, delle schede di rete. Le vulnerabilità firmware sono particolarmente insidiose perché:

Persistenza
Il firmware risiede in memoria non volatile (flash ROM). Una reinstallazione del sistema operativo o la formattazione del disco non rimuove il malware installato nel firmware. Sopravvive anche alla sostituzione del disco.
Difficoltà di rilevamento
Il firmware viene eseguito prima del sistema operativo, quindi i tradizionali strumenti di sicurezza (AV, EDR) che girano sull'OS non possono rilevarlo.
Aggiornamenti rari e complessi
Gli aggiornamenti firmware (es. BIOS update, router firmware update) sono spesso ignorati dagli amministratori per paura di brick del dispositivo, lasciando vulnerabilità aperte a lungo.
EXAM TIP — Firmware = Hardware-specific vulnerability
"Which of the following is a hardware-specific vulnerability?" → Firmware version. Buffer overflow, SQL injection e XSS sono vulnerabilità software/applicative, non hardware-specific. Il firmware è strettamente legato all'hardware su cui gira. "Vendor sends security bulletin recommending a BIOS update" → vulnerabilità di tipo Firmware.

End-of-life (EOL) / Legacy

Un sistema o un'applicazione raggiunge lo stato di End-of-Life (EOL) quando il vendor non rilascia più patch di sicurezza, aggiornamenti o supporto tecnico. Il rischio critico non è la mancanza di nuove funzionalità, ma la mancanza di aggiornamenti di sicurezza: le vulnerabilità scoperte dopo la data EOL non vengono mai corrette, lasciando il sistema permanentemente esposto.

I sistemi legacy sono sistemi obsoleti ancora in uso perché il costo o la complessità della sostituzione è troppo elevata (es. sistemi di controllo industriale, applicazioni custom critiche). La loro vulnerabilità principale è spesso l'impossibilità di ricevere patch — a cui si risponde con controlli compensativi (segmentazione di rete, VLAN isolate, monitoring rinforzato).

ATTENZIONE — EOL: il rischio principale non è tecnico
Quando un'applicazione raggiunge EOL, il rischio critico è la mancanza di aggiornamenti di sicurezza (lack of security updates) — non la mancanza di nuove funzionalità, non la mancanza di supporto tecnico (quella è una conseguenza), e non la mancanza di accesso al codice sorgente. Il vendor smette di patchare: qualsiasi vulnerabilità scoperta d'ora in poi rimane aperta per sempre.

Vulnerabilità di virtualizzazione (Virtualization Vulnerabilities)

VM escape (Fuga dalla VM)

Il VM escape (fuga dalla macchina virtuale) è una delle vulnerabilità più critiche in ambienti virtualizzati: un attaccante che ha compromesso una macchina virtuale (guest) riesce a "uscire" dal suo ambiente isolato e ad accedere all'hypervisor — il software che gestisce tutte le VM sul host fisico. Una volta sull'hypervisor, l'attaccante può accedere e compromettere tutte le altre VM sullo stesso host fisico.

Il VM escape è raro ma catastrofico: sfrutta vulnerabilità nell'hypervisor stesso o nei driver virtuali (es. la scheda grafica virtuale, il controller USB virtuale). In un cloud pubblico, un VM escape potrebbe permettere a un tenant di accedere ai dati di altri tenant sullo stesso hardware fisico — violazione totale dell'isolamento multi-tenant.

Resource reuse (Riutilizzo delle risorse)

Il resource reuse (riutilizzo delle risorse) è una vulnerabilità che si verifica quando risorse hardware o software (memoria, storage, registri) vengono riassegnate a un nuovo utente o processo senza essere prima azzerata o bonificata. I dati del precedente utente possono essere letti dal nuovo utente. In ambienti virtualizzati e cloud, questo è particolarmente rilevante: quando una VM viene terminata e le sue risorse (RAM, storage) vengono allocate a un nuovo tenant, se la bonifica non è completa i dati del vecchio tenant possono essere esposti.

Supply chain e Cloud

Supply chain (Catena di fornitura)

Una supply chain vulnerability (vulnerabilità della catena di fornitura) si verifica quando un attaccante compromette un fornitore (software, hardware o servizi) per poi raggiungere i clienti finali attraverso prodotti o servizi apparentemente legittimi. Non si attacca direttamente il bersaglio — si attacca un suo fornitore di fiducia, sfruttando il livello di trust implicito in quella relazione.

Software provider (Fornitore software)
Attacchi come SolarWinds (2020): gli attaccanti hanno compromesso il sistema di build del vendor, inserendo malware nell'aggiornamento legittimo distribuito a 18.000+ clienti. Il codice malevolo era firmato digitalmente dal vendor — i clienti non avevano modo di distinguerlo da un aggiornamento legittimo.
Hardware provider (Fornitore hardware)
Componenti hardware (chip, schede di rete, router) vengono compromessi durante la produzione o la distribuzione. Il malware è nel firmware o nell'hardware stesso, invisibile al software di sicurezza tradizionale.
Service provider (Fornitore di servizi)
Un Managed Service Provider (MSP) o un SaaS provider che gestisce sistemi per molti clienti diventa un bersaglio ad alto valore: compromettere l'MSP significa accedere a tutti i suoi clienti simultaneamente.
SBOM (Software Bill of Materials)
Un inventario completo di tutti i componenti, librerie e dipendenze incluse in un prodotto software. Permette di identificare rapidamente quali dipendenze sono vulnerabili e quali prodotti ne sono affetti.
EXAM TIP — Supply chain: lo scenario "malicious update" distribuito
"Un malicious update è stato distribuito a una piattaforma software comune, disabilitando servizi in molte organizzazioni" → Supply chain vulnerability. Non è un DDoS (non è un attacco di rete), non è un insider threat (è un aggiornamento distribuito), non è un rogue employee. È un supply chain attack perché il vettore è il sistema di distribuzione del software.

Cloud-specific (Vulnerabilità specifiche del cloud)

Le cloud-specific vulnerabilities sono rischi unici o amplificati dall'architettura cloud. La più comune e sottovalutata è la credential leakage nei code repository: gli sviluppatori commettono accidentalmente credenziali cloud (API key, access token, password di database) in repository Git pubblici o privati. Chiunque acceda al repository può usare quelle credenziali per accedere alle risorse cloud.

Misconfigured storage buckets
Bucket S3, Azure Blob Storage o Google Cloud Storage configurati come pubblici per errore espongono dati sensibili a chiunque conosca l'URL (o anche senza conoscerlo, tramite tool di enumerazione).
Insecure APIs
Le API cloud prive di autenticazione, autorizzazione o rate limiting possono essere sfruttate per accedere, modificare o cancellare risorse cloud senza autorizzazione.
Shared responsibility model violations
I clienti cloud non comprendono correttamente cosa è responsabilità del provider e cosa è responsabilità loro, lasciando gap di sicurezza in aree di cui credevano il provider si occupasse.

Vulnerabilità crittografiche e di configurazione

Cryptographic vulnerabilities (Vulnerabilità crittografiche)

Le vulnerabilità crittografiche derivano dall'uso di algoritmi deprecati, chiavi di lunghezza insufficiente, implementazioni scorrette di algoritmi corretti, o dalla gestione non sicura delle chiavi. Non è necessario "rompere" la crittografia matematicamente — spesso è sufficiente sfruttare implementazioni deboli o algoritmi obsoleti per cui esistono già attacchi pratici.

Algoritmi deprecati
MD5 e SHA-1 per l'hashing: vulnerabili a collision attack. RC4 per la cifratura simmetrica: vulnerabile a attacchi statistici. DES con chiave a 56 bit: forzabile con brute-force moderno in ore. L'uso di questi algoritmi introduce vulnerabilità crittografiche.
Chiavi deboli o corte
RSA a 512 o 1024 bit è considerato insicuro. Chiavi simmetriche a 56 o 64 bit sono forzabili. La lunghezza della chiave è direttamente correlata alla sicurezza: chiavi più corte = vulnerabilità crittografiche.
Password cracking via weak crypto
Se le password sono hashate con algoritmi deboli (MD5, SHA-1 non salted), un attaccante con le hash può usare rainbow table o GPU per trovare le password originali rapidamente.

Misconfiguration (Errata configurazione)

La misconfiguration (errata configurazione) è una delle cause più frequenti di violazioni di sicurezza: sistemi tecnicamente validi configurati in modo non sicuro. La complessità dei sistemi moderni (cloud, container, microservizi, firewall) rende facile commettere errori di configurazione che espongono risorse critiche.

Default credentials (Credenziali predefinite)
Dispositivi (router, switch, IoT) con username/password di fabbrica non cambiati. Sono pubblicamente note — gli scanner automatizzati le provano sistematicamente.
Open permissions (Permessi aperti)
File, directory, bucket cloud o database con permessi troppo permissivi: accesso in lettura/scrittura per "tutti" o per gruppi troppo ampi.
Configurazioni di sicurezza disabilitate
Firewall con regole troppo permissive, logging disabilitato, TLS non configurato su servizi che supportano solo HTTP, autenticazione disabilitata per "comodità".
Esempio pratico — Environmental variables come misconfiguration
Le variabili d'ambiente (environment variables) sono configurazioni che modificano il comportamento di applicazioni e sistemi. Possono contenere: stringhe di connessione al database, API key, flag di debug, configurazioni di sicurezza. Se una variabile d'ambiente contiene un valore non sicuro (es. debug=true in produzione, una chiave API con permessi eccessivi), influenza direttamente la portata e l'impatto di una vulnerabilità già esistente. Una XSS in un'applicazione con debug=true può esporre stack trace con informazioni sensibili; senza debug, la stessa XSS è meno informativa per l'attaccante.

Vulnerabilità mobile (Mobile Device Vulnerabilities)

Side loading (Caricamento laterale)

Il side loading è l'installazione di applicazioni su un dispositivo mobile attraverso canali non ufficiali — al di fuori dell'app store ufficiale del produttore (App Store per iOS, Google Play per Android). L'app store ufficiale esegue controlli di sicurezza sulle app (revisione del codice, scansione malware, verifica della firma). Le app installate via side loading bypassano questi controlli, aumentando significativamente il rischio di installare malware o app non attendibili.

Jailbreaking / Rooting

Il jailbreaking (per iOS) e il rooting (per Android) sono processi che rimuovono le restrizioni imposte dal produttore sul sistema operativo mobile, ottenendo accesso root (privilegio massimo) al dispositivo. Dal punto di vista della sicurezza aziendale, il jailbreaking è critico perché:

Bypass dei controlli di sicurezza
Il jailbreaking rimuove o disabilita il sandboxing, il secure boot, e altri controlli di sicurezza del sistema operativo. Le app malevole possono accedere a risorse di sistema normalmente inaccessibili.
Rischio BYOD (Bring Your Own Device)
In un programma BYOD, se un dipendente usa un dispositivo jailbroken per accedere alle risorse aziendali, tutti i dati aziendali sul dispositivo sono potenzialmente esposti. La policy BYOD deve proibire esplicitamente il jailbreaking.
MDM bypass
I dispositivi jailbroken spesso bypassano le soluzioni di Mobile Device Management (MDM), rendendo impossibile per l'IT aziendale applicare policy di sicurezza e gestire il dispositivo.
ATTENZIONE — Side loading vs Jailbreaking
Side loading: installazione di app da fonti non ufficiali, senza modificare il sistema operativo. Il device rimane "normale", ma l'app non è stata controllata. → "unauthorized installation through means other than the official app store".
Jailbreaking: modifica del sistema operativo per rimuovere le restrizioni del produttore, ottenendo accesso root. → "modifying the operating system on mobile devices".

Zero-day (Vulnerabilità zero-day)

Una vulnerabilità zero-day è una vulnerabilità software che è stata scoperta ma per cui il vendor non ha ancora rilasciato una patch. Il termine "zero-day" indica che il difensore ha avuto zero giorni per prepararsi: nel momento in cui la vulnerabilità viene exploited per la prima volta (o pubblicata), non esiste ancora una correzione disponibile. Le organizzazioni sono per definizione esposte perché non c'è niente da patchare.

Le zero-day sono particolarmente pericolose nelle dipendenze open-source: una libreria usata da decine di migliaia di applicazioni (come Log4j nel 2021) può avere una zero-day che espone simultaneamente tutte le applicazioni che la includono — e le organizzazioni che dipendono da librerie open source non controllano il codice di quelle librerie.

I nation-state actor e il crimine organizzato più sofisticato investono significativamente per scoprire e acquistare zero-day, pagando da decine di migliaia a milioni di dollari per exploit su piattaforme ad alto valore. Quando una zero-day diventa pubblica, il tempo tra la divulgazione e lo sviluppo di exploit "di massa" si è ridotto da settimane a ore nell'ultimo decennio.

EXAM TIP — Zero-day in sistemi con dipendenze open source
"Un'azienda usa librerie open-source. Quale tipo di vulnerabilità sarebbe più difficile da rimediare?" → Zero-day. Le librerie open source non sono sotto il controllo dell'organizzazione: non possono patchare da sole, devono aspettare il maintainer della libreria. Per buffer overflow, SQLi e XSS nel codice proprio, l'organizzazione può agire direttamente.
